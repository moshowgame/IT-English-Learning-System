Running a Code Review Session

Article 06 · Module 2: Design & Code Quality · Tech Lead English

⏱ 8 min read 🟡 Intermediate 🎯 PR review etiquette, blocking vs non-blocking, mentoring

1. Scenario 场景背景

It is Tuesday, 2:00 PM HKT. You are reviewing a pull request from Lin, a junior developer. The PR implements the new idempotency key generator for the payment service. The PR has 240 lines of code changes, and you notice: (a) the function is 80 lines long, (b) 6 places where exceptions are swallowed silently, and (c) no test for the 11-digit HKID edge case.

周二下午2点香港时间，你正在 review Lin（junior 开发）的 PR。这个 PR 实现了支付服务的幂等键生成器。PR 改了 240 行代码，你发现：(a) 函数 80 行，(b) 6 处异常被静默吞掉，(c) 11 位香港身份证边界场景没测试。

2. Dialogue 英文对话

Lin (Slack DM):Hey Wei, I just pushed PR #428. Could you take a look when you get a chance?嘿 Wei，我刚推了 PR #428。有空能帮我看一下吗？
You (TL):Sure, Lin. Give me 30 minutes — I'll do a full pass and leave inline comments.好 Lin。给我 30 分钟——我完整过一遍，留行内评论。
You (TL, PR comment):Thanks for the PR, Lin — great work overall. I've left three blocking comments and four non-blocking suggestions. Let me walk you through the blocking ones first.谢谢你 PR Lin——整体做得不错。我留了3 条阻塞评论和4 条非阻塞建议。我先过一下阻塞的。
You (TL, PR comment):Blocking 1: the function is 80 lines. I'd extract the HKID-validation logic into a separate method — single-responsibility principle. This is a structural change, not a style change.阻塞 1：函数 80 行。我建议把香港身份证校验逻辑抽到单独方法——单一职责原则。这是结构改动，不是风格改动。
You (TL, PR comment):Blocking 2: 6 places where exceptions are silently swallowed. Each catch (Exception e) { } is a debugging nightmare waiting to happen. At minimum, log the exception with context.阻塞 2：6 处异常被静默吞掉。每个 catch (Exception e) { } 都是调试噩梦的伏笔。至少记录异常加上下文。
You (TL, PR comment):Blocking 3: missing test for the 11-digit HKID edge case. HKID can be 8 or 11 chars. Can you add a unit test for both forms — and a 7-char invalid form?阻塞 3：缺 11 位香港身份证边界场景测试。香港身份证可以是 8 或 11 位。能给两种形式都加单元测试吗？再加一个 7 位无效形式。
You (TL, PR comment):Non-blocking suggestions: variable naming in lines 87-92, adding Javadoc on the public method, comment explaining why you chose UUID v7 over v4. These are nice-to-haves, not must-haves.非阻塞建议：第 87-92 行变量命名、给 public 方法加 Javadoc、加注释解释为什么选 UUID v7。这些是 nice-to-have，不是 must-have。
Lin (Slack DM):Got it, Wei. The blocking items make sense. Quick question — for exception handling, is it OK to use a global handler instead of logging in each catch?明白了 Wei。阻塞项有道理。问一句——异常处理，用全局 handler 行吗？不用每个 catch 都记录。
You (TL):Yes, that's actually a cleaner approach. Use a global @ControllerAdvice handler, and log with a correlation ID. Nice instinct.可以，这其实更干净。用全局 @ControllerAdvice handler，记日志时带上 correlation ID。直觉不错。

3. Key Phrases 关键句型

  • Open Great work overall.整体做得不错。Always open a code review with a positive. The junior needs to know what's right, not just what's wrong.
  • Tag Three blocking comments and four non-blocking suggestions.3 条阻塞评论和 4 条非阻塞建议。Always label comments as blocking or non-blocking. The author needs to know what blocks merge.
  • Reason This is a structural change, not a style change.这是结构改动，不是风格改动。Explain WHY something is blocking. Structure is a maintainability concern; style is a preference.
  • Risk A debugging nightmare waiting to happen.调试噩梦的伏笔。Frame the consequence, not the rule.
  • Specific Can you add a unit test for both forms?能给两种形式都加单元测试吗？Ask for the specific test, not 'more tests'.
  • Soft Won't block merge.不阻塞合并。Make non-blocking explicitly non-blocking.
  • Praise Nice instinct.直觉不错。End with a small praise when the junior proposes a good idea.

4. Vocabulary 词汇表

  • PR (Pull Request) n.. A proposed code change submitted for review before merge. 拉取请求
  • Blocking comment n.. A review comment that must be addressed before merge. 阻塞评论
  • Non-blocking suggestion n.. A review comment that should be considered but won't stop merge. 非阻塞建议
  • Inline comment n.. A review comment attached to a specific line in the diff. 行内评论
  • Single-responsibility principle n.. Each function should do one thing well. 单一职责原则
  • Exception swallowing n.. Catching an exception and doing nothing with it. 静默吞掉异常
  • Correlation ID n.. A unique ID that ties together all log lines for one request. 关联 ID
  • ControllerAdvice n.. Spring annotation for global exception handlers. Spring 全局异常处理
  • Idempotency key n.. A unique key that ensures a payment is processed only once. 幂等键
  • Mentoring via PR n.. Using code review as a teaching tool, not a gate. PR 中的导师制

5. Grammar Notes 语法点

① 'This is X, not Y' — distinguishing the kind of issue

Use this is X, not Y to label the type of feedback. This is a structural change / This is a security concern, not a style preference. The label tells the author how seriously to take it.

用 this is X, not Y 标记反馈类型。

② 'Nice-to-haves, not must-haves' — the de-escalation

Use nice-to-have / must-have / should-have / could-have to label priority. Pair with not to soften: nice-to-haves, not must-haves.

用 nice-to-have / must-have / should-have / could-have 标记优先级。

③ 'Nice instinct' — micro-praise in code review

Use short micro-praise phrases when a junior proposes a good idea. Nice instinct / Good catch / I like that / That's cleaner.

用简短的微表扬词。

6. Cultural Tips 文化 / 职场 Tips

  • PRs are mentoring tools, not gates. A TL's code review is the cheapest mentoring you do.
  • Label every comment: blocking or non-blocking. The author needs to know what blocks merge.
  • Open with a positive. 'Great work overall' sets the tone.
  • Explain the WHY for blocking items. 'Extract this method' is a rule. The principle is a teaching.
  • Frame the consequence for risky patterns. 'Swallowing exceptions = debugging nightmare' is more memorable.
  • Make non-blocking explicitly non-blocking. 'Won't block merge' is the magic phrase.
  • End with a micro-praise when earned. It reinforces the next good idea.

7. Practice 练习题

Exercise 1 — Translate the code review

Translate the following code review comments into natural Chinese.

"Thanks for the PR, Lin — great work overall. I've left three blocking comments and four non-blocking suggestions. Let me walk you through the blocking ones first. Blocking 1: the function is 80 lines. I'd extract the HKID-validation logic into a separate method — single-responsibility principle. This is a structural change, not a style change. Blocking 2: 6 places where exceptions are silently swallowed. Each catch is a debugging nightmare waiting to happen. At minimum, log the exception with context. Blocking 3: missing test for the 11-digit HKID edge case. Can you add a unit test for both forms — and a 7-char invalid form? Non-blocking suggestions (won't block merge, but worth doing): variable naming in lines 87-92, adding Javadoc on the public method, and a comment explaining why you chose UUID v7 over v4. These are nice-to-haves, not must-haves."

"谢谢你 PR Lin——整体做得不错。我留了 3 条阻塞评论和 4 条非阻塞建议。我先过一下阻塞的。阻塞 1：函数 80 行。我建议把香港身份证校验逻辑抽到单独方法——单一职责原则。这是结构改动，不是风格改动。阻塞 2：6 处异常被静默吞掉。每个 catch 都是调试噩梦的伏笔。至少记录异常加上下文。阻塞 3：缺 11 位香港身份证边界场景测试。能给两种形式都加单元测试吗？再加一个 7 位无效形式。非阻塞建议（不阻塞合并，但值得做）：第 87-92 行变量命名、给 public 方法加 Javadoc，加注释解释为什么选 UUID v7。这些是 nice-to-have，不是 must-have。"

Exercise 2 — Spot the TL mistake

Which of these code review lines is most 'TL-mentor-like'? Why?

  1. 'This is bad. Rewrite the whole thing.'
  2. 'Thanks for the PR — great work overall. Three blocking items, four non-blocking. The first blocking: extract this method. This is a structural change, not a style change.'
  3. 'LGTM.'
  4. 'Variable names are wrong. Fix.'

② is TL-mentor-like. ① and ④ are gate-keeper-like. ③ is rubber-stamping. A common mistake is either gate-keeping or rubber-stamping. The middle path is the TL's path.

Exercise 3 — Your turn

You're reviewing a PR from a junior. The PR has 200 lines. You found: (a) one missing test for an edge case, (b) two places where exceptions are swallowed, (c) a 60-line function. Write your TL code review in English. Use at least 3 patterns from Key Phrases.

"Thanks for the PR — great work overall. The new validation logic is much cleaner. I've left two blocking comments and three non-blocking suggestions. Blocking 1: the function is 60 lines. I'd extract the email-format validation into a separate method — single-responsibility principle. This is a structural change, not a style change. Blocking 2: two places where exceptions are silently swallowed. Each catch is a debugging nightmare waiting to happen. At minimum, log the exception with context. Plus a missing test for the empty-string edge case. Can you add a unit test that confirms the validator rejects empty input? Non-blocking suggestions (won't block merge, but worth doing): variable naming in lines 23-27, adding Javadoc on the public method. These are nice-to-haves, not must-haves. Nice instinct on the new error-message structure, by the way."
