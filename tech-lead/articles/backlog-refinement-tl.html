Backlog Refinement as Tech Lead

Article 03 · Module 1: Daily Tech-Lead Communication · Tech Lead English

⏱ 8 min read 🟡 Intermediate 🎯 Ask sharp questions, clarify AC, raise technical concerns

1. Scenario 场景背景

It is Thursday, 11:00 AM HKT. Your squad's weekly backlog refinement is running. The product owner, James, walks through 8 stories for the next sprint. Two of them have technical concerns: ONE-401 requires integrating with a third-party sanctions list that has rate limits, and ONE-415 involves storing PII data subject to HKMA reporting.

周四上午11点香港时间，你的 squad 周会 backlog refinement 正在进行。PO James 介绍下个 Sprint 的 8 个故事。其中 2 个有技术问题：ONE-401 需要对接有速率限制的第三方制裁名单，ONE-415 涉及存储受 HKMA 报送约束的 PII 数据。

As TL, your job is to: challenge assumptions, clarify acceptance criteria, surface technical risks. You are not the one writing the story — but you are the one who has to make it work.

作为 TL，你的职责：质疑假设、澄清验收标准、暴露技术风险。你不用写故事，但你要让它能跑通。

2. Dialogue 英文对话

James (PO): OK, next one — ONE-401. We need to integrate with Refinitiv's World-Check API for sanctions screening. Five points.好，下一个——ONE-401。我们要对接 Refinitiv 的 World-Check API 做制裁名单筛查。5 个点。
You (TL): Hold on, James. Can we walk through the acceptance criteria? Specifically, what's our expected peak QPS? World-Check has a published rate limit of 10 QPS per API key — if we expect 50 QPS at peak, 5 points isn't enough for the rate-limiting work.等一下 James。我们过一下验收标准？具体地，预期峰值 QPS 是多少？ World-Check 每个 API key 公开限速 10 QPS——如果我们预期峰值 50 QPS，5 个点不够做限流工作。
James (PO): Hmm, the regulator asks for real-time screening on every payment above HKD 100,000. We process about 200 of those per hour at peak.嗯，监管要求每笔 10 万港币以上的支付都要实时筛查。我们峰值时每小时大约 200 笔。
You (TL): 200 per hour is way below the rate limit — about 0.06 QPS average. But the burst pattern matters: if 50 payments arrive in the same second, we'll hit the 10 QPS cap and fail screening. I'd add a 2-point spike for a token bucket + retry queue to absorb the burst.每小时 200 笔远低于限速——平均 0.06 QPS。但突发模式很关键：如果 50 笔支付同一秒到达，会撞到 10 QPS 上限导致筛查失败。我建议加 2 个点做令牌桶 + 重试队列来吸收突发。
Sarah (SM): Good catch. Updated estimate: 7 points for ONE-401.好眼力。ONE-401 改成 7 个点。
James (PO): OK, noted. Next — ONE-415. We need to store customer PII for 7 years for HKMA reporting. Three points.好，记下。下一个——ONE-415。我们把客户 PII 存 7 年给 HKMA 报送。3 个点。
You (TL): Wait. Which data fields are PII here? If it's just the HKID number, we can mask it at rest with AES-256. But if it's full name, address, DOB, and account number, we'll need field-level encryption plus a key rotation policy — that's more like 8 points, not 3.等等。哪些字段是 PII？如果只是香港身份证号，用 AES-256 静态加密就行。但如果是全名、地址、生日和账号，需要字段级加密加上密钥轮换策略——那要8 个点左右，不是 3 个。
James (PO): It's HKID + name + DOB — the minimum the regulator asks for.是香港身份证 + 姓名 + 生日——监管要求的最小字段集。
You (TL): OK, 3 PII fields is doable at 5 points — but I'd flag a dependency on InfoSec for the key rotation policy. If they don't sign off by sprint start, we can't ship. Can you raise that escalation this week?好，3 个 PII 字段 5 个点能做——但我得提个依赖给 InfoSec 签密钥轮换策略。如果 Sprint 开始前签不下来，我们交付不了。你这周能提一下这个 escalation 吗？

3. Key Phrases 关键句型

  • AC Can we walk through the acceptance criteria? 我们过一下验收标准？ Never accept a story with vague AC. Always walk through it line by line.
  • Quantify What's our expected peak QPS? 预期峰值 QPS 是多少？ Always ask for numbers in refinement. "A lot" doesn't size a story. "50 QPS" does.
  • Trade-off 5 points isn't enough for the rate-limiting work. 5 个点做限流不够 Challenge estimates with technical reasoning, not gut feel. "The rate limit forces a retry queue" wins.
  • Burst The burst pattern matters. 突发模式很关键 Average load hides burst load. Always ask "what's the peak-second rate, not just the average-hour rate?"
  • Scope Which data fields are PII here? 哪些字段是 PII？ PII scope drives the encryption scope. Always clarify the data fields before sizing the story.
  • Flag I'd flag a dependency on InfoSec for the key rotation policy. 提一个依赖给 InfoSec 签密钥轮换策略 Use "flag" for soft dependencies, "block" for hard ones. Both need to be raised before sprint start.
  • Escalate Can you raise that escalation this week? 你这周能提一下这个 escalation 吗？ Don't carry the escalation alone. Ask the PO to raise it — they have more political weight.

4. Vocabulary 词汇表

  • Acceptance criteria (AC) n. 验收标准 The list of conditions a story must satisfy to be "done".
  • QPS (queries per second) n. 每秒查询数 A measure of API throughput.
  • Rate limit n. 速率限制 A cap on the number of API calls allowed per second.
  • Token bucket n. 令牌桶 A rate-limiting algorithm that allows controlled bursts.
  • Retry queue n. 重试队列 A buffer that holds failed requests for later retry.
  • Burst pattern n. 突发模式 How requests cluster in time, not just the average rate.
  • PII (Personally Identifiable Information) n. 个人身份信息 Any data that can identify a specific person.
  • HKMA n. 香港金管局 Hong Kong Monetary Authority — the local banking regulator.
  • Field-level encryption n. 字段级加密 Encrypting individual fields in a database row, not just the whole row.
  • Key rotation policy n. 密钥轮换策略 Rules for how often encryption keys are replaced.
  • Spike n. 探索性工作 A short time-boxed investigation to reduce technical risk.
  • Estimation n. 估算 The team's collective guess of how many points a story is worth.

5. Grammar Notes 语法点

① "If it's just X, we can Y" — branching by scope

Use if X, then Y / but if Z, then W to size stories by scope. If it's just HKID, we can mask it. But if it's full name + DOB, we need field-level encryption. This is the cleanest way to defend a re-estimate.

用 if X, then Y / but if Z, then W 按 scope 估故事。

② "I'd flag a dependency on X" — soft authority

Use I'd flag / I'd raise / I'd recommend instead of we need / we must have. You're surfacing a risk, not demanding an action. The PO can decide whether to act on it.

用 I'd flag / I'd raise / I'd recommend 而不是 we need / we must have。

③ "The burst pattern matters" — present-tense emphasis

Use present-tense verbs (matters / drives / hides) to make a technical point hit hard. The burst pattern matters is more impactful than the burst pattern will matter.

用现在时动词（matters / drives / hides）让技术观点更有力。

6. Cultural Tips 文化 / 职场 Tips

  • Never accept a story without clear AC. "Implement sanctions screening" is not a story. "Implement sanctions screening with 10 QPS capacity and 99.5% success rate" is. Always push for the second.
  • Quantify in refinement, not in sprint. If the story goes into sprint without numbers, you'll discover the gap mid-sprint. Always size before commitment.
  • Burst beats average. Averages hide peaks. "200 per hour" sounds fine; "50 in one second" is the actual problem. Always ask for the burst pattern.
  • Soft dependencies need POs. "I have a soft dependency on InfoSec" is weaker than "PO, can you raise this escalation with InfoSec this week?". POs have political weight.
  • Frame re-estimates as scope-driven. "5 points isn't enough" is opinion. "5 points isn't enough because the rate limit forces a retry queue" is engineering. The second one wins.
  • PII stories are never just 3 points. Once you say "PII", you're signing up for encryption, key rotation, audit logs, and InfoSec review. Always add a flag for InfoSec dependency.
  • Challenge assumptions, not people. "5 points isn't enough because the rate limit forces a retry queue" is a technical challenge. "James, you underestimated this" is personal. Always choose the first.

7. Practice 练习题

Exercise 1 — Translate the refinement discussion

Translate the following backlog refinement into natural Chinese. Pay attention to how the TL questions the assumptions.

Exercise 2 — Spot the TL mistake

Which of these refinement lines is most "TL-like" and which is most "dev-like" or "PO-like"? Why?

  1. "Yesterday I finished the unit tests for the eKYC vendor callback. Coverage is at 91%."
  2. "Hold on, James. Can we walk through the acceptance criteria? What's our expected peak QPS?"
  3. "We need to integrate with Refinitiv's World-Check API for sanctions screening."
  4. "I'd add a 2-point spike for a token bucket + retry queue to absorb the burst."

② and ④ are TL-like (challenge AC, propose technical spike). ① is dev-like. ③ is PO-like (story statement, no challenge). A common mistake new TLs make is accepting the PO's framing without questioning. Always push for the AC and the burst pattern.

Exercise 3 — Your turn

You're in backlog refinement. The PO has brought a story: "We need to integrate with a third-party credit bureau API for real-time credit checks. Five points." Write your TL's challenge in English. Use at least 3 patterns from the Key Phrases section. Assume the bureau has a 5 QPS rate limit, your peak is 80 checks/minute with bursts of 20 in 1 second, and the response contains PII (name, HKID, income).

"Hold on, James. Can we walk through the acceptance criteria? Specifically, what's our expected peak QPS? The bureau has a published rate limit of 5 QPS per API key. 80 per minute is about 1.3 QPS average — well below the limit. But the burst pattern matters: if 20 checks arrive in the same second, we'll hit the 5 QPS cap and fail credit checks. I'd add a 2-point spike for a token bucket + retry queue. Also, which data fields are PII in the response? If it's just HKID, we can mask it. But if it's name, HKID, and income, we'll need field-level encryption plus a key rotation policy. I'd estimate 7 points, not 5. I'd flag a dependency on InfoSec for the key rotation policy — if they don't sign off by sprint start, we can't ship. Can you raise that escalation this week?"
