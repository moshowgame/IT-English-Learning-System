Code Review Summary Email

Article 11 · Module 3: Written Communication · Tech Lead English

⏱ 8 min read 🟡 Intermediate 🎯 Async written updates, structured review summaries, blockers up

1. Scenario 场景背景

It is Friday, 6:00 PM HKT. You are wrapping up the week. Three PRs are still open: PR #428 (idempotency-key generator), PR #431 (rate-limiter), and PR #435 (audit-log retention). You need to send a weekly code-review summary to the squad, your Chief Architect Margaret, and PM James.

周五晚6点香港时间。收尾这一周。还有 3 个 PR 开着：PR #428（幂等键生成器）、PR #431（限流器）、PR #435（审计日志保留）。你需要给 squad、首席架构师 Margaret、PM James 发一份本周 code review 摘要。

3. Key Phrases 关键句型

  • Subject Weekly code review summary — Week 27本周 code review 摘要——第 27 周Subject line is the only thing 80% of people read. Lead with what + when.
  • Structure Quick summary of the X open PRs and the Y that merged.X 个未合并 PR 和 Y 个本周已合并 PR 的简明摘要。Structure first: open vs merged. Readers scan the structure, not the prose.
  • Status icons ✅ / 🟡 / ⚠️✅ / 🟡 / ⚠️Icons are scannable. ✅ merged, 🟡 in flight, ⚠️ needs action.
  • Flag Flagged for Margaret / Flagged for James.请 Margaret 关注 / 请 James 关注Tag the action-owner in the email body. Don't make them guess.
  • Ask Can you weigh in by Monday?周一前能给个意见吗？End with the ask. 'FYI' emails without an ask get archived, not answered.
  • Blocker framing The current approach works but I'm not sure it scales to 50k TPS.当前方案能用，但我不确定能不能撑到 5 万 TPS。Frame the design question as a scaling concern, not a personal preference.

4. Vocabulary 词汇表

  • Code review summary n.. A weekly written digest of PR statuses and blockers. Code review 摘要
  • PR status n.. Where a PR stands: open, approved, merged, deployed. PR 状态
  • Regression n.. A new bug introduced by a code change. 回归
  • Canary n.. A deployment to a small percentage of traffic to test in production. 金丝雀发布
  • Design question n.. A PR comment that asks for a senior's view on the approach. 设计问题
  • Compliance review n.. A check by InfoSec/risk/compliance teams that a change meets regulations. 合规审核
  • ETA (Estimated Time of Arrival) n.. When something is expected to be done. 预计完成时间
  • TPS (Transactions Per Second) n.. A measure of system throughput. 每秒交易数
  • Flag for action v.. To mark an item as needing someone's attention. 标出需关注
  • FYI vs Ask n.. An 'FYI' is informational; an 'ask' requires action. 知会 vs 请求

5. Grammar Notes 语法点

① 'Quick summary of the X' — opening line

Open with quick summary of / quick update on / short note on. This sets the register: not a novel, just the highlights.

用 quick summary of / quick update on 开头，定调。

② 'Can you weigh in by Monday?' — the closing ask

Close every written update with a specific ask: can you weigh in / can you confirm / can you review. The closer is what gets action.

每封书面更新结尾用具体 ask。

③ 'The current approach works but…' — the design-question framing

Frame a design question as a scaling or risk concern, not a personal preference. It works, but I'm not sure it scales / I'm worried about the operational risk / I see a regulator exposure.

把设计问题框定为可扩展性或风险。

6. Cultural Tips 文化 / 职场 Tips

  • Subject line is everything. 80% of readers see the subject and decide whether to open. Lead with what + when.
  • Structure first: open vs merged. Readers scan the structure, not the prose.
  • Use icons (✅ 🟡 ⚠️) for scannability. A senior scans 20 emails a day. Make yours scannable.
  • Tag the action-owner in the body, not just in the To-line. 'Flagged for Margaret' is unmissable.
  • End with the ask. 'FYI' emails without an ask get archived.
  • Frame the design question as a scaling or risk concern. 'I'm not sure it scales' beats 'I don't like it'.
  • Keep it to 1 screen. If it's longer than 1 screen, the key items get lost.

Exercise 2 — Spot the email anti-pattern

Which of these weekly code review summaries is the most 'TL-as-async-communicator'?

  1. 'Hi all, lots of PRs this week. Reviewing them. Will update Monday.'
  2. 'Quick summary. 3 open, 2 merged. PR #428 has 3 blocking items. PR #431 needs Margaret's view. PR #435 needs James's confirmation. See comments for details. Thanks.'
  3. 'Long essay on every PR with full diff links and my opinions on each.'
  4. 'Quick summary. 3 open, 2 merged. PR #428 3 blocking items, author Lin Monday. PR #431 2 blocking + 1 design Q, flagged for Margaret by Monday. PR #435 compliance review ETA Tuesday, flagged for James.'

④ is the most scannable and action-oriented. ② is acceptable. ① is a non-update. ③ is a 5-screen novel. A common mistake is either (a) 'too short — no info' or (b) 'too long — 5 screens, key items lost'. The TL's job is to compress 50 PR comments into 1 screen.
