Banking Systems & Integration

Article 20 · Module 5: Banking Context · Tech Lead English

⏱ 8 min read 🔴 Senior 🎯 Core banking systems, integration patterns, ownership boundaries, migration risk

1. Scenario 场景背景

It is Friday, 11:00 AM HKT. You are the TL for the RBWM Digital tribe. The new OneWealth platform needs to integrate with 5 core banking systems: core banking (FCDB), customer master (GLCM), trading platform (Midas), treasury (Opics), and document store (Filenet). Steven (Group Architecture) is the system-owner liaison.

周五上午11点香港时间。你是 RBWM 数字化 tribe 的 TL。新的OneWealth 平台需要和 5 个银行核心系统集成：核心银行（FCDB）、客户主数据（GLCM）、交易平台（Midas）、资金（Opics）、文档存储（Filenet）。Steven（集团架构）是系统 owner 联络人。

3. Key Phrases 关键句型

  • Ownership boundary Each has a different ownership boundary, SLA, and integration pattern.每个有不同的 owner 边界、SLA、集成模式。When integrating with core systems, the first question is: who owns it, what is the SLA, what is the pattern.
  • Latency budget Latency budget: 100ms P99. 50ms P99. 500ms P99.延迟预算：100ms P99。50ms P99。500ms P99。P99 = 99th percentile latency. Bank integrations have tight P99 budgets. 50ms for trading is normal.
  • Sandbox Sandbox: yes. Sandbox: limited (market hours).沙箱：有。沙箱：有限（交易时段）。Sandbox availability affects test strategy. Midas has limited sandbox (market hours only) — plan tests accordingly.
  • Integration matrix Let me confirm the integration matrix.我确认一下集成矩阵。End every system walkthrough with an integration matrix. Owner, SLA, pattern, latency, sandbox.
  • Three constraints Three constraints: Midas 50ms, Opics batch window, Filenet async.三个约束：Midas 50ms、Opics 批量窗口、Filenet 异步。When system owners flag constraints, capture them as a numbered list. The list is your design input.
  • Async upload Do not wait for the upload to complete before responding to the user.别等上传完成再响应用户。Async upload is a UX pattern. The user gets a pending upload state; the system uploads in the background.
  • Design review with owners Design review with you and the 5 system owners the following week.下周跟你和 5 个系统 owner 做设计评审。Always include the system owners in the design review. Each owner signs off on their integration.

4. Vocabulary 词汇表

  • Core banking system n.. The system of record for customer accounts and balances. 核心银行系统
  • FCDB n.. The bank main core banking ledger. FCDB 核心账本
  • GLCM n.. Global Legal Customer Master — the canonical customer record. 全球客户主数据
  • Midas n.. A trading platform used in major banks for FX, rates, equities. Midas 交易平台
  • Opics n.. A treasury / risk management system. Opics 资金系统
  • Filenet n.. An enterprise document management system. Filenet 文档系统
  • P99 latency n.. 99th-percentile latency — the slowest 1% of requests. P99 延迟
  • TPS (Transactions Per Second) n.. A throughput measure. 每秒交易数
  • FIX protocol n.. A messaging standard for financial trading. FIX 协议
  • Batch window n.. The time period when batch jobs run (e.g., EOD reconciliation). 批量窗口

5. Grammar Notes 语法点

① 'Each has a different X, Y, Z' — the system-owner walkthrough

When the system owner walks you through integrations, listen and capture: owner, SLA, integration pattern, latency, sandbox. The matrix is your design input.

系统 owner 走查时，记 5 件事：owner、SLA、集成模式、延迟、沙箱。

② 'Latency budget: 100ms P99' — quantifying latency

Quantify latency: 100ms P99, 50ms P99, 500ms P99. P99 is the standard bank latency metric. Numbers beat adjectives.

用 P99 量化延迟。

③ 'Three constraints: X, Y, Z' — the design input list

When system owners flag constraints, capture them as a numbered list. Three constraints: Midas 50ms, Opics batch window, Filenet async.

把 owner 提的约束整理成编号清单。

6. Cultural Tips 文化 / 职场 Tips

  • When integrating with core systems, the first question is: who owns it. Owner, SLA, pattern, latency, sandbox.
  • Quantify latency in P99. 100ms P99 for core banking is normal. 50ms P99 for trading is tight.
  • Sandbox availability affects test strategy. Midas has market-hours-only sandbox — plan tests accordingly.
  • End every walkthrough with an integration matrix. Owner, SLA, pattern, latency, sandbox.
  • Capture constraints as a numbered list. The list is your design input.
  • Async upload is a UX pattern. User gets pending upload state; system uploads in background.
  • Always include system owners in the design review. Each owner signs off on their integration.

Exercise 2 — Spot the integration mistake

Which of these is the most TL-as-integration-aware?

  1. We will just call the APIs directly. Easy.
  2. I will capture the integration matrix: FCDB (REST+Kafka, 100ms P99, 200 TPS), GLCM (REST, 200ms P99, 1k TPS), Midas (REST+FIX, 50ms P99, market-hours sandbox), Opics (batch EOD + REST, 1s P99), Filenet (REST+SFTP, 500ms P99 retrieval). Three constraints: Midas 50ms tight, Opics batch window 22:00-02:00, Filenet async upload.
  3. Sandbox? We can just test in production.
  4. We will figure out the SLAs after we go live.

② is the integration-aware TL. ① ignores the matrix. ③ violates no testing in production (regulator-reportable). ④ is post-hoc. A common mistake is treating core-system integration as just calling APIs — but each system has unique SLA, latency, and sandbox constraints.
