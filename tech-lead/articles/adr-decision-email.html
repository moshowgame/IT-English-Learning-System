Architecture Decision Record (ADR) Email

Article 12 · Module 3: Written Communication · Tech Lead English

⏱ 8 min read 🟠 Advanced 🎯 Decision announcement, context + decision + consequences, addressing dissent

1. Scenario 场景背景

It is Tuesday, 11:00 AM HKT. The chapter review for ADR-0142 (OneWealth API Gateway strategy) concluded. The decision is final: AWS API Gateway with mutual TLS for partner-facing traffic only. Raj voted against — he wanted to include internal traffic. Diego was neutral. Margaret approved. You are sending the formal decision email to the wider RBWM Digital tribe.

周二上午11点香港时间。ADR-0142（OneWealth API Gateway 策略）的 chapter 评审结束。决策定稿：合作方侧用 AWS API Gateway + 双向 TLS。Raj 投了反对——他希望把内部流量也包含进来。Diego 中立。Margaret 批准。你正在向更广的 RBWM 数字化 tribe 发正式的决策邮件。

3. Key Phrases 关键句型

  • Subject ADR-0142 final decision — OneWealth API Gateway strategyADR-0142 最终决策——OneWealth API Gateway 策略Subject = ADR number + 'final decision' + topic. Scannable, specific.
  • TL;DR TL;DR: AWS API Gateway with mTLS for partner-facing traffic only.TL;DR：合作方侧用 AWS API Gateway + mTLS。Always lead with a TL;DR. Most readers will only read the TL;DR.
  • Context We evaluated three options for north-south API traffic.我们评估了 3 个方案用于南北向 API 流量。Context = the alternative options that were considered. Skipping the context turns the ADR into a 'because I said so'.
  • Decision AWS API Gateway with mTLS, partner-facing traffic only.AWS API Gateway + mTLS，仅合作方流量。The decision is one short sentence. If it takes more than one sentence, you don't have a decision.
  • Rationale Rationale: best security alignment, lowest time-to-market, lowest operational risk.理由：安全对齐最佳、上市最快、运维风险最低。Rationale is the 'why'. 3 short reasons. No essays.
  • Consequences Consequences: who is impacted, what changes, hard dependencies.影响：谁受影响、什么变化、硬依赖。Consequences = the action items. The decision is not done until consequences are listed.
  • Dissent Raj raised a valid concern — however, we will revisit in ADR-0150.Raj 提了合理关切——然而，我们会在 ADR-0150 重审。Address dissent explicitly. 'Valid concern, however, X' is the right register. Burying dissent breeds resentment.
  • Lock After Friday, the decision is locked.周五后，决策锁定。Set a lock date. ADR emails without a lock date invite endless re-debate.

4. Vocabulary 词汇表

  • ADR (Architecture Decision Record) n.. A short document capturing a decision + context + consequences. 架构决策记录
  • Chapter review n.. The architecture community of practice's review meeting. Chapter 评审
  • TL;DR n.. Too Long; Didn't Read — a one-sentence summary. 一句话摘要
  • Rationale n.. The reasoning behind a decision. 理由
  • Consequences n.. The downstream effects of a decision. 影响
  • North-south traffic n.. Traffic between client and server (vs. east-west = service-to-service). 南北向流量
  • Private API mesh n.. An internal service-to-service API layer. 私有 API mesh
  • mTLS (mutual TLS) n.. Both client and server present certificates. 双向 TLS
  • Group Cyber n.. The bank's central cybersecurity function. 集团网安
  • Lock date n.. The date after which a decision can no longer be re-debated. 锁定日期

5. Grammar Notes 语法点

① 'TL;DR: X' — the one-sentence summary

Open with TL;DR: [decision in one sentence]. Most executives and stakeholders will only read the TL;DR.

用 TL;DR: [决策] 开头。

② 'X raised a valid concern — however, Y' — addressing dissent

Address dissent explicitly. X raised a valid concern / X's point is well-taken — however, [reasoning]. We will revisit in [next ADR / Q4].

显式回应反对意见。

③ 'After Friday, the decision is locked' — the lock

End with a lock date. After [date], the decision is locked. Reply-all by [date]. The lock date prevents endless re-debate.

结尾加锁定日期。

6. Cultural Tips 文化 / 职场 Tips

  • Subject = ADR number + 'final decision' + topic. Scannable, specific.
  • Always lead with TL;DR. Most readers will only read the TL;DR.
  • Context = the alternatives considered. Skipping the context turns the ADR into 'because I said so'.
  • Decision is one short sentence. If it takes more than one sentence, you don't have a decision.
  • Rationale = 3 short reasons. No essays.
  • Consequences = who/what/dependencies. The decision is not done until consequences are listed.
  • Address dissent explicitly. 'Valid concern, however, X' is the right register.
  • Set a lock date. ADR emails without a lock date invite endless re-debate.

Exercise 1 — Translate the ADR email

Translate the following ADR email into natural Chinese.

"Subject: ADR-0142 final decision — OneWealth API Gateway strategy. Hi RBWM Digital tribe. ADR-0142 (OneWealth API Gateway strategy) has been finalised after chapter review yesterday. TL;DR: AWS API Gateway with mTLS for partner-facing traffic only. Internal RBWM traffic stays on the existing private API mesh. Context: We evaluated three options for north-south API traffic: AWS API Gateway with mTLS, Kong with mTLS, and extending the existing on-prem gateway. Scoring was on 4 criteria: time-to-market, cost, operational fit, security alignment. Decision: AWS API Gateway with mTLS, partner-facing traffic only. Rationale: best security alignment with Group Cyber, lowest time-to-market, lowest operational risk. Consequences: Partner-facing squads design your integrations against AWS API Gateway, target go-live Q3. Internal squads: no change. We will revisit internal traffic in ADR-0150 (Q4) — Raj is on the working group. Read the full ADR here: [link]. Questions or push-back: reply-all by Friday. After Friday, the decision is locked. Thanks all. Wei (TL, RBWM Digital)."

"主题：ADR-0142 最终决策——OneWealth API Gateway 策略。RBWM 数字化 tribe 大家好。ADR-0142（OneWealth API Gateway 策略）昨天 chapter 评审后定稿。TL;DR：合作方侧用 AWS API Gateway + mTLS。RBWM 内部流量继续走现有的私有 API mesh。背景：我们评估了 3 个方案用于南北向 API 流量：AWS API Gateway + mTLS、Kong + mTLS、扩展现有 on-prem 网关。按 4 个标准打分：上市时间、成本、运维契合度、安全对齐度。决策：AWS API Gateway + mTLS，仅合作方流量。理由：与 Group Cyber 安全对齐最佳、上市最快、运维风险最低。影响：合作方 squad：按 AWS API Gateway 设计集成，Q3 上线。内部 squad：无变化。InfoSec：双向 TLS 证书轮换策略是硬依赖，预计周五。Raj 提了合理关切——然而，我们没有证据说现有私有 mesh 不行。我们会在 ADR-0150（Q4）重审内部流量——Raj 在工作小组里。完整 ADR 见：[Confluence 链接]。有疑问或反对：周五前 reply-all。周五后，决策锁定。各位辛苦。Wei（TL，RBWM 数字化）。"

Exercise 2 — Spot the ADR mistake

Which of these ADR emails is the most 'TL-as-decision-driver'?

  1. 'Hi all, we decided to go with AWS. Thanks.'
  2. 'Hi all, we considered AWS, Kong, and on-prem. We're going with AWS. Read the wiki for details.'
  3. 'Hi all, TL;DR: AWS API Gateway + mTLS, partner-facing only. Context: 3 options evaluated. Decision: AWS + mTLS, partner-facing. Rationale: best security, lowest TTM, lowest ops risk. Consequences: partner squads target Q3; InfoSec mTLS rotation hard dep, ETA Friday. Dissent: Raj wanted internal; addressed — revisit in ADR-0150 Q4. Reply-all by Friday. After Friday, locked. — Wei.'
  4. 'Hi all, Raj thinks we should use Kong. I think we should use AWS. Let's discuss.'

③ is the model ADR. ① has no context. ② has context but no rationale/consequences/lock date. ④ is a debate, not a decision. A common mistake is calling the decision 'final' but not setting a lock date. Without a lock date, the decision isn't final — it's just the latest opinion.

Exercise 3 — Your turn

Your chapter just approved ADR-0201 (use PostgreSQL with read-replica for the new audit ledger). The alternatives considered were: (1) PostgreSQL with read-replica, (2) Cassandra, (3) MongoDB. Scoring was on 4 criteria: time-to-market, cost, scalability, operational fit. The decision: PostgreSQL with read-replica, 2-sprint POC first, then full migration. Mei wanted MongoDB. Write the ADR email in English. Include TL;DR, context, decision, rationale, consequences, dissent handling, and a lock date.

"Subject: ADR-0201 final decision — Audit ledger on PostgreSQL. Hi RBWM tribe. ADR-0201 finalised after chapter review. TL;DR: PostgreSQL with read-replica, 2-sprint POC, then full migration. Context: Evaluated PostgreSQL with read-replica, Cassandra, and MongoDB. Scoring: time-to-market, cost, scalability, operational fit. Decision: PostgreSQL with read-replica, 2-sprint POC, then full migration. Rationale: best time-to-market, lowest cost, sufficient scalability for our 5-year projection, mature operational fit. Consequences: Audit squad designs against PostgreSQL, POC target Q2, full migration Q3. InfoSec: encryption-at-rest hard dep, ETA Friday. Mei wanted MongoDB — valid concern about document flexibility, however, our access pattern is 95% keyed lookup, not document scan. We will revisit in ADR-0210 (Q4) if access patterns change. Read the full ADR: [link]. Reply-all by Friday. After Friday, locked. — Wei."
