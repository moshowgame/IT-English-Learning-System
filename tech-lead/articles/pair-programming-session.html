Pair Programming Session

Article 09 · Module 2: Design & Code Quality · Tech Lead English

⏱ 8 min read 🟡 Intermediate 🎯 Driver/navigator rhythm, teaching vs gatekeeping, async handoff

1. Scenario 场景背景

It is Thursday, 2:00 PM HKT. You are pairing with Lin, a junior developer, on a tricky bug: the idempotency key collision in the payment service, where 1 in 10,000 transactions is generating a duplicate key. You have 90 minutes before your next meeting.

周四下午2点香港时间。你和 junior Lin 配对修一个棘手 bug：支付服务的幂等键冲突，1/10000 的交易生成了重复键。下一个会议前你只有 90 分钟。

2. Dialogue 英文对话

You (TL):OK Lin, let's pair on this. You drive, I navigate. Switch every 15 minutes.好 Lin，我们配对一下。你写，我导航。每 15 分钟切换一次。
Lin:Wait, the user said 1 in 10,000. That sounds like a race condition. Where do we start?等等，用户说是 1/10000。这听起来像竞态条件。从哪里开始？
You (TL):Good instinct. Start with the reproduction. Don't guess the root cause yet. Get a test that fails 1 in 10,000 times.直觉不错。先复现。先别猜根因。先写一个 1/10000 失败的测试。
Lin:How do I reliably reproduce a 1 in 10,000 race?怎么稳定复现 1/10000 的竞态？
You (TL):Stress test with 1 million calls in a tight loop. If the race exists, you'll see it. Run it 5 times in parallel to amplify the chance.紧凑循环里压 100 万次调用。如果竞态存在，你会看到。并行跑 5 次放大机会。
Lin (typing):OK, I'm writing the stress test now… Got the failure! It's deterministic when we run 5x in parallel. The collision happens when 2 threads enter the key-gen block at the same nanosecond.OK，我在写压测……复现成功了！并行跑 5 次是确定性的。两个线程在同一个纳秒进入 key-gen 块时就冲突。
You (TL):Excellent. That's the root cause. What's the fix?漂亮。这就是根因。怎么修？
Lin:I think we need atomic increment on the counter. Use synchronized block or AtomicLong?我觉得需要对计数器做原子自增。用 synchronized 块还是 AtomicLong？
You (TL):AtomicLong is cleaner. Lock-free, single-counter hot path. Write a test that fails with the old code, passes with AtomicLong. Show me the diff.AtomicLong 更干净。无锁，单计数器热路径。写个测试，旧代码失败、AtomicLong 通过。给我看 diff。
Lin (after 10 min):Done. Test passes, stress test passes 10/10.好了。测试通过，压测 10/10 过。
You (TL):Nice work. Document the race in a comment — future maintainers need to know why it's AtomicLong, not just that it is. Open the PR and Slack me the link.漂亮。在注释里把竞态记下来——以后的维护者需要知道为什么用 AtomicLong，不只是用了。提 PR，链接发我 Slack。

3. Key Phrases 关键句型

  • Roles You drive, I navigate. Switch every 15 minutes.你写，我导航。每 15 分钟切换。Always declare driver/navigator. Unstructured pairing becomes silent rubber-ducking.
  • Hypothesis Good instinct.直觉不错。Acknowledge the junior's hypothesis before steering. Otherwise you're just lecturing.
  • Order Start with the reproduction. Don't guess the root cause yet.先复现，先别猜根因。Reproduction first, hypothesis second. The reverse is how juniors spend a day chasing ghosts.
  • Method Stress test with 1 million calls in a tight loop.紧凑循环里压 100 万次调用。Show the method, not the answer. 'How do I prove this exists?' is a better lesson than 'use AtomicLong'.
  • Elicit What's the fix?怎么修？Elicit the answer from the junior, don't dictate. Pairing = teaching, not typing-for.
  • Why Document the why, not just the that.记下为什么，不只是做了什么。Comments that explain the why are 10x more valuable than comments that explain the what.

4. Vocabulary 词汇表

  • Pair programming n.. Two developers working on the same code at the same time. 配对编程
  • Driver n.. The person typing the code in a pair. 驾驶员
  • Navigator n.. The person reviewing and steering the code in a pair. 导航员
  • Reproduction n.. A test that reliably triggers the bug. 复现
  • Stress test n.. A test that pushes the system to its limits to expose failures. 压测
  • Race condition n.. A bug where the output depends on unpredictable timing. 竞态条件
  • Atomic operation n.. An operation that completes in a single, uninterruptible step. 原子操作
  • AtomicLong n.. A Java class for lock-free thread-safe long counters. Java 原子长整型
  • Hot path n.. The most-executed code in a system, where performance matters most. 热路径
  • Lock-free adj.. Concurrency design that avoids explicit locks for performance. 无锁

5. Grammar Notes 语法点

① 'You drive, I navigate' — the role declaration

Open every pair session with you drive, I navigate / let's switch every 15 min. The role declaration prevents silent rubber-ducking.

每次配对开头声明角色。

② 'What's the fix?' — eliciting over dictating

Use what's the fix / how would you approach this / what does the test look like to elicit. Pairing is teaching, not typing-for.

用 what's the fix / how would you approach this 引导。

③ 'Document the why, not just the that' — the junior's growth

Use document the why, not just the that / explain the rationale, not the rule in code review and pair review.

用 document the why, not just the that。

6. Cultural Tips 文化 / 职场 Tips

  • Always declare driver/navigator. Unstructured pairing becomes silent rubber-ducking.
  • Acknowledge the junior's hypothesis before steering. Otherwise you're lecturing, not pairing.
  • Reproduction first, hypothesis second. The reverse is how juniors spend a day chasing ghosts.
  • Show the method, not the answer. 'How do I prove this exists?' is a better lesson than 'use AtomicLong'.
  • Elicit, don't dictate. 'What's the fix?' is teaching. 'Use AtomicLong' is lecturing.
  • Document the why. Comments that explain the why are 10x more valuable than comments that explain the what.
  • End with the next action. 'Open the PR and Slack me the link' is a clean handoff.

Exercise 2 — Spot the pair mistake

Which of these is the most 'TL-as-teacher' pair behaviour?

  1. 'Lin, watch me type. This is how you fix a race.'
  2. 'Good instinct. Start with the reproduction. What's the fix? Show me the diff.'
  3. 'AtomicLong. Done. Next ticket.'
  4. 'Lin, can you watch while I fix this in 5 minutes?'

② is TL-as-teacher. ① and ④ are TL-as-typist. ③ skips the teaching entirely. A common mistake is taking the keyboard away from the junior to 'just fix it'. That saves 10 minutes and costs 10 weeks of growth.

Exercise 3 — Your turn

You're pairing with a junior on a NPE in a Spring Boot service. The NPE only happens in prod, never locally. Write your opening lines as a TL: declare roles, ask for the junior's hypothesis, propose the next debugging step. Write in English.

"OK Lin, let's pair on this. You drive, I navigate. Switch every 15 minutes. The NPE only happens in prod, never locally — what's your first hypothesis? Good instinct. Let's pull the prod logs and look at the stack trace first. Don't guess yet — let's see what the JVM actually says. Show me the relevant log lines. Got it. Now, what does the local test environment look like? How is it different from prod? Good. Document the difference in a comment in the code — future maintainers need to know why this only blows up in prod, not just that it does. Open the PR and Slack me the link."
