KYC / AML Business Rules & UAT

Article 20 · Module 5: HSBC Banking Context · BA English

⏱ 12 min read 🔴 Advanced 🧪 Run UAT for KYC refresh, sanctions screening, and EDD triggers

1. Scenario 场景背景

It is UAT week on the OneScreening 2.4 release — the periodic KYC refresh and sanctions screening engine upgrade for the GPB Onboarding tribe. You are the BA running the User Acceptance Test (UAT) alongside Pradeep (QA Lead, Bangalore), Charlotte (Compliance Tester, London), and Roberto (the business lead, Miami). The UAT environment was promoted on Sunday. You have forty-two test scripts to run in five days, with three regulator sign-offs needed before the production release on Friday 18:00 GMT.

到了 OneScreening 2.4 发布版本 的 UAT 周——GPB 开户 tribe 周期 KYC 刷新 和制裁名单筛查引擎的升级。你是 BA，和 Pradeep（QA 负责人，班加罗尔）、 Charlotte（合规测试员，伦敦）、Roberto（业务负责人，迈阿密） 一起跑 UAT。UAT 环境周日上线。你有 42 个测试脚本，5 天内必须跑完，并在周五 18:00 GMT 生产发布前拿到三个监管签字。

Day 2 at 11 AM HKT: three P1 defects land in the daily triage — the EDD trigger fires twice for a PEP match, the sanctions screening misses a fuzzy name match for "Mohammed" vs "Muhammad", and the periodic-refresh batch job skips clients with a NULL risk rating. You have to triage live, in English, with the dev team in Hyderabad and Compliance in London.

第 2 天上午 11 点香港时间：每日 triage 会议上来 3 个 P1 缺陷——EDD 触发对 PEP 匹配重复触发 两次；制裁筛查对"Mohammed"和"Muhammad"的模糊匹配漏报；定期刷新批处理跳过 风险评级为 NULL 的客户。你必须和海得拉巴的开发、伦敦的合规一起用英语现场 triage。

2. Dialogue 英文对话

Pradeep (QA Lead, Bangalore): Morning, all. Quick one — yesterday's run surfaced three P1s on the test report. Let's triage live — Wei, do you want to walk us through the AC impact?大家早上好。简单说一下——昨天的测试报告上出现了 3 个 P1。我们现场 triage——Wei，你先讲一下对 AC 的影响？
You (BA): Sure, let me take them in order. P1-A: the EDD workflow fires twice for a single PEP match — the AC says "trigger once and only once", so this is a direct AC failure. P1-B: the sanctions screening missed a fuzzy match between "Mohammed" and "Muhammad" — that one has a MAS TRM implication, so it's not just a bug, it's a potential control failure. P1-C: the periodic-refresh job skipped clients with a NULL risk rating — that's about 4% of the book.好，我按顺序讲。P1-A：单个 PEP 匹配下 EDD 工作流触发两次——AC 是"仅触发一次"，所以这是直接的 AC 失败。P1-B：制裁筛查漏掉了"Mohammed"和"Muhammad"的模糊匹配——这有 MAS TRM 含义，所以不只是 bug，是潜在的控制失效。P1-C：定期刷新作业跳过了风险评级为 NULL 的客户——约占客户群的 4%。
Charlotte (Compliance Tester, London): On P1-B — from a control perspective, that's a walk-back. If MAS sees this in the audit trail, we'll have an observation letter. What's the proposed fix?关于 P1-B——从控制角度看，这是"回退"。如果 MAS 在审计追踪里看到，会发观察函。提议怎么修？
You (BA): Akash from the dev team is on the call. Akash — can you walk us through the matching algorithm and whether the fuzzy threshold can be tightened for Arabic transliterations?开发的 Akash 在会上。Akash——能讲一下匹配算法，以及阿拉伯语音译的模糊阈值能否收紧吗？
Akash (Dev, Hyderabad): Sure. The current threshold is a Jaro-Winkler score of 0.85. For transliterations like Mohammed / Muhammad, we're scoring around 0.78, which is just below the cut-off. We can either lower the threshold to 0.75 or add an alias list for the top 50 transliterations. Trade-off: lower threshold = more false positives; alias list = more manual maintenance.好。目前阈值是 Jaro-Winkler 0.85。对于 Mohammed / Muhammad 这种音译，分数在 0.78 左右，恰好低于阈值。两个选项：把阈值降到 0.75，或者给最常见的 50 个音译加别名列表。权衡：阈值降会产生更多误报，别名列表需要更多人工维护。
You (BA): From a UAT perspective, my recommendation is the alias list — it's deterministic and auditable. Charlotte, does that hold for Compliance?从 UAT 角度，我推荐别名列表——它是确定性的、可审计的。Charlotte，合规这边 ok 吗？
Charlotte (Compliance Tester, London): Yes, alias list is preferred. However, we need the alias list to be reviewed by FCC at least every six months, and we need the change logged in the Model Risk register as a parameter change, not a model change.好，别名列表更可取。但是，别名列表需要至少每 6 个月由 FCC 审查一次，并且变更需要记入模型风险登记簿，归为参数变更而非模型变更。
Roberto (Business Lead, Miami): On P1-C — the NULL risk rating, that's a data quality issue from the legacy CRM, right? Can we just default them to medium in the rule engine, rather than skipping them?关于 P1-C——NULL 风险评级是老 CRM 的数据质量问题对吧？能否在规则引擎里默认设为"中"，而不是跳过？
You (BA): Good question, Roberto. Per the AC, the engine should not skip NULL clients — it should default to high and trigger EDD. That's the regulatory safer option. Quick check — Akash, is the rule in code or in the BRD rule table?Roberto，好问题。根据 AC，引擎不应跳过 NULL 客户——应默认为高并触发 EDD。这是监管上更稳妥的选项。快速确认——Akash，这条规则在代码里还是在 BRD 规则表里？
Akash (Dev, Hyderabad): It's in the rule table. So the fix is a one-line configuration change — no code deploy needed. I can have the updated table in UAT by tomorrow.在规则表里。所以改起来是一行配置变更——不需要代码部署。我明天可以把更新后的表放 UAT。
You (BA): Perfect. To wrap up: P1-A needs a root cause analysis (RCA) by EOD, P1-B gets the alias-list fix with the Model Risk parameter change logged, P1-C is a one-line rule table change tomorrow. I'll re-execute the affected test scripts on Thursday, and we'll do a final go/no-go on Friday morning. Charlotte, can you confirm the Compliance sign-off is on track for Friday 12:00 GMT?完美。总结一下：P1-A 需要 EOD 前出 RCA；P1-B 走别名列表修复，模型风险记为参数变更；P1-C 是规则表一行配置变更明天上。我会在周四重跑受影响的测试脚本，周五上午做最终 go/no-go 决策。Charlotte，能确认合规签字能在周五 12:00 GMT 前到位吗？

3. Key Phrases 关键句型

  • Triage Let's triage live. 我们现场 triage 一下 "Triage" = sort defects by severity, owner, and fix path. A UAT ritual. Always run live with the dev team present.
  • AC failure That's a direct AC failure. 这是直接的 AC 失败 "AC failure" is the strongest defect classification. A P1 that breaks AC is a release blocker — no negotiation.
  • Control From a control perspective, that's a walk-back. 从控制角度看，这是回退 "Control" = a regulatory safeguard. "Walk-back" = a regression. Compliance uses these to push P1s to P0.
  • Trade-off Lower threshold = more false positives; alias list = more manual maintenance. 降阈值会有更多误报，别名列表需要更多人工维护 Always present a trade-off, not a single fix. Lets the room choose — and shows you've thought it through.
  • Deterministic The alias list is deterministic and auditable. 别名列表是确定性的、可审计的 "Deterministic" = same input always gives the same output. Compliance loves deterministic — random / probabilistic is hard to audit.
  • Parameter change It should be logged as a parameter change, not a model change. 应记为参数变更，不是模型变更 "Parameter change" is a lighter Model Risk approval (days). "Model change" needs full governance (weeks). Choose carefully.
  • Default The engine should default to high and trigger EDD. 引擎应默认为高并触发 EDD When in doubt, default to the safer regulatory option. "Skip" is almost never acceptable in compliance UAT.
  • RCA We need a root cause analysis by EOD. 我们需要在 EOD 前出 RCA "RCA" = Root Cause Analysis. The 5-whys or fishbone exercise. Required for every P1 before a release decision.
  • Re-execute I'll re-execute the affected test scripts on Thursday. 我会在周四重跑受影响的测试脚本 After a fix, never "trust" the change — re-execute the script. "Re-execute" is QA's preferred verb.
  • Go / no-go We'll do a final go/no-go on Friday morning. 周五上午做最终 go/no-go 决策 "Go/no-go" = the binary release decision. Always scheduled in advance, with named attendees and a clear escalation path.

4. Vocabulary 词汇表

  • UAT (User Acceptance Testing) n. 用户验收测试 The final test phase before production. Owned by the business, not by IT.
  • AC (Acceptance Criteria) n. 验收标准 The bullet list of "done" conditions in a user story. A P1 that breaks AC is a release blocker.
  • P1 / P2 / P3 n. 缺陷等级 P1 = release blocker; P2 = must-fix-this-sprint; P3 = nice-to-have.
  • Defect triage n. 缺陷分类 The daily meeting to classify, prioritise, and assign defects. Run live with the dev team.
  • RCA (Root Cause Analysis) n. 根因分析 The exercise to identify why a defect happened. Required for every P1.
  • Jaro-Winkler n. 字符串相似度算法 A common string-similarity algorithm for fuzzy name matching. Score 0-1, higher = more similar.
  • Fuzzy match n. 模糊匹配 A match between two strings that are similar but not identical. Critical for sanctions screening.
  • False positive / false negative n. 误报 / 漏报 False positive = clean client flagged. False negative = risky client missed. Regulators hate the second.
  • Rule table n. 规则表 The externalised business rules (e.g. Drools, OPA). A change to the table is a parameter change, not a code change.
  • Model Risk register n. 模型风险登记簿 HSBC's central log of all model and parameter changes, owned by Group Model Risk.
  • Parameter change vs model change n. 参数变更/模型变更 Parameter = tuning an existing model (faster approval). Model = new algorithm (heavy approval).
  • Go / no-go n. 发布决策 The final binary decision to release to production. Scheduled in advance, owned by the release manager.
  • OneScreening n. 汇丰统一筛查平台 HSBC's unified KYC and sanctions screening engine. Consolidates the legacy ACME and WorldCheck stacks.
  • BRD (Business Requirements Document) n. 业务需求文档 The formal document that captures the business rules. The rule table is the operational expression of the BRD.

5. Grammar Notes 语法点

① "It should trigger" / "It should not skip" — the regulatory should/should not

Compliance UAT loves the should / should not modal pair. The engine should default to high and trigger EDD — the system has an obligation. The engine should not skip NULL clients — an explicit prohibition. Use these in AC for unambiguous test assertions. Must is reserved for regulator-defined absolutes; should is for your own design decisions.

合规 UAT 的 AC 用 should / should not 表达系统义务和禁止——比 must 更适合设计决策。

② "I can have" / "I'll have" — the BA's "I will deliver" pattern

I'll have the updated table in UAT by tomorrow. "I'll have X by Y" is a stronger, more time-bound commitment than I'll try to get X done. Use it to close out triage decisions — it creates the audit trail and the social commitment.

I'll have X by Y 比"我尽量"更有时间承诺感——triage 收尾时用这个句型。

③ "That's a control failure" — the "that's a X" identification pattern

The pattern that's a + classification noun (control failure, AC failure, model change, parameter change) is the standard way to label an issue in triage. Once labelled, the room knows how to react. Get comfortable using the noun the regulator uses.

triage 中用 that's a + 分类名词 给问题贴标签——会议室会知道如何反应。

6. Cultural Tips 文化 / 职场 Tips

  • The BA runs triage, not the QA lead. In HSBC squads, the BA is the AC authority. The QA Lead runs the test execution, but the BA decides whether a defect is "in scope" of AC. If you don't chair the triage, you'll be overridden by the loudest dev in the room.
  • "Skip" is a four-letter word in compliance UAT. Never let "skip" through as a fix for a NULL or missing field. Default to the safer option (high risk rating, manual review) and raise a follow-on story to clean the data.
  • Parameter change vs model change is a 10x time difference. A parameter change is a 3-day Model Risk review. A model change is a 30-day one. If your fix is a threshold or a list, make sure it's classified as a parameter change, not a model change. Ask the Group Model Risk team if in doubt.
  • Compliance testers don't sign off on technical correctness. They sign off on whether the AC satisfies the regulatory intent. Don't ask Charlotte "does the algorithm work" — ask "does this fix satisfy the MAS TRM control standard".
  • Cross-region UAT means someone is always late or early. With Bangalore, Hyderabad, London, Miami, and HK on a UAT call, the standup time will inconvenience someone. Rotate the time fairly across releases — it builds trust across the tribe.
  • Document the "go/no-go" decision in writing — same day. Even a 2-line email: "Release 2.4 signed off at 12:30 GMT by Charlotte, Roberto, Wei. Defects: 0 P1 open, 2 P2 deferred to 2.4.1." That email is what Audit reads first.
  • Test the negative cases, not just the positive ones. A UAT that only tests "EDD triggers for a PEP" misses the bug that fires EDD twice. Always script the "what if the data is NULL", "what if the alias list is empty", "what if the threshold is exactly the cut-off".
  • Alias lists must be reviewed every 6 months, not once. A common squad mistake is to fix the fuzzy match and forget the maintenance. Bake the 6-month review into the operational runbook before you release.

Exercise 1 — Translate the triage

Translate the following BA summary into natural Chinese. Pay attention to the trade-off framing and the go/no-go commitment.

"To wrap up: P1-A needs a root cause analysis by EOD. P1-B gets the alias-list fix with the Model Risk parameter change logged. P1-C is a one-line rule table change tomorrow. I'll re-execute the affected test scripts on Thursday, and we'll do a final go/no-go on Friday morning. Charlotte, can you confirm the Compliance sign-off is on track for Friday 12:00 GMT? From a control perspective, the alias list is deterministic and auditable, which is the regulator-friendly option."

"总结一下：P1-A 需要 EOD 前出根因分析。P1-B 走别名列表修复，模型风险记为参数变更。P1-C 是规则表一行配置变更，明天上。我会在周四重跑受影响的测试脚本，周五上午做最终 go/no-go 决策。Charlotte，能确认合规签字能在周五 12:00 GMT 前到位吗？从控制角度看，别名列表是确定性的、可审计的，是监管更友好的选项。"

Exercise 2 — Spot the UAT mistake

During UAT triage, which BA response is the strongest "release-ready" pattern?

  1. "Let's just skip the NULL clients for now and clean the data next sprint."
  2. "The engine should not skip NULL clients. It should default to high and trigger EDD. Per the AC, that's the regulatory safer option. Akash, can you confirm the rule is in the BRD rule table and apply the one-line change?"
  3. "I'm not sure — let's ask the dev team."
  4. "Maybe we can ship without fixing it and add it to 2.5?"

② is the strongest. It cites the AC, the regulatory principle (default to safer), assigns the owner (Akash), and asks for a specific confirmation (rule table, not code). ① and ④ both propose "skip" or "defer" — patterns that rarely survive Compliance review. ③ abdicates the BA's AC authority.

The business lead is on the call. Write a 5-7 sentence BA response that (a) labels the defect against AC, (b) distinguishes the duplicate-send (release blocker) from the address-update (data issue), (c) proposes a triage path. Use at least 3 patterns from the Key Phrases section.

"OK, let me triage this live. The AC for the periodic-refresh story says: 'send a single reminder to the client's registered email, with a 7-day expiry'. The duplicate send is a direct AC failure — the system is firing twice on a single refresh cycle, which is a release blocker. I'd call that an AC failure, and it should be P1. The address-update piece is a separate issue: if the consent ledger has the old email, that's a data quality problem in the consent ledger, not a bug in the refresh job. I'd split it: P1 fix the duplicate send before Friday go/no-go; raise a follow-on P2 story to scope the consent-ledger address-validation logic. Akash, on the duplicate send — is the trigger in the rule table or in code? If it's in the rule table, can you have the fix in UAT by tomorrow EOD? I'll re-execute test script T-238 after the fix lands. Roberto, on the data side — can you confirm with the data steward whether the consent-ledger address-validation is in flight under a different squad, so we don't double-track? I want to make sure the Friday go/no-go is a clean decision, not a 'fix it in production' decision."

8. Module 5 Wrap-up — HSBC Banking Context

This article closes Module 5: HSBC Banking Context (Articles 18-20). The three articles together form the BA's "compliance toolkit" — the language and the discipline you need when requirements meet regulators.

Article 18 — Compliance & Regulators (回顾)

  • Core pattern: Cite the regulator with the specific standard (per MAS TRM, per HKMA SPM, per Group AML Policy).
  • Core phrase: I'm proposing an interim control + waiver reference + expiry date. The trio is non-negotiable.
  • Core rule: If you said it in the meeting, write it in the email the same day. Compliance officers quote section numbers back at you — match their precision.

Article 19 — Banking Terminology (回顾)

  • Core pattern: Speak the regulator's local filing language — SAR (US/UK) vs STR (HK/SG); JFIU (HK) vs STRO (SG) vs FinCEN (US) vs NCA (UK).
  • Core phrase: SoW (how the wealth was built) and SoF (where this transaction's money is from) are NOT the same. Write separate AC for each.
  • Core rule: Periodic refresh vs event-driven review are different triggers. Event-driven trumps periodic. Sanctions screening runs on every payment, not just onboarding.

Article 20 — KYC / AML UAT (本文)

  • Core pattern: The BA chairs triage, not the QA Lead. Label every defect (AC failure, control failure, parameter change, model change).
  • Core phrase: Default to the safer regulatory option — never "skip" a NULL or missing field. Default to high, trigger EDD, fix the data later.
  • Core rule: A P1 needs an RCA, a re-execute, and a Compliance sign-off before the go/no-go. The go/no-go email is what Audit reads first — make it 2 lines, dated, named.
🎓 Module 5 Takeaway

The three articles share one principle: in HSBC, a BA's vocabulary IS the control. If you write the wrong AC, the system makes the wrong decision. If you say "skip" in triage, the regulator's question becomes "why did you allow a skip?". The words you choose at the desk become the audit trail in the regulator's office. Choose them with the same care Compliance chooses them.
