Lessons Learned Document — Closing the Loop

Article 14 · Module 3: Written Communication · PM English

⏱ 7 min read 🟡 Intermediate 🎯 What went well, what didn't, what we change, retrospective, blame-free language

1. Scenario 场景背景

It is Friday 14:00 HKT. You are running the Project Closure Retrospective for the Group-wide Sanctions Screening Platform. The project went live 6 weeks ago and is now in BAU (Business as Usual). The Sponsor and the workstream leads are in the room. Your job is to capture what went well, what didn't, and what we will change next time — and circulate the lessons learned document by EOD Monday.

周五 14:00 香港时间，你在主持集团级制裁名单筛查平台的项目收尾回顾会。项目 6 周前已上线，现进入 BAU（日常运营）。Sponsor 和工作流负责人在场。你的职责是捕获什么做得好、什么没做好、下次我们要改什么，并在下周一 EOD 前发出经验教训文档。

A lessons learned document is the PM's institutional memory. Without it, the next project repeats the same mistakes. With it, the bank gets smarter. The document must be blame-free, actionable, and circulated.

经验教训文档是 PM 的组织记忆。没有它，下个项目重蹈覆辙。有了它，银行变聪明。文档必须不追责、可操作、已发出。

2. Sample Retrospective 回顾会样例

The following is a dialogue from the retrospective, then a lessons learned document structure.

以下是回顾会的对话，以及经验教训文档的结构。

You (PM): Thanks all. Goal of today's retrospective: capture what went well, what didn't, and what we will do differently next time. Karen, please take notes; I'll circulate the document by Monday EOD.感谢各位。今天回顾会的目标：捕获什么做得好、什么没做好、下次怎么做不同。Karen 请记笔记，我下周一 EOD 前发出文档。
You (PM): Blame-free ground rule: this is about the system, not the people. Let's start with what went well.不追责基本规则：这是关于系统，不是关于个人。先说什么做得好。
Daniel (TL): What went well: the phase gate reviews caught the architecture issue early. We caught the NFR gap in Phase 2 and fixed it before M3 build started. Lesson: invest in phase gate discipline, it pays back in build quality.做得好：阶段门评审早期捕获了架构问题。我们在 Phase 2 捕获 NFR 缺口，M3 构建前修复。教训：投入阶段门纪律，会在构建质量上回报。
Mark (Data Quality): What didn't go well: we discovered the data migration mapping issue in M4 SIT — too late. We should have discovered it in M2 design. Lesson: build a data migration mock-up in M2, not in M4. Action: Wei to add this to the lessons learned document.没做好：我们在 M4 SIT 才发现数据迁移映射问题——太晚。我们应该在 M2 设计阶段就发现。教训：在 M2 做数据迁移 mock-up，而不是 M4。Action: Wei 写进经验教训文档。
Mei (Compliance): What went well: the regulator engagement plan worked. We had no findings, no exceptions, no escalations across HKMA, MAS TRM, and FCA. Lesson: engage regulators early, not at submission time. We started 8 months before submission.做得好：监管沟通计划有效。HKMA、MAS TRM、FCA 无新发现、无例外、无升级。教训：早沟通监管，不要等到提交时。我们提前 8 个月开始。
Steve (Cyber): What didn't go well: the pen test was scheduled too late — it landed 2 weeks before go-live, leaving no time to remediate findings. Lesson: schedule the pen test at M3 build completion, not M4 SIT. Action: Wei to reflect this in the lessons learned.没做好：渗透测试排得太晚——上线前 2 周才做，没时间修复发现。教训：渗透测试排在 M3 构建完成时，不是 M4 SIT。Action: Wei 写进经验教训。
You (PM): Top 3 changes for next time. One: data migration mock-up in M2. Two: pen test at M3 build completion. Three: regulator engagement from project kick-off, not from M4. Margaret, anything to add?下次要改的前 3 项。一：M2 做数据迁移 mock-up。二：渗透测试排在 M3 构建完成时。三：监管沟通从项目 kick-off 开始，不是 M4。Margaret，要补充吗？
Margaret: Add one more: the vendor cutover workshop should be a formal gate, not an informal session. We caught the cutover issue, but only because Wei hosted the workshop. Make it a standard gate in the next project.再加一个：厂商切换工作坊应作为正式门，不是非正式会。我们捕获了切换问题，但只是因为 Wei 主持了工作坊。下次作为标准门。
You (PM): Captured. Four changes for next time. Action: Karen to circulate the lessons learned by Monday EOD for review. Margaret, please sign off on the document before I file it in the Group PMO repository.已记。下次要改的 4 项。Action: Karen 周一 EOD 前发出经验教训供 review。Margaret，在我归档到集团 PMO 仓库前请签批文档。

Lessons Learned Document Structure 经验教训文档结构

1. Project summary

  • Name: Group-wide Sanctions Screening Platform
  • Duration: 18 months (Jan 2025 – Jun 2026)
  • Outcome: Delivered on time, on budget. HKMA and MAS TRM milestones met with zero findings.

2. What went well

  • Phase gate reviews caught the NFR gap in Phase 2, before M3 build. Saved an estimated 6 weeks of rework.
  • Regulator engagement from project kick-off produced zero findings across HKMA, MAS TRM, FCA.
  • Cross-team sync meetings kept the four workstreams aligned on dependencies and handoffs.

3. What didn't go well

  • Data migration mapping issue was discovered in M4 SIT. Should have been caught in M2 design.
  • Pen test scheduled too late (2 weeks before go-live). Left no time to remediate findings.
  • Vendor cutover workshop was informal. Should have been a formal gate.

4. Top changes for next time

  1. Build a data migration mock-up in M2 design, not in M4 SIT.
  2. Schedule the pen test at M3 build completion, not in M4 SIT.
  3. Start regulator engagement from project kick-off, not from M4.
  4. Make the vendor cutover workshop a formal gate with documented exit criteria.

5. Owner of follow-up

  • PM (Wei): circulate by Monday EOD; file in Group PMO repository.
  • Sponsor (Margaret): sign off the document.
  • Group PMO (Karen): incorporate the four changes into the next programme charter template.

3. Key Phrases 关键句型

  • Goal Goal: capture what went well, what didn't, and what we will do differently next time. 目标：捕获做得好、没做好、下次怎么做不同 Always open a retrospective with the three buckets. Drift is the enemy of retrospectives.
  • Blame-free Blame-free ground rule: this is about the system, not the people. 不追责基本规则：这是关于系统，不是关于个人 State the ground rule at the start. People only speak honestly when they feel safe.
  • Went well What went well: the phase gate reviews caught the NFR gap early. 做得好：阶段门评审早期捕获 NFR 缺口 Always start with what went well. It sets the tone and surfaces practices to keep.
  • Didn't go well What didn't go well: data migration discovered in M4 SIT, too late. 没做好：M4 SIT 才发现数据迁移问题，太晚 "Didn't go well" is blame-free language. Never "Steve messed up" or "we failed".
  • Lesson Lesson: invest in phase gate discipline, it pays back in build quality. 教训：投入阶段门纪律，会在构建质量上回报 A lesson is the generalisation from a specific event. The generalisation is what next project uses.
  • Change for next time Top 3 changes for next time. 下次要改的前 3 项 Three is the magic number. Fewer feels thin. More is too many to remember.
  • Owner of follow-up Owner of follow-up: PM to circulate, Sponsor to sign off, Group PMO to template. 跟进 owner：PM 发出，Sponsor 签批，集团 PMO 模板化 A lesson without an owner of follow-up is a wish. Always name the owner who will template it.
  • File File in the Group PMO repository. 归档到集团 PMO 仓库 Lessons learned that don't get filed don't get reused. The repository is the institutional memory.

4. Vocabulary 词汇表

  • Retrospective n. 回顾会 A meeting at the end of a project to capture lessons learned.
  • Lessons learned n. 经验教训 Documented findings from a closed project, kept for future projects.
  • What went well n. 做得好 Practices and decisions that worked, to be repeated.
  • What didn't go well n. 没做好 Practices and decisions that didn't work, to be changed.
  • What we will do differently n. 下次怎么做不同 The actionable changes for the next project.
  • Blame-free n. 不追责 A ground rule that ensures the conversation is about the system, not people.
  • Institutional memory n. 组织记忆 The bank's accumulated lessons across projects, stored in repositories.
  • Group PMO repository n. 集团 PMO 仓库 The central store where lessons learned are filed and searched.
  • Generalisation n. 普遍性 The lesson abstracted from a specific event, applicable to future projects.
  • Owner of follow-up n. 跟进 owner The person responsible for circulating, signing off, and templating the lessons.
  • BAU (Business as Usual) n. 日常运营 The steady state of running a system, after the project is done.

5. Grammar Notes 语法点

① "What went well / didn't go well" — past simple as retrospective tense

Retrospectives use past simple. What went well? What didn't go well? What will we do differently? Three tenses, three buckets. Memorise the rhythm. The present-tense and future-tense phrases pair with the past-tense phrases.

回顾会用过去时。什么做得好？什么没做好？下次怎么做不同？

② "Lesson: invest in X, it pays back in Y" — the generalisation pattern

A lesson is a generalisation, not a story. Lesson: invest in phase gate discipline, it pays back in build quality. The pattern is: practice + consequence. "We did X" is a story. "Investing in X pays back in Y" is a lesson.

教训是普遍性，不是故事。教训：投入 X 会在 Y 上回报。

③ "PM to circulate, Sponsor to sign off, Group PMO to template" — the owner chain

The follow-up section names an owner chain. Each owner has one verb. PM to circulate, Sponsor to sign off, Group PMO to template. Same infinitive-verb pattern for each. The chain is auditable.

跟进部分列出 owner 链。每个 owner 一个动词。PM 发出，Sponsor 签批，集团 PMO 模板化。

6. Cultural Tips 文化 / 职场 Tips

  • State the blame-free ground rule first. People only speak honestly when they feel safe. Say it out loud at the start of the retrospective.
  • Start with what went well. It sets the tone and surfaces practices to keep. Pure-blame retrospectives get shorter attendance next time.
  • Lessons are generalisations, not stories. "We did X" is a story. "Investing in X pays back in Y" is a lesson. Train yourself to abstract.
  • Three changes for next time. Fewer feels thin. More is too many to remember. Three is the magic number.
  • Owner of follow-up is the close-out. A lesson without an owner is a wish. Name the owner who will template it into the next charter.
  • File in the Group PMO repository. Lessons learned that don't get filed don't get reused. The repository is the institutional memory.
  • Sign-off from the Sponsor. Sponsor sign-off on the lessons learned document is the formal close. Without it, the document is informal.
  • Circulate by Monday EOD. Late lessons learned are wasted lessons learned. The retrospective on Friday, the document on Monday.

7. Practice 练习题

Exercise 1 — Translate the retrospective

Translate the retrospective excerpt into natural Chinese.

"Goal of today's retrospective: capture what went well, what didn't, and what we will do differently next time. Blame-free ground rule: this is about the system, not the people. Let's start with what went well. Daniel: What went well — the daily standups caught the integration issue in week 2. We saved 2 weeks of rework. Lesson: invest in standup discipline. What didn't go well: we discovered the data quality issue in UAT, too late. Lesson: build a data quality check into M3, not UAT. Action: Wei to add this to the lessons learned. Top 3 changes for next time: one, data quality check in M3; two, contractor SOW signed at project kick-off, not M2; three, vendor cutover workshop as formal gate. Margaret, please sign off the document before I file it in the Group PMO repository."

"今天回顾会的目标：捕获做得好、没做好、下次怎么做不同。不追责基本规则：这是关于系统，不是关于个人。先说什么做得好。Daniel：做得好——每日站会第 2 周捕获了集成问题。我们节省了 2 周返工。教训：投入站会纪律。没做好：UAT 才发现数据质量问题，太晚。教训：M3 做数据质量检查，不要到 UAT。Action: Wei 写进经验教训。下次要改的前 3 项：一、M3 数据质量检查；二、外包 SOW 项目 kick-off 签，不要到 M2；三、厂商切换工作坊作为正式门。Margaret，在我归档到集团 PMO 仓库前请签批文档。"

Exercise 2 — Spot the retrospective mistake

Which retrospective line is most blame-free and useful? Why?

  1. "Steve missed the pen test schedule."
  2. "The pen test was scheduled too late, leaving no remediation time. Lesson: schedule at M3 build completion."
  3. "Pen tests are always chaotic, what can you do."
  4. "We should try harder next time."

② is the most blame-free and useful. It names the problem (pen test late) and the lesson (schedule at M3). ① names a person, not a system. ③ is fatalism. ④ is hand-waving. Lessons must generalise the system, not blame the person.

Exercise 3 — Run a 30-minute retrospective

You are the PM. Build a 30-minute retrospective for a project that: (a) delivered on time and on budget, (b) had 1 vendor slip recovered, (c) had 1 pen test late, (d) had a successful regulator engagement. Capture: 2 went well, 2 didn't, 3 changes, 3 owner of follow-up.

"Goal: capture what went well, what didn't, what we change next time. Blame-free ground rule. What went well: 1) phase gate reviews caught the NFR gap in Phase 2; 2) regulator engagement from kick-off produced zero findings. What didn't go well: 1) vendor slipped M3 by 1 week; 2) pen test landed 2 weeks before go-live. Top 3 changes: 1) include a vendor SLA review at M2; 2) schedule pen test at M3 build completion; 3) start regulator engagement at kick-off, not M4. Owner of follow-up: PM to circulate by Mon EOD, Sponsor to sign off, Group PMO to template the 3 changes into the next charter."
