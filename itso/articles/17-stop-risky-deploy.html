Stopping a Risky Deploy — Pre-Prod Finding, Blast Radius, Decision Tree

Article 17 · Module 4: Difficult Security Conversations · ITSO English

⏱ 8 min read 🟠 Advanced 🎯 Pre-prod finding, blast radius, decision tree, alternative, no surprise

1. Scenario 场景背景

It is Thursday 21:00 HKT, the night before a scheduled M4 production deploy. You are the AppSec Lead. A pre-production security scan has just flagged a Critical SQL injection in the new customer-onboarding-svc endpoint. The endpoint touches customer PII (name, HKID, address). The deploy is scheduled for 02:00 HKT Friday. The Engineering Lead, Mark, has called an emergency call with you, the Tech Lead Daniel, and the PM Wei. The business has been promised the feature for a Monday launch. You have 30 minutes to decide: go, no-go, or deferred go with conditions.

周四 21:00 香港时间，M4 生产发布前一晚。你是AppSec 负责人。生产前安全扫描刚在新的customer-onboarding-svc端点标记了一个严重 SQL 注入。端点接触客户 PII（姓名、HKID、地址）。发布定于周五 02:00 HKT。工程负责人 Mark 召集了与你、Tech Lead Daniel、PM Wei 的紧急电话。业务已被承诺该功能周一上线。你有 30 分钟决定：go、no-go、或有条件延迟 go。

Stopping a risky deploy is a 30-minute decision under uncertainty. The instinct is to let it go and "fix in production". The professional pattern is: present the finding, name the blast radius, walk the decision tree, propose an alternative, no surprise to the CISO or the sponsor. The deploy is the symptom; the decision is the discipline.

叫停高风险发布是不确定性下的 30 分钟决策。本能是放行然后"生产中修复"。专业模式是：陈述发现、具名爆炸半径、过决策树、提出替代方案、不让 CISO 或 sponsor 意外。发布是症状；决策是纪律。

2. The Emergency Call — 5 Sections, 30 Minutes

Mark (Eng Lead): Mark: AppSec scan flagged a Critical SQL injection in customer-onboarding-svc at 20:30 HKT. Deploy is in 5 hours. What's the call?Mark：AppSec 扫描 20:30 HKT 标记 customer-onboarding-svc 严重 SQL 注入。5 小时后发布。怎么决定？
You (AppSec): Three pieces of info first, then the call.先三块信息，再决定。
You (AppSec): Finding. SQL injection in the new /onboard/validate endpoint. The endpoint accepts a customer name and a HKID; both are concatenated into a raw SQL string without parameterisation. Exploit: a single crafted request returns any customer's PII from the database. CVSS 9.1 (Critical). Public exploit pattern: well-known.发现。新 /onboard/validate 端点 SQL 注入。端点接收客户姓名和 HKID；两者未参数化直接拼接到 SQL 字符串中。利用：单个构造请求可返回数据库中任何客户的 PII。CVSS 9.1（严重）。公开利用模式：广为人知。
You (AppSec): Blast radius if exploited in production. The endpoint is customer-facing, reachable from the public internet (no authentication required for /onboard/validate — it's a pre-auth check used by the marketing site). Database affected: customer-onboarding-db, ~3.2M customer records (HK retail, PII only, no transaction history). Estimated impact: full PII exfiltration in less than 1 hour using a single automated script. Regulator exposure: PDPO data breach notification required, HKMA Cyber Resilience finding likely.生产中被利用的爆炸半径。端点是客户面，可从公网访问（/onboard/validate 无需认证——是营销网站使用的预认证检查）。受影响数据库：customer-onboarding-db，约 320 万客户记录（HK 零售，仅 PII，无交易历史）。预计影响：使用单个自动化脚本不到 1 小时完整 PII 外泄。监管暴露：需 PDPO 数据泄露通知，可能 HKMA 网络韧性发现。
You (AppSec): Fix. Replace the raw SQL with a parameterised query. Code change in 1 place, the DAO layer. Daniel, how long to fix, test, and re-deploy?修复。将原始 SQL 替换为参数化查询。1 处代码变更，DAO 层。Daniel，修复、测试、重新发布要多长时间？
Daniel (Tech Lead): Daniel: Code change: 1 hour. Unit test: 1 hour. Re-scan: 1 hour. Re-deploy: 30 minutes. Total: 3.5 hours. Deploy at 06:00 HKT Friday instead of 02:00. 4-hour delay.Daniel：代码变更：1 小时。单元测试：1 小时。重新扫描：1 小时。重新发布：30 分钟。总计：3.5 小时。周五 06:00 HKT 发布代替 02:00。延迟 4 小时。
You (AppSec): Decision tree. Three options. Option A — go at 02:00 HKT as planned. Critical SQL injection in production, public endpoint, no auth. I cannot recommend this. The blast radius is full PII exfiltration, regulator exposure, and a likely PDPO notification within 72 hours. Option B — defer to 06:00 HKT, fix in the meantime. 4-hour delay, no production exposure, no regulator exposure. The Monday launch is still on track. Option C — go at 02:00 HKT with the endpoint disabled (dark-launch the deployment, do not enable the route). The deployment ships, the feature is on standby, the endpoint is not yet reachable. Fix on Friday, re-enable on Saturday morning. 0-hour delay to the business, 24-hour delay to the feature.决策树。三个方案。方案 A——02:00 HKT 按计划发布。生产中严重 SQL 注入，公网端点，无需认证。我不能推荐。爆炸半径是完整 PII 外泄、监管暴露、72 小时内可能的 PDPO 通知。方案 B——推迟到 06:00 HKT，期间修复。延迟 4 小时，无生产暴露，无监管暴露。周一上线仍在轨道上。方案 C——02:00 HKT 发布但端点禁用（暗启动部署，不启用路由）。部署发布，功能待机，端点暂不可达。周五修复，周六早上重新启用。业务 0 延迟，功能 24 小时延迟。
Wei (PM): Wei: Option C. Monday launch is intact. The 24-hour feature delay is acceptable to the sponsor — I checked.Wei：方案 C。周一上线完整。功能 24 小时延迟 sponsor 可接受——我确认过。
Mark: Mark: Option C from engineering too. Deploy the binary, leave the route off, fix in the morning. No rollback risk.Mark：工程也方案 C。部署二进制，路由关闭，早上修复。无回滚风险。
You (AppSec): Agreed: Option C. Three things to confirm before 02:00 HKT. One: Daniel, the route stays off — verified by the load balancer health check at 02:30 HKT. Two: the fix is in by 09:00 HKT Friday, re-scan by AppSec by 10:00 HKT, route enabled by Wei with PM sign-off by 11:00 HKT. Three: I will brief the Group CISO at 22:00 HKT tonight, so the CISO is not surprised. No surprise to the CISO, no surprise to the sponsor.同意：方案 C。02:00 HKT 前确认三件事。一：Daniel，路由保持关闭——02:30 HKT 负载均衡健康检查验证。二：修复周五 09:00 HKT 前完成，AppSec 10:00 HKT 前重新扫描，Wei 11:00 HKT 前 PM 签批启用路由。三：我今晚 22:00 HKT 简报集团 CISO，不让 CISO 意外。不让 CISO 意外，不让 sponsor 意外。

3. Key Phrases 关键句型

  • Three pieces of info first, then the call Three pieces of info first, then the call. 先三块信息，再决定 The discipline that prevents decisions under panic. Finding, blast radius, fix time — in that order. The call comes after, never before.
  • Blast radius if exploited in production Blast radius if exploited in production: full PII exfiltration in less than 1 hour. 生产中被利用的爆炸半径：不到 1 小时完整 PII 外泄 "Blast radius" is the language that makes the risk concrete. Numbers (3.2M records, 1 hour) are what the CISO remembers.
  • I cannot recommend this Critical SQL injection in production, public endpoint, no auth. I cannot recommend this. 生产中严重 SQL 注入，公网端点，无需认证。我不能推荐 "I cannot recommend" is the polite refusal. It states the risk and refuses the option, without taking the decision away from the call owner.
  • Decision tree Decision tree. Three options. Option A, Option B, Option C. 决策树。三个方案。方案 A、方案 B、方案 C A decision tree replaces panic with structure. Three options is the cognitive budget for a 30-minute call.
  • Dark-launch Go at 02:00 HKT with the endpoint disabled (dark-launch the deployment, do not enable the route). 02:00 HKT 发布但端点禁用（暗启动部署，不启用路由） "Dark-launch" is the engineering word for shipping the binary without enabling the route. The deploy lands; the feature is on standby.
  • No surprise to the CISO I will brief the Group CISO at 22:00 HKT tonight, so the CISO is not surprised. 我今晚 22:00 HKT 简报集团 CISO，不让 CISO 意外 "No surprise to the CISO" is the discipline that protects the AppSec Lead. The CISO hears the news first, from the AppSec Lead, not from the Slack channel.
  • No rollback risk Deploy the binary, leave the route off, fix in the morning. No rollback risk. 部署二进制，路由关闭，早上修复。无回滚风险 "No rollback risk" is the engineering word that makes Option C safe. If the fix doesn't land, the production state is unchanged because the route is off.

4. Vocabulary 词汇表

  • AppSec (Application Security) n. 应用安全 The team responsible for secure coding and pre-production scanning.
  • Pre-production finding n. 生产前发现 A security issue flagged before the code reaches production.
  • SQL injection n. SQL 注入 A vulnerability class where user input is concatenated into a SQL string.
  • Parameterised query n. 参数化查询 The fix for SQL injection — user input is bound, not concatenated.
  • Blast radius n. 爆炸半径 The scope of damage if a vulnerability is exploited in production.
  • Decision tree n. 决策树 A structured set of options (go, no-go, deferred go) under uncertainty.
  • Dark-launch n. 暗启动 Shipping a binary without enabling the route — the feature is on standby.
  • No surprise n. 不意外 The discipline of briefing the CISO and the sponsor before the news reaches Slack.
  • Route n. 路由 The load balancer or API gateway path that exposes an endpoint.
  • Re-scan n. 重新扫描 Running the security scan again after the fix to confirm the finding is closed.
  • Rollback risk n. 回滚风险 The risk that a fix or rollback itself causes an outage.
  • PII exfiltration n. PII 外泄 The unauthorised transfer of personally identifiable information out of the bank.

5. Cultural Tips 文化 / 职场 Tips

  • 5 sections, fixed order. Finding, blast radius, fix time, decision tree, no surprise. The call owner reads the structure, not the prose.
  • Three pieces of info first, then the call. The discipline that prevents decisions under panic. Finding, blast radius, fix time — in that order.
  • "Blast radius" with numbers, not adjectives. "Bad" is not a blast radius. "3.2M records, full PII exfiltration in 1 hour" is a blast radius.
  • "I cannot recommend" is the polite refusal. It states the risk and refuses the option, without taking the decision away from the call owner. The call owner decides; the AppSec Lead recommends.
  • Decision tree replaces panic with structure. Three options, named (A, B, C), each with a clear trade. The call owner picks; the call moves forward.
  • Dark-launch is the engineering compromise. The binary ships; the route is off. The deploy lands; the feature is on standby. No rollback risk; no business delay.
  • No surprise to the CISO or the sponsor. Both hear the news from the AppSec Lead, not from Slack. The CISO respects the AppSec Lead who briefs first; the sponsor respects the PM who briefs first.

6. Practice 练习题

Exercise 1 — Translate the decision tree

Translate the decision tree into natural Chinese.

"Decision tree. Three options. Option A — go at 02:00 HKT as planned. Critical SQL injection in production, public endpoint, no auth. I cannot recommend this. Blast radius: full PII exfiltration in less than 1 hour. Option B — defer to 06:00 HKT, fix in the meantime. 4-hour delay, no production exposure, no regulator exposure. Monday launch still on track. Option C — go at 02:00 HKT with the endpoint disabled (dark-launch the deployment, do not enable the route). Deploy the binary, route off, fix in the morning, route on Saturday. 0-hour business delay, 24-hour feature delay. No rollback risk."

"决策树。三个方案。方案 A——02:00 HKT 按计划发布。生产中严重 SQL 注入，公网端点，无需认证。我不能推荐。爆炸半径：不到 1 小时完整 PII 外泄。方案 B——推迟到 06:00 HKT，期间修复。延迟 4 小时，无生产暴露，无监管暴露。周一上线仍在轨道上。方案 C——02:00 HKT 发布但端点禁用（暗启动部署，不启用路由）。部署二进制，路由关闭，早上修复，周六启用路由。业务 0 延迟，功能 24 小时延迟。无回滚风险。"

Exercise 2 — Spot the deploy call mistake

Which line would damage the deploy call the most? Why?

  1. "Critical SQL injection, CVSS 9.1, public endpoint, no auth. Blast radius: 3.2M PII, full exfiltration in 1 hour. PDPO notification likely. Fix: parameterised query, 1 place, 3.5 hours. Decision tree: A, B, C. I recommend C."
  2. "It's just a small bug, ship it, we'll fix in production. The business is waiting."
  3. "No surprise to the CISO. I will brief the CISO at 22:00 HKT tonight. No surprise to the sponsor — Wei briefs the sponsor at 22:30 HKT."
  4. "Option C: dark-launch. Deploy the binary, leave the route off, fix in the morning, route on Saturday. No rollback risk. The deploy lands; the feature is on standby."

② is the credibility-damager. "It's just a small bug, ship it, fix in production" replaces the finding, the blast radius, and the decision tree with a hope. If the SQL injection is exploited in production, the bank is looking at 3.2M records, a PDPO notification, and a HKMA Cyber Resilience finding. ① names the finding, the CVSS, the blast radius, the PDPO exposure, the fix, and the recommendation. ③ commits to no surprise to the CISO and the sponsor. ④ names the dark-launch, the route, the no-rollback-risk. ② is the path to the 02:00 HKT call the CISO does not want to take.

Exercise 3 — Build a deploy call

Pre-prod scan flags a Critical SSRF (server-side request forgery) in a new payment-gateway endpoint, 2 hours before the 02:00 HKT deploy. The endpoint is internal-only, no internet exposure, but it can reach internal services that are not customer-facing. Database affected: none directly. Regulator exposure: low. Fix: input validation on the URL field, 1 place, 2 hours. Build the 5 sections: (1) finding, (2) blast radius, (3) fix, (4) decision tree, (5) no surprise.

"Section 1 — Finding. Critical SSRF in the new /payment/validate-callback endpoint. The endpoint accepts a callback URL from an upstream service and fetches it server-side. The URL is not validated. An attacker who can submit a callback URL can point it to internal services (e.g., the AWS metadata service, the internal admin API on 169.254.169.254) and read the response. CVSS 8.6 (High — rated Critical by the bank's internal standard for SSRF on payment paths). Section 2 — Blast radius if exploited in production. Endpoint is internal-only, no internet exposure. But the endpoint can reach internal services, including the AWS metadata service, which exposes IAM credentials. If exploited, the attacker can pivot to the bank's AWS account using the metadata credentials. Database affected: none directly, but a successful pivot can reach the payment-gateway-db via the IAM credentials. Estimated impact: potential unauthorised access to the payment-gateway-db (~12M records, transaction history). Regulator exposure: medium (PCI-DSS scope expansion if IAM credentials are exfiltrated). Section 3 — Fix. Input validation on the URL field — allowlist of internal callback hosts, reject any URL pointing to a non-allowlisted host. Code change in 1 place, the URL parser. Daniel, how long? Section 4 — Decision tree. Option A — go at 02:00 HKT. SSRF in production, internal pivot to AWS credentials. I cannot recommend this. Option B — defer to 04:00 HKT, fix in the meantime. 2-hour delay, no production exposure. Option C — go at 02:00 HKT with the callback validation in 'log-only' mode (the validation runs but does not block the request; only logs the violation). The deploy lands, the validation logs suspicious callbacks, the fix is in by 04:00 HKT, the validation is in 'block' mode. 0-hour business delay, 2-hour window of log-only. I recommend Option C — log-only mode is a known mitigation, the bank has run it before, and it provides visibility without exposing the production state. Section 5 — No surprise. I will brief the Group CISO at 23:00 HKT tonight. Wei, please brief the sponsor at 23:30 HKT. No surprise to the CISO, no surprise to the sponsor."
