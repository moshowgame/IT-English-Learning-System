Participating in Sprint Planning

Article 06 · Module 2: Meeting Workshops · Developer English

⏱ 9 min read 🟡 Intermediate 📋 Commit to user stories, raise dependencies, ask sharp questions

1. Scenario 场景背景

It is Sprint Planning Day for the next 2-week sprint. The whole team is on a Teams call: your Scrum Master David (HK), tech lead Priya (UK), BA Jonathan (HK), and three developers (you, Sam, Lily). Jonathan presents three candidate user stories for the sprint, and you need to size them, raise dependencies, and commit to what you'll own.

冲刺规划日到了，下一个双周冲刺。团队在 Teams 上：Scrum Master David（香港）， 技术负责人 Priya（英国），BA Jonathan（香港），还有三个开发（你、Sam、Lily）。 Jonathan 介绍了 三张候选用户故事，你需要估时、提出依赖、确认你负责哪些。

2. Dialogue 英文对话

Jonathan (BA): OK, story number one — ONB-1420: "As a customer, I want to upload my proof of address as a PDF so I can complete onboarding." Acceptance criteria are: support PDF and JPG, max 5MB, virus scan must pass, store in AWS S3 with KMS encryption. Any takers? How do you size this?好，第一个故事——ONB-1420：作为客户，我希望上传住址证明 PDF，以完成开户。验收标准：支持 PDF 和 JPG，最大 5MB，必须过病毒扫描，存到AWS S3用KMS 加密。谁来接？这个怎么估？
You: Quick sizing question, Jonathan — does the virus scan need to be synchronous (block the upload) or async (scan in background)?问一下 Jonathan——病毒扫描是同步（阻塞上传）还是异步（后台扫）？
Jonathan (BA): Compliance said async — the file is quarantined first, scanned in the background, and the user is notified when it's cleared.Compliance 说异步——文件先隔离，后台扫描，通过后通知用户。
You: Got it. With async scanning, I'd say 5 points — the upload itself is straightforward, but the quarantine + scan + notify flow has a few moving parts. I can take this one.明白。异步扫描的话我估5 个点——上传本身简单，但隔离+扫描+通知这条链路有几个环节。我来接。
Priya (TL): Quick concern — KMS encryption is in scope, right? We need to make sure the InfoSec-approved key is already provisioned, or we'll hit the same blocker we had last sprint.一个关注点——KMS 加密在范围内吧？需要确保InfoSec 批准的密钥已配好，否则会像上个冲刺一样卡住。
Jonathan (BA): Yes, I've already raised the KMS key request with InfoSec. Should be ready by sprint start.已经提了KMS key 的申请给 InfoSec，冲刺开始前应该能就绪。
David (SM): OK, ONB-1420 is 5 points, owner Wei. Moving on — ONB-1425: "As a customer, I want to receive an SMS OTP so I can verify my phone." Sam, you picked this one up in refinement, right?好，ONB-1420 是5 个点，owner 是 Wei。下一张——ONB-1425：作为客户，我希望接收短信 OTP 以验证手机。Sam，你在 refinement 上接的，对吧？
Sam: Yeah, sized it at 3 points. The Twilio integration is mostly done, just need to wire up the OTP validation endpoint and add rate limiting.是的，我估了3 个点。Twilio 集成基本做完，只剩OTP 校验接口和限流。
You: Quick one on ONB-1425 — has the OTP rate limit been confirmed with InfoSec? Last time they wanted max 5 attempts per hour per number.问一下 ONB-1425——OTP 限流和 InfoSec 确认了吗？上次他们要求每个号码每小时最多 5 次。
Jonathan (BA): Good catch. I'll double-check with InfoSec and update the AC. Not blocking but should be confirmed by Wednesday.抓得准。我和 InfoSec 再对一下，更新到 AC 里。不阻塞，但周三前要确认。
You: Cool, no blockers from my side.好，我这边没阻塞。
David (SM): Last story — ONB-1430: "As a compliance officer, I want to download a daily KYC report so I can review onboarding activity." This is a reporting story — pulling from the existing data warehouse, generating a CSV, and uploading to SFTP. Anyone?最后一张——ONB-1430：作为合规官，我希望下载每日 KYC 报告来审核开户情况。这是报表类故事——从现有数仓取数，生成 CSV，上传到 SFTP。有人接吗？
You: I'll let Sam or Lily take this one — I'm already committed to two stories this sprint. Happy to help review the PR though.让 Sam 或 Lily 接吧——这个冲刺我已经认领两张了。不过我可以帮忙 review PR。
Lily: I can take it. Looks like 2-3 points — most of the ETL pipeline is reusable from the trade reporting project.我来接。看着是2-3 个点——大部分ETL 管道可以从交易报表项目复用。

3. Key Phrases 关键句型

  • Size How do you size this? 这个怎么估？ Standard Scrum phrase to ask for estimation.
  • Commit I can take this one. 我来接 Use for taking ownership of a story. Pick it up is also common.
  • Concern Quick concern — KMS encryption is in scope, right? 一个关注点——KMS加密在范围内吧？ Use quick concern / quick question to flag something without derailing the meeting.
  • Catch Good catch. 抓得好 / 提到点上了 Acknowledges a sharp observation. Good point / fair point are alternatives.
  • Blocker Not blocking, but should be confirmed by Wednesday. 不阻塞，但周三前需要确认 Differentiates blocking (now) from non-blocking risk (later).
  • Capacity I'm already committed to two stories this sprint. 这个冲刺我已经认领两张了 Use this to politely decline without sounding lazy. Capacity is a real constraint.
  • Help Happy to help review the PR though. 不过可以帮忙看PR Even when not taking, offer a smaller contribution. Builds team spirit.
  • AC Update the AC. 更新验收标准 AC = Acceptance Criteria. Standard Scrum abbreviation.

4. Vocabulary 词汇表

  • user story n. 用户故事 A short description of a feature from the user's perspective. Format: "As a…, I want…, so that…"
  • acceptance criteria (AC) n. 验收标准 Conditions that must be met for a story to be considered done.
  • size / estimate v. 估时 (故事点) Use the Fibonacci sequence: 1, 2, 3, 5, 8, 13. Bigger = more uncertain.
  • synchronous / asynchronous 同步 / 异步 Sync = caller waits. Async = caller moves on, gets notified later.
  • KMS (Key Management Service) AWS AWS 密钥管理服务 Managed encryption key service. Banks require InfoSec-approved keys.
  • OTP (One-Time Password) banking 一次性密码 / 动态口令 Common for phone / transaction verification. SMS-OTP and TOTP (authenticator app).
  • rate limiting n. 速率限制 / 限流 Restrict the number of requests per time window. Anti-abuse measure.
  • ETL (Extract, Transform, Load) n. 数据抽取-转换-加载 A data pipeline pattern for reporting. Common in banking data warehousing.
  • SFTP (Secure File Transfer Protocol) n. 安全文件传输协议 Used to deliver reports to external parties (regulators, vendors).

5. Grammar Notes 语法点

① "Quick concern" / "Quick question" — Soft Interruptions

These phrases let you jump in without sounding rude:

  • Quick concern — KMS is in scope, right?
  • Quick question on AC — has rate limit been confirmed?
  • Quick one on the timeline — when do we need this by?

② "Happy to…" — 跨场景的礼貌万能句

Already seen in Article 04, but it's the most versatile phrase in Scrum meetings:

  • Happy to help review.
  • Happy to drive the spike.
  • Happy to sync offline on this.

③ Hedging Size with "I'd say…" / "My guess is…"

Sizing is never exact. Use hedging language:

  • ❌ "This is 5 points." (over-confident)
  • ✅ "I'd say 5 points."
  • ✅ "My guess is 3, but it could be 5."

6. Cultural Tips 文化/职场 Tips

🎯 Sprint Planning Best Practices at HSBC
  • Always ask clarifying questions BEFORE sizing. Most estimation errors come from unclear AC.
  • Flag dependencies early. KMS keys, InfoSec approvals, upstream teams — all are common blockers.
  • Don't over-commit. Aim for 70-80% capacity utilization. Leave 20-30% for incident support and BAU.
  • Use the "I can take / I can't take" structure. It's cleaner than "I want to" or "I'll do".
  • Speak up on concerns, even as a junior. Sprint Planning is the right time to flag risk. Better now than mid-sprint.
⚠️ Common Mistakes
  • ❌ "I'll take it." (over-eager; sounds like over-commitment without thinking)
  • ✅ "Let me think about capacity… I can take it, but I won't be able to help on the other story."
  • ❌ Staying silent when you spot a dependency — this is exactly when you should speak up.
  • ❌ Sizing 1 point on a 5-point story to look good — damages the team and your credibility.

7. Practice 练习题

✍️ Practice Drills
Q1. Translate to natural English:

"问一下，KMS key 准备好了吗？没有的话要早点找 InfoSec 申请，否则会卡住。"

"Quick question — is the KMS key already provisioned? If not, we should raise the request to InfoSec as early as possible, otherwise we'll hit a blocker."
Q2. Politely decline a story while staying engaged:

"Wei, can you take ONB-1430 too? It's only 2 points."

"I'd love to help, but I'm already committed to two stories this sprint — ONB-1420 and helping Sam on the OTP rate limit work. Let me hand it over to Lily or Sam. Happy to be a secondary reviewer on the PR though."
Q3. Fill in the right Scrum phrase:

"This is _____ 3 points, but could be 5. / I'll _____ it. / Has the InfoSec approval been _____ yet? / Quick _____ on the AC — the rate limit is missing."

I'd say / take / confirmed / question
Q4. Role-play:

In sprint planning, your BA presents ONB-1500: "As a customer, I want to set a daily transfer limit so I can control my spending." AC: configurable, default 50k HKD, requires SMS-OTP confirmation when changed. What 2-3 questions would you ask before sizing? What size would you give?

"Quick questions before I size this:
  1. Does the SMS-OTP confirmation use the same Twilio integration as ONB-1425? If yes, we can reuse the OTP service — saves effort.
  2. Is the 50k default a regulatory requirement or business-configurable? If regulatory, where is it documented?
  3. Does changing the limit need to be logged in the audit trail? Compliance usually requires this for any limit change.
I'd say 3 points if we reuse the OTP service, 5 if it's a fresh integration. I can take it."
