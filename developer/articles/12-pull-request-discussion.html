Pull Request Description & Discussion

Article 12 · Module 3: Written Communication · Developer English

⏱ 8 min read 🟡 Intermediate 📝 Write clear PR titles, descriptions, and respond professionally

1. Scenario 场景背景

You are submitting a PR for PAY-2847 — the HSBCnet API callback retry logic. You've just finished coding, tests are green, and now you need to write a PR description that helps your reviewer Priya (UK) understand what changed, why, and how to test it in under 2 minutes of reading.

你正在为 PAY-2847（HSBCnet API 回调重试逻辑）提交 PR。代码完成、测试通过， 现在需要写一份 PR 描述，让英国 reviewer Priya 在 2 分钟内明白 改了什么、为什么、怎么测试。

3. PR Title & Description Example

✅ Good PR Title

PR Title: [PAY-2847] Add configurable retry logic for HSBCnet API callbackPR 标题：[PAY-2847] 为 HSBCnet API 回调添加可配置的重试逻辑

❌ Bad PR Title

PR Title: updatePR 标题（不够具体）：update——应改为更具体的描述

✅ Good PR Description

📋 Description Jira: PAY-2847 — HSBCnet API callback retry logic
Type: Feature + Refactor
Reviewer: @priya.shah
Estimated review time: ~20 min
🎯 What & Why The HSBCnet API callback currently fails silently if the downstream is briefly unavailable, causing stuck payment records in the reconciliation job. This PR adds:
  • Configurable retry count (default: 3) via application.yml
  • Configurable backoff (exponential, base 2s)
  • Idempotency check via Idempotency-Key header to avoid duplicate processing
🛠 How to test
  1. Deploy to UAT (Jenkins job: hsbcnet-callback-uat)
  2. Trigger a payment with the test merchant ID MT-TEST-001
  3. Simulate a 5xx response by throttling the UAT HSBCnet endpoint (see test guide)
  4. Verify the retry counter increments in the logs and the payment eventually succeeds
✅ Checklist
  • [x] Unit tests added (12 cases, 100% pass)
  • [x] Integration test on the retry flow
  • [x] Confluence updated with the new config properties
  • [x] No new audit logging required (existing loggers cover it)
  • [x] InfoSec approval: KMS-RET-001 confirmed
📸 Screenshots / Logs [Attached: retry-counter-increments.log]附件：retry-counter-increments.log（重试计数器递增日志）
⚠️ Out of scope • Dead-letter queue (DLQ) — separate ticket PAY-2901
• Retry metrics dashboard — separate ticket PAY-2910

2. PR Discussion Example 评审对话

@Priya (Reviewer): Thanks Wei, well-structured PR 🙌. One question on the backoff calculation in RetryExecutor.java#L88 — I see we're using a decorrelated jitter pattern. Curious why we chose that over a simple exponential?谢谢 Wei，PR 结构清晰 🙌。RetryExecutor.java#L88 的退避算法有个问题——看到用的是 decorrelated jitter 模式，为什么不直接用简单的指数退避？
@Wei (Author): Good question. From what I read in the AWS Architecture Blog, decorrelated jitter reduces the thundering-herd risk when many callbacks fail at the same time. Simple exponential tends to cluster retries across services. Happy to add a comment in the code with the link if useful.好问题。根据 AWS 架构博客的说法，decorrelated jitter 能降低大量回调同时失败时的 thundering-herd 风险。简单指数退避容易让各服务的重试时间集中。如有需要，我可以加代码注释并附上链接。
@Priya (Reviewer): Yes please — a code comment would be great. One more thing — should we also cap the maximum backoff at, say, 30 seconds? Right now it can theoretically grow without bound.麻烦加个代码注释。还有一个——是否要把最大退避时间设个上限，比如 30 秒？目前理论上可以无限增长。
@Wei (Author): Good catch. I'll add a cap — default 30s, configurable. Quick one though — would 30s be the max wait per retry, or the total wait across all retries? I assume the former.提得好。我加上限——默认 30 秒，可配置。顺便问下——30 秒是单次重试的最大等待，还是所有重试的总等待？我理解是前者。
@Priya (Reviewer): Per retry, yeah. That makes sense.单次重试，对，没错。
@Wei (Author): Will do. Pushing an update shortly with the cap and the code comment. Tagging you back when it's ready for another look.好的。我马上推个更新，加上上限和代码注释，更新好后 @ 你再 review 一次。
@Priya (Reviewer): Perfect. Almost there. 👍完美。差不多搞定了 👍
@Wei (Author): Update pushed. Added: (1) max backoff cap at 30s (configurable via retry.max-backoff), (2) code comment with the AWS blog link. Ready for re-review.更新已推送。新增：(1) 最大退避时间 30 秒（通过 retry.max-backoff 可配置）；(2) 附 AWS 博客链接的代码注释。准备好再次 review。
@Priya (Reviewer): Approved ✅. Nice work, Wei. Merging now.已批准 ✅。干得漂亮，Wei。现在合并。

4. Key Phrases 关键句型

  • Title [TICKET-ID] Action verb + brief description [工单号] 动词 + 简短描述 Action verbs: Add / Refactor / Fix / Remove / Migrate / Update. Always past-tense-friendly.
  • Structure What & Why / How to test / Checklist / Out of scope 改了什么 / 怎么测 / 自检 / 范围外 The four-section structure is the gold standard for PR descriptions.
  • Open Well-structured PR. 结构清晰的 PR Open reviews with a positive. Sets a collaborative tone.
  • Question One question on… 关于…有一个问题 Use one question / one comment / one nit to introduce each piece of feedback.
  • Curious Curious why we chose X over Y? 好奇为什么选 X 而不是 Y？ Curious is a soft way to challenge a design decision. Suggests learning, not criticism.
  • Catch Good catch. 抓得好 / 提到点上了 Acknowledge the reviewer's input. Builds respect.
  • Promise Pushing an update shortly. 马上推更新 Always commit to a follow-up. "Will do" / "On it" / "Pushing shortly".
  • Status Update pushed. Ready for re-review. 更新已推，请重新 review Always tell the reviewer when you've responded. "Tagging you back" / "Ready for re-review".
  • Approval Approved ✅ Nice work. Merging now. 已批准，干得漂亮，现在合并 End the conversation cleanly. "LGTM" (Looks Good To Me) is the abbreviated form.

5. Vocabulary 词汇表

  • PR / Pull Request n. 合并请求 Also called MR (Merge Request) in GitLab.
  • LGTM (Looks Good To Me) abbr. 看着不错 (批准用语) The standard approval abbreviation in PR reviews.
  • idempotency / Idempotency-Key n. 幂等性 / 幂等键 A property where repeated operations produce the same result. Critical for retries.
  • backoff (exponential / decorrelated jitter) n. 退避策略（指数 / 去相关抖动） A retry timing strategy. Decorrelated jitter reduces thundering-herd risk.
  • thundering herd n. 惊群效应 (大量重试同时发生) When many clients retry simultaneously, overwhelming the downstream.
  • DLQ (Dead-Letter Queue) n. 死信队列 (处理失败的消息归宿) Where messages go after all retries are exhausted. Out of scope in this PR.
  • checklist n. 自检清单 Use checkboxes [x] to show what's been done. Increases reviewer confidence.

6. Grammar Notes 语法点

① PR Title Verbs — Always in Base Form or Gerund

Standard PR title verbs (used in base form / past participle):

  • Add configurable retry logic… ✅
  • Refactor retry executor… ✅
  • Fix stuck payment records… ✅
  • Remove deprecated API call… ✅
  • Migrate to Spring Boot 3.x… ✅
  • ❌ update (too vague)
  • ❌ fix bug (no detail)

② "Will do" vs. "On it" vs. "Pushing shortly"

Three ways to commit to a follow-up:

  • Will do. (short, standard)
  • On it. (slightly more casual)
  • Pushing an update shortly. (commitment + timing)

③ "Approved" / "LGTM" / "Looks good" / "Ship it"

Four ways to approve a PR (in order of casual → enthusiastic):

  • Approved ✅ (formal, GitHub standard)
  • LGTM (Looks Good To Me — abbreviated)
  • Looks good (informal)
  • Ship it (enthusiastic, casual)

7. Cultural Tips 文化/职场 Tips

📝 The "Two-Minute Review" Rule

A good PR description lets your reviewer understand the change in 2 minutes of reading. Use the four-section structure:

  1. What & Why — what changed and the business reason. Link the Jira ticket.
  2. How to test — concrete steps the reviewer can follow. With commands, endpoints, test data.
  3. Checklist — what you've already done (tests, docs, security checks).
  4. Out of scope — what you intentionally did NOT do. Sets expectations.

Reviewers who can understand your PR in 2 minutes are 10x more likely to review it quickly.

⚠️ PR Anti-Patterns
  • ❌ "update" / "fix" / "WIP" as the title (zero information)
  • ✅ "[PAY-2847] Add configurable retry logic for HSBCnet API callback"
  • ❌ PR with 2000 lines of changes across 30 files (un-reviewable)
  • ✅ PR with focused changes — split if needed
  • ❌ No description — "see diff"
  • ❌ No tests added in the PR
  • ❌ Pushing force after reviewer approved (don't rebase after approval — breaks trust)

8. Practice 练习题

✍️ Practice Drills
Q1. Improve this PR title:

"update"

"[ONB-1420] Add async virus scan for PDF uploads with SQS + Lambda" (if it's a new feature)

Q2. Write a PR description for this scenario:

You're adding a new endpoint POST /api/v2/customers/{id}/addresses to the onboarding service. The endpoint allows adding a new address with validation (postcode format, country whitelist, required fields). Response returns the new address with a 201 status. 8 new unit tests, 2 new integration tests. InfoSec reviewed the new fields.

[ONB-1450] Add POST /api/v2/customers/{id}/addresses endpoint

Jira: ONB-1450
Type: Feature
Reviewer: @priya.shah
Review time: ~15 min

What & Why: Adds a new REST endpoint to allow onboarding service to register a new customer address. Currently, address registration requires multiple service calls — this consolidates it into one. AC includes postcode validation, country whitelist, and required-field checks.

How to test:
1. Deploy to UAT (Jenkins: onboarding-uat).
2. curl -X POST https://uat.hsbc.local/api/v2/customers/{id}/addresses -d '{...}'
3. Verify 201 response and DB record.
4. Test invalid postcode (expect 400).

Checklist:
[x] 8 unit tests, all green
[x] 2 integration tests covering happy path + invalid postcode
[x] OpenAPI spec updated
[x] Confluence updated with the new endpoint
[x] InfoSec review: address fields confirmed PII-compliant

Out of scope: Address update / delete endpoints (separate ticket ONB-1460).

Q3. Translate to native-sounding review English:

"我刚把更新推上去了。加了30秒的上限，还加了代码注释说明为什么用去相关抖动。@priya 可以再看一下吗？"

"Update pushed with two changes: (1) added the 30s max backoff cap (configurable via retry.max-backoff); (2) added a code comment explaining the decorrelated jitter choice with a link to the AWS blog. Tagging you back — ready for re-review when you have a moment."
Q4. Role-play:

You submitted a PR 2 days ago. Reviewer left these comments: (a) "Please add a null check on the response." (b) "Could you explain why we chose Redis over Memcached?" (c) "Nit: typo on line 42." Write 3 polite responses, one for each, and close the loop when you push the update.

"Thanks for the review, @priya 👌

(a) Null check — good catch, adding it now with a unit test for the null case.

(b) Redis vs Memcached — fair question. We picked Redis primarily for the persistence + replication — Memcached is pure in-memory, and we wanted the cache to survive a pod restart without a cold-cache storm. Happy to add this to the ADR as well if useful.

(c) Typo — good eye, fixed.

Update pushed. All three addressed. Tagging you back for re-review."
