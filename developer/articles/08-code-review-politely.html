Code Review — Polite Suggestions

Article 08 · Module 2: Meeting Workshops · Developer English

⏱ 9 min read 🟡 Intermediate 💬 Suggest improvements without sounding judgmental or arrogant

1. Scenario 场景背景

Your colleague Sam has submitted a PR for the OneWealth statement service. The code works, but you noticed a few things during review: (1) the retry logic doesn't have a max retry count, (2) the SQL query is doing an N+1 pattern, (3) some hardcoded test values are still in a config file. You need to give feedback that's constructive, not condescending.

同事 Sam 提交了 OneWealth 对账单服务 的 PR。代码能跑，但你在 review 时发现：(1) 重试逻辑 没有 最大重试次数；(2) SQL 查询是 N+1 模式；(3) 配置里还有一些 硬编码的测试值。 你需要给出 建设性、而不是居高临下 的反馈。

3. Sample PR Comments 评审示例

💬 Comment 1: Suggesting max retry

@Wei (Reviewer): Thanks for the PR, Sam — overall looks good. One small thing on the retry logic: I noticed we don't have a max retry count. Could we add one to avoid the risk of infinite retries if the downstream is consistently down? Happy to suggest a default of 3 if useful. Just a thought — let me know what you think.感谢 PR，Sam——整体不错。关于重试逻辑有一个小建议：我注意到没有最大重试次数，能不能加一个，避免下游持续不可用时出现无限重试的风险？可以建议默认 3 次，看是否有用。只是想法——看你怎么想。

💬 Comment 2: N+1 query

@Wei (Reviewer): Quick observation on the customer query in CustomerService.java#L142 — looks like we might be hitting an N+1 pattern with the orders fetch. Have you considered doing a single JOIN query instead? I did a quick benchmark — the JOIN approach took ~80ms vs. the current 600ms+ on 100 customers. Up to you, but might be worth optimising before we hit UAT.一个快速观察——CustomerService.java#L142里的客户查询，看起来取订单可能是N+1 模式。考虑过用单次 JOIN 查询吗？我快速 benchmark 了一下——JOIN 方案 100 个客户约 80ms，当前是 600ms+。由你决定，但UAT 前值得优化。

💬 Comment 3: Hardcoded test values

@Wei (Reviewer): Heads up — I spotted a few hardcoded test values in application-dev.yml (e.g., the test customer ID on line 38 and the test account number on line 45). Could you move them to application-dev-sample.yml or use @Value placeholders? Otherwise we risk leaking test data into prod by accident. Worth a follow-up ticket if it's out of scope for this PR.提醒一下——我在application-dev.yml里发现几个硬编码的测试值（比如 38 行测试客户 ID和 45 行测试账号）。能挪到application-dev-sample.yml，或者用@Value占位符吗？否则可能误把测试数据带到生产。如果这个 PR 范围外，值得做一张 follow-up ticket。

💬 Closing comment

@Wei (Reviewer): Overall solid work, Sam — the structure is clean and the test coverage is good. Just the three points above. Let me know if you want to walk through any of them — happy to pair on it.整体做得不错，Sam——结构清晰，测试覆盖到位，就上面三点。有需要随时叫我一起过一下，可以结对。

💬 Sam's reply

@Sam: Thanks Wei, good catches — all three valid. The retry one is a quick fix, I'll push an update shortly. The N+1 one is interesting, can we hop on a quick call to discuss the JOIN approach? I'd like to keep the eager-loading as an option for the bulk export use case.谢谢 Wei，三条都抓得好。重试那个我马上推个更新。N+1 那个有意思，能约个短会聊下 JOIN 方案吗？批量导出场景我想保留急加载作为备选。
@Wei: Sure, let's do it. I've got a 30-min slot at 3pm HKT — grab it from my calendar if it works for you.好，没问题。我香港时间下午 3 点有个30 分钟的空档——你从日历上直接订就行。

4. Key Phrases 关键句型

  • Open Thanks for the PR — overall looks good. 感谢 PR——整体不错 ALWAYS open with a positive. Sets a collaborative tone.
  • Soften One small thing… / Quick observation… 一个小建议 / 简单观察 Use small / quick / minor to defuse the suggestion. Never start with "This is wrong".
  • Hedge Just a thought — let me know what you think. 只是想法——看你怎么想 Make suggestions feel like options, not orders. "Happy to discuss" closes the loop.
  • Risk To avoid the risk of X… 为了避免 X 风险 Frame suggestions around risk, not correctness. Lower friction.
  • Question Have you considered…? 你考虑过…吗？ An indirect suggestion pattern. "Could we…?" / "What if we…?" work too.
  • Data I did a quick benchmark — the JOIN approach took 80ms vs. 600ms+. 我快速 benchmark 了一下——JOIN 80ms 对比 600ms+ Data is the most persuasive argument. Show numbers when possible.
  • Defer Worth a follow-up ticket if it's out of scope. 超出范围的话值得做 follow-up ticket Don't block the PR for non-critical issues. Suggest a separate ticket.
  • Close Let me know if you want to walk through any of them. 有需要随时叫我一起过一下 Offer to help — shows you're collaborative, not just critical.
  • Praise Solid work — the structure is clean. 做得很扎实——结构清晰 Be specific in praise. "Clean code" / "good test coverage" beat generic "nice job".

5. Vocabulary 词汇表

  • PR / Pull Request n. 合并请求 A formal request to merge code changes. Reviewed before integration.
  • code review n. 代码评审 The process of peer-reviewing code before merge. Mandatory at HSBC.
  • N+1 pattern n. N+1 查询模式 (反模式) Querying parent + N children in N+1 separate queries. Should use JOIN or eager loading.
  • JOIN / eager loading n. JOIN 查询 / 急加载 Two ways to avoid N+1. JOIN in SQL, eager loading in ORM.
  • hardcoded value n. 硬编码值 A literal value in code rather than a configurable parameter. Anti-pattern.
  • infinite retry n. 无限重试 A retry loop with no termination condition. Causes resource exhaustion.
  • benchmark v. / n. 基准测试 / 性能对比 Measure performance. Always benchmark before optimising.
  • test coverage n. 测试覆盖率 % of code covered by automated tests. HSBC requires > 80% for critical services.

6. Grammar Notes 语法点

① Softening Pattern: "Have you considered…?"

Indirect suggestions are much more polite than direct ones:

  • ❌ "You should use a JOIN query." (commanding)
  • ✅ "Have you considered a JOIN query?" (suggestion)
  • ✅ "Could we add a max retry count?" (collaborative)
  • ✅ "What if we use @Value placeholders?" (exploratory)

② Past Tense in "I noticed…" / "I spotted…"

When reporting findings, use past tense for facts:

  • I noticed we don't have…
  • I spotted a few hardcoded values.
  • Looks like we might be hitting an N+1 pattern.

③ Modal verbs for Suggestion

Five levels of softness in suggestions:

  1. "We need to add a max retry." (strongest, almost demanding)
  2. "We should add a max retry." (strong)
  3. "We could add a max retry." (neutral)
  4. "Could we add a max retry?" (polite question)
  5. "Have you considered adding a max retry?" (most polite)

7. Cultural Tips 文化/职场 Tips

💬 The "Sandwich" Method in Code Review

Most senior engineers at HSBC follow the "sandwich" pattern in code reviews:

  1. Top bun (positive): What works well — overall structure, tests, naming.
  2. Filling (constructive): Specific, actionable suggestions with reasoning.
  3. Bottom bun (positive): Reaffirm overall quality, offer to help.

This pattern keeps reviews psychologically safe and avoids the "you're bad at code" feeling.

⚠️ Code Review Anti-Patterns
  • ❌ "This is wrong." (judgmental, not actionable)
  • ✅ "Have you considered a JOIN query? I saw the perf difference in a benchmark."
  • ❌ "Why didn't you write tests?" (rhetorical, hostile)
  • ✅ "Could we add a test for the retry edge case?"
  • ❌ Nitpicking naming conventions when logic is correct — pick your battles.
  • ❌ Reviewing without trying to understand intent — always read the linked ticket first.

8. Practice 练习题

✍️ Practice Drills
Q1. Soften this review comment using "Have you considered" or "Could we":

"This method is too long. Break it into smaller methods."

"Quick one on processOrder() — it's getting a bit long (~80 lines). Have you considered breaking it down into smaller methods, e.g., validateOrder, applyDiscounts, saveOrder? Might be easier to test and reuse. Up to you — just a thought."
Q2. Apply the "sandwich" method to this review:

The PR adds a new endpoint. The good: tests are comprehensive. The bad: there's a hardcoded API URL, and no input validation. Write a 3-part review.

"Thanks for the PR, Lily — overall solid work. The test coverage is comprehensive and the endpoint structure is clean.

Two small things:
  1. I noticed the API URL is hardcoded in OrderController.java#L42. Could we move it to application.yml as a configurable property? Otherwise, switching between UAT and prod will require a code change.
  2. I'd suggest adding input validation on the request body — e.g., @Valid + @NotNull on the customer ID. Right now a null ID would throw a NullPointerException deep in the service layer.
Happy to pair on the validation if helpful. Otherwise, solid PR — just the two points above."
Q3. Translate to native-sounding review English:

"这个方法有bug。在空集合的时候会抛 NullPointerException。建议加个 null check。"

"Quick one — I noticed getOrders() in CustomerService.java#L88 might throw a NullPointerException if the orders list is empty. Have you considered adding a null check (or using Optional)? Easy fix, but might cause unexpected errors in production otherwise."
Q4. Role-play:

Your colleague Lily submitted a PR with: (a) 200 lines of changes, (b) no tests, (c) one method that does 3 things, (d) uses an outdated library. Write 4 polite, specific review comments — one for each issue. Use the sandwich structure overall.

"Thanks for the PR, Lily — the new endpoint looks promising and the API contract is well-thought-out.

A few suggestions:
  1. Tests — I see no test coverage in this PR. Could we add at least the happy path + 1-2 edge case tests before merging? I can help with the test setup if needed.
  2. Single responsibility — processPayment() in PaymentService.java#L55 seems to be doing 3 things: validation, fee calculation, and persistence. Have you considered splitting it into validate(), calculateFees(), savePayment()? Easier to test and reuse.
  3. Outdated library — we're using commons-lang3 v3.9, but the team upgraded to v3.14 last month (security patches). Could you bump the version in pom.xml?
Happy to walk through any of these. Solid foundation overall — just needs a bit of cleanup."
