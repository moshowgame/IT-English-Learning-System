Pushing Back on Over-Engineering

Article 14 · Module 5: Difficult Conversations · Architect English

⏱ 9 min read 🟠 Upper-intermediate 🎯 YAGNI: keep it simple, ship value, refactor later — polite but firm pushback

1. Scenario 场景背景

It is Friday, 4:00 PM HKT. A design review is running on Microsoft Teams for the RBWM Mobile squad's new spending-insights notification service. The Tech Lead, Arjun (Bangalore), has proposed a design that uses: Kafka + Event Sourcing + CQRS + a custom saga orchestrator to send a push notification when a customer's monthly spend crosses a threshold.

周五下午 4 点香港时间，RBWM 移动 squad 的一个新功能设计评审正在 Teams 上进行： 消费洞察通知服务。Tech Lead Arjun（班加罗尔）提的设计是： Kafka + Event Sourcing + CQRS + 自研的 saga 编排器， 用于在客户月消费超过阈值时发送推送通知。

You — the Solution Architect covering RBWM Mobile — are politely concerned. The use case is a single read-after-write check on a single aggregate. Event sourcing, CQRS, and a saga orchestrator are heavy patterns built for multi-service transactional flows. You believe the squad is over-engineering the problem, and that a simple cron + DB trigger would ship the same value in 2 weeks instead of 8. Margaret (Chief Architect) is also on the call but has asked you to lead the pushback — this is exactly the kind of "architect says no" conversation she wants you to own.

你——覆盖 RBWM Mobile 的 Solution 架构师——礼貌地感到担忧。这个用例就是一个 单聚合的 read-after-write 检查。Event sourcing、CQRS、saga 编排器 这些都是为多服务事务流设计的重模式。你认为 squad 过度设计了这个问题， 简单的 cron + DB trigger 同样能交付价值，2 周搞定 vs 8 周。 Margaret（Chief Architect）也在会议里，但她请你主导这次 pushback—— 这正是她希望你承担的"架构师说不"的对话。

2. Dialogue 英文对话

Arjun (Tech Lead, RBWM Mobile): OK, so — the design is Kafka for the event bus, then we persist every state change as an event using event sourcing. The read side is a CQRS projection updated by a saga orchestrator that handles the threshold check and the notification fan-out. Eight weeks, end-to-end.好的，设计是Kafka 做事件总线，然后用 event sourcing 把每次状态变化作为事件持久化。读侧是 CQRS 投影，由 saga 编排器驱动，处理阈值检查和通知扇出。端到端 8 周。
You (Architect): Thanks, Arjun — solid work on the design. Before we sign off, I want to push back gently on the architecture choice. The use case here is a single aggregate check — a customer's monthly spend versus a threshold. Honestly, this doesn't need event sourcing, CQRS, or a saga.Arjun，谢谢——设计做得很扎实。签批前，我想就架构选择温和地提一下不同意见。这个用例本质是单个聚合检查——客户的月消费对一个阈值。坦白说，这里不需要 event sourcing、CQRS 或 saga。
You (Architect): YAGNI — You Ain't Gonna Need It. Right now, the read is always after a single write, on one aggregate, with no cross-service consistency. Event sourcing pays off when you have many readers with different shapes, or when you need time-travel replay. We don't have either.YAGNI——You Ain't Gonna Need It。目前，读取总是发生在单次写后，针对单个聚合，没有跨服务一致性。Event sourcing 在需要多读者不同形态或时间旅行重放时才划算。这两个我们都不需要。
Arjun (Tech Lead): Fair point. But I was thinking — what if we extend this to investment insights, or cross-product thresholds? We're going to need the event-sourced view eventually.有道理。但我在想——如果我们扩展到投资洞察，或者跨产品阈值呢？我们迟早需要 event-sourced 的视图。
You (Architect): Good question. But that is a YAGNI argument. We don't have the cross-product requirement in the backlog today. And if we do need it in 6 months, refactoring to event sourcing from a clean monolith is straightforward — much easier than refactoring out of an event-sourced system that's serving one use case.好问题。但这就是 YAGNI 论点。今天的 backlog 里没有跨产品需求。而且就算 6 个月后真有，从一个干净的 monolith 重构成 event sourcing 是直接的——比从服务一个用例的 event-sourced 系统重构出去容易得多。
You (Architect): My recommendation — KISS. A monthly cron job that reads the spend aggregate, a simple comparison, and a push-notification fan-out via the existing HSBCnet notification service. Two weeks, ship value, learn from real usage. If the cross-product case materialises, we revisit the architecture with real data, not speculation.我的建议是——KISS。一个月度 cron 任务读取消费聚合，简单的比较，再走现有的 HSBCnet 通知服务做推送扇出。2 周上线，交付价值，从真实使用里学习。如果跨产品用例真的出现，我们用真实数据再重新评估架构，而不是靠推测。
Arjun (Tech Lead): I hear you. But the 2-week version doesn't have a DR strategy or a retry policy. Won't the cron + DB approach hit us when the cron fails silently?听明白了。但2 周的版本没有DR 策略和重试策略。cron 静默失败时，cron + DB 方案不就坑了我们吗？
You (Architect): Right, so — good challenge. We add a DLQ on the cron trigger (Group Resilience requires it anyway), idempotency on the notification (UUID v7), and a Group Cyber SDL threat model as a non-negotiable. Same controls, less architecture. The cost of those controls is 3 days of dev, not 6 weeks.好的，——挑战得好。我们加 cron 触发器的 DLQ（集团 Resilience 本来就要求），通知的幂等性（UUID v7），以及 集团 Cyber 的 SDL 威胁模型作为不可省项。同样的控制，更少的架构。这些控制的成本是3 天开发，不是 6 周。
Margaret (Chief Architect): I'm aligned with Wei. Just to flag — chapter has seen three YAGNI violations this quarter — saga for a one-step flow, CQRS for a single read, and now this. We recommend simple-first by default. Arjun, are you comfortable with the 2-week KISS path?我同意 Wei 的看法。提一下——chapter 这个季度已经看到 3 次 YAGNI 违规——一步流程上 saga、单个读上 CQRS、再加上现在这个。我们默认推荐 simple-first。Arjun，2 周 KISS 方案你 OK 吗？
Arjun (Tech Lead): Honest answer — I was hoping to use the heavy patterns as a CV-driven-development moment. You're right — the squad value is shipping insights, not showcasing patterns. Let's go with the KISS path. I'll update the design doc and re-circulate by Monday.坦白说——我本想用这些重模式给自己简历加一笔。你说得对——squad 的价值是交付洞察，不是展示模式。那就 KISS 路线。我周一前更新设计文档重新发出来。

3. Key Phrases 关键句型

  • Soft I want to push back gently on the architecture choice. 我想就架构选择温和地提一下不同意见 "Push back gently" sets the tone: disagreement without hostility. The opposite — "I disagree" — sounds personal.
  • YAGNI YAGNI — You Ain't Gonna Need It. YAGNI——你不会需要它的 YAGNI is the architect's strongest word against over-engineering. Always pair it with a concrete future scenario the team fears.
  • Spec Right now, the read is always after a single write, on one aggregate. 目前，读取总是发生在单次写后，针对单个聚合 "Right now" is the keyword. You're not saying the future doesn't need it; you're saying the present doesn't.
  • Risk Refactoring to event sourcing from a clean monolith is straightforward — much easier than refactoring out of an event-sourced system. 从干净 monolith 重构到 event sourcing 是直接的——比从 event-sourced 系统重构出去容易得多 This is the "reversibility" argument. Heavy architectures are hard to leave; simple architectures are easy to extend.
  • KISS My recommendation — KISS. Ship value, learn from real usage. 我的建议是——KISS。交付价值，从真实使用里学习 KISS = Keep It Simple, Stupid. Pair it with "ship, then learn" — over-engineering is often fear of the second iteration.
  • Speculation We revisit the architecture with real data, not speculation. 我们用真实数据再重新评估架构，而不是靠推测 "Speculation" is a politeness word for "we don't actually have evidence this is needed".
  • CV-driven I was hoping to use the heavy patterns as a CV-driven-development moment. 我本想用这些重模式给自己简历加一笔 "CV-driven development" — a recognised anti-pattern. Self-deprecating humour works here when admitting it; it lands better than blaming the squad.
  • Value The squad value is shipping insights, not showcasing patterns. squad 的价值是交付洞察，不是展示模式 Reframe the conversation from "which is technically elegant" to "what value does the customer get". This is the chapter's frame, not the architect's ego.
  • Default We recommend simple-first by default. 我们默认推荐 simple-first "By default" reverses the burden of proof. The architect doesn't have to justify simple; the squad has to justify complex.
  • Honest Honest answer — I was hoping to use the heavy patterns. 坦白说——我本想用这些重模式 Self-deprecation with "honest answer" closes the conversation gracefully. Saves face for both sides.

4. Vocabulary 词汇表

  • YAGNI (You Ain't Gonna Need It) n. 你不会需要它 An Extreme Programming principle: don't build for hypothetical future needs.
  • KISS (Keep It Simple, Stupid) n. 保持简单 A design principle: most systems work best if they are kept simple rather than made complex.
  • Over-engineering n. 过度设计 Building a solution more complex than the problem requires — often in anticipation of needs that may never come.
  • Event Sourcing n. 事件溯源 A pattern where state changes are stored as a sequence of events, rather than the current state.
  • CQRS (Command Query Responsibility Segregation) n. 命令查询职责分离 A pattern that separates the write model (commands) from the read model (queries).
  • Saga orchestrator n. Saga 编排器 A pattern for managing long-running, multi-service transactions by coordinating local transactions.
  • Aggregate n. 聚合 A cluster of domain objects treated as a single unit for write operations (DDD term).
  • Read-after-write n. 写后读 A consistency pattern where a write is immediately visible to the subsequent read.
  • CV-driven development n. 简历驱动开发 An anti-pattern where a developer picks a technology to learn or showcase, not because the problem needs it.
  • Cron job n. 定时任务 A scheduled task that runs at fixed intervals (e.g., monthly, daily) on a server.
  • Reversibility n. 可逆性 The property that a decision can be undone cheaply. Architects prefer reversible decisions.
  • Simple-first n. 简单优先 The default at HSBC: ship the simple version, refactor when real evidence demands complexity.

5. Grammar Notes 语法点

① "I want to push back gently" — the softener pattern

When you disagree as an architect, wrap the disagreement in a softener. Three common patterns: I want to push back gently on… (the softest); I'm not sure I'm convinced by… (slightly firmer, invites evidence); I have a different view on… (assertive but collaborative). Avoid: "This is wrong" / "This doesn't work" / "I disagree". They sound personal and start fights the chief architect has to break up.

架构师表达反对的三种礼貌句式：push back gently（最软）、not convinced（邀请举证）、different view（坚定但协作）。

② "What if" + future speculation — the YAGNI counter

The Tech Lead argues "What if we extend to cross-product?". You counter with the conditional past + YAGNI: "If we do need it in 6 months, refactoring to event sourcing from a clean monolith is straightforward." The grammar: If + present, will/be + adjective + a reversibility argument. YAGNI lives or dies on whether the future scenario is reversible.

反驳 YAGNI 论点：用 If + 现在时 提假设，再用可逆性论证反驳。

③ Self-deprecation as a closing device

The Tech Lead admits: "Honest answer — I was hoping to use the heavy patterns as a CV-driven-development moment." The honest answer + self-deprecating noun phrase pattern de-escalates. Compare: "You're wrong and we should always do simple-first." (creates a fight) vs. "Honest answer — I was hoping to use the heavy patterns. You're right." (lands the same point without resentment). Use it when you (or your counterpart) need to back down gracefully.

Honest answer + 自嘲 是优雅退让的句式，让双方都下得来台。

6. Cultural Tips 文化 / 职场 Tips

  • YAGNI is HSBC's default. Margaret explicitly said "chapter has seen three YAGNI violations this quarter". The chapter is signalling: simple-first, always. If you propose complexity, the burden of proof is on you.
  • "Push back gently" is a politeness technology. HSBC's culture runs on face-saving. A blunt "this is wrong" is career-limiting. A gentle pushback that invites the Tech Lead to arrive at the same conclusion is the architect's signature move.
  • Heavy patterns need a multi-aggregate or cross-service trigger. Event sourcing, CQRS, and sagas pay off when there are multiple writers, multiple read shapes, or cross-service consistency. Single-aggregate single-write is the textbook YAGNI case.
  • Reframe from elegance to value. Tech Leads often fall in love with patterns. The architect's job is to reframe the conversation: "the squad value is shipping insights, not showcasing patterns". This is chapter language, not personal opinion.
  • Self-deprecation is permitted — encouraged, even. Arjun admitting "CV-driven development" is a healthy team dynamic. It only works if the chief architect has set the tone that simple is OK. If the chapter punishes simple, the team will over-engineer to look serious.
  • "By default" reverses the burden of proof. Once the chapter says "simple-first by default", the squad has to argue for complexity with evidence (predicted RPS, multiple read shapes, cross-service need). Speculation doesn't count.
  • Refactoring out is harder than refactoring in. The strongest YAGNI argument: a simple system is easy to extend with event sourcing later; an event-sourced system serving one use case is hard to refactor away from. Reversibility is the architect's ally.
  • Don't make it personal. Even when the Tech Lead is wrong, frame the conversation as "the use case doesn't need it" not "you over-engineered it". The former is about the problem; the latter is about the person.

7. Practice 练习题

Exercise 1 — Translate the YAGNI pushback

Translate the following YAGNI pushback into natural Chinese. Notice how the architect reframes from elegance to value.

"Thanks, Lin — strong design doc. I want to push back gently on the CQRS choice. Right now, the read shape is identical to the write shape — both return the same JSON. CQRS pays off when the read and write models diverge, or when you need a denormalised projection for query speed. We have neither today. YAGNI — if the read shape diverges in 6 months, refactoring from a single model to CQRS is straightforward. The squad value is shipping the dashboard, not showcasing patterns. My recommendation: KISS, single read/write model, ship in 3 weeks. We revisit the architecture with real data, not speculation."

"Lin，谢谢——设计文档做得很扎实。我想就 CQRS 的选择温和地提一下不同意见。目前读侧形态和写侧形态完全一致——返回同样的 JSON。CQRS 在读侧和写侧模型分叉，或需要反范式投影来加速查询时才划算。今天两个条件都不成立。YAGNI——如果 6 个月后读侧形态真的分叉，从单模型重构到 CQRS 是直接的。Squad 的价值是交付 dashboard，不是展示模式。我的建议是：KISS，单读/写模型，3 周上线。我们用真实数据再重新评估架构，而不是靠推测。"

Exercise 2 — Spot the difference: gentle vs harsh pushback

Which of these would land well in a HSBC design review, and which would create friction? Why?

  1. "This design is over-engineered. Strip it back."
  2. "Thanks, Akash. I want to push back gently on the saga orchestrator. Right now, the flow is a single service writing to a single DB. Sagas pay off when you have cross-service consistency to manage. We don't. YAGNI. My recommendation: a simple in-service transaction, ship in 2 weeks. We revisit when there's a real cross-service need."
  3. "I disagree with the CQRS choice. Use a single model."
  4. "Honest answer — I'm not convinced the event sourcing pays off here. The use case is a single aggregate, single read. The pattern's value is in time-travel replay and many-reader fan-out, neither of which we have. Refactoring to event sourcing from a clean monolith later is straightforward. Recommend: KISS path, ship in 4 weeks."

② and ④ land well. ② uses "push back gently" + YAGNI + reversibility + value reframe. ④ uses "honest answer" + "not convinced" + scenario-by-scenario counter + reversibility. ① and ③ are too blunt — they state a verdict without an argument and create a face-loss dynamic. Even if you're right, blunt pushback is a poor architect move at HSBC.

Exercise 3 — Your turn

You are the Solution Architect in the GPB Digital tribe. A Tech Lead has proposed using a custom saga orchestrator + Kafka for a "transfer between two internal accounts" flow. The flow is: (a) debit account A, (b) credit account B, both in the same core banking system. Write a 4–5 sentence YAGNI pushback that: (1) thanks the Tech Lead, (2) uses "push back gently" or similar, (3) explains why the pattern doesn't fit, (4) proposes a KISS alternative, (5) ends with a reversibility argument. Use at least 3 patterns from the Key Phrases section.

"Priya, thanks for the design — really thorough work. I want to push back gently on the saga orchestrator here. Right now, the flow is two operations on the same core banking system — that's a single ACID transaction, not a cross-service saga. Sagas pay off when you have distributed transactions across services that can't share a DB. We don't have that here. YAGNI. My recommendation — KISS: a single in-service transaction, debit + credit in one call, ship in 1 sprint. Refactoring from a clean in-service transaction to a saga later is straightforward if we genuinely need cross-service consistency. The squad value is shipping the transfer feature, not showcasing a saga pattern."
