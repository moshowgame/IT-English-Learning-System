API Design Discussion

Article 09 · Module 5: Cross-Cutting Architecture · Architect English

⏱ 9 min read 🟠 Upper-intermediate 🎯 Choose between REST, GraphQL, and gRPC; pin down versioning, idempotency, and pagination

1. Scenario 场景背景

It is 11:00 AM HKT. You are a Solution Architect in the GPB Digital tribe, running a 60-minute API design discussion in a Hong Kong meeting room. The topic is the GPB onboarding API surface — the new set of APIs that will be exposed to the OnePay mobile app and to two external fintech partners via the partner gateway. Akash is dialling in from Bangalore, Mei is in Shanghai, and Diego (peer architect, UK) is on the line.

上午 11 点香港时间，你是 GPB 数字化 tribe 的 Solution 架构师，在香港会议室主持一场 60 分钟的 API 设计讨论。议题是 GPB 开户 API 接口——即将暴露给 OnePay 手机 App 以及 通过合作方网关暴露给两家外部金融科技合作方的一组新 API。Akash 从班加罗尔拨入，Mei 在上海， Diego（同侪架构师，英国）在电话里。

The squad's Tech Lead wants to use GraphQL for the mobile client to avoid over-fetching. The platform team has just published a Group Standard saying REST is the default for partner-facing APIs. You have ADR-0179 in flight and need to land on a protocol choice, a versioning policy, an idempotency strategy, and a pagination standard — all in 60 minutes.

Squad 的 Tech Lead 想用 GraphQL 给手机端以避免 over-fetching。平台团队刚发布了一个 集团标准，说合作方 API 默认走 REST。ADR-0179 已经在进行中，需要在 60 分钟内敲定协议选型、版本策略、幂等策略和分页标准。

2. Dialogue 英文对话

You (Architect): Right, everyone — we have four things to land today: the protocol choice, the versioning policy, the idempotency strategy, and the pagination standard. Let's start with protocol — that's the loudest disagreement.好——今天要敲定 4 件事：协议选型、版本策略、幂等策略和分页标准。先从协议开始——这是分歧最大的。
Mei (Tech Lead, Shanghai): I'm pushing for GraphQL on the mobile client. Right now we're over-fetching — the OnePay onboarding screen needs customer name, KYC status, and risk grade, and the REST endpoint returns the whole customer object. GraphQL lets the client ask for exactly what it needs.我倾向手机端用 GraphQL。现在我们是 over-fetching——OnePay 开户屏要客户姓名、KYC 状态和风险等级，但 REST 接口返回整个客户对象。GraphQL 让客户端按需取。
You (Architect): Good question to push on. Honest answer — I agree with the diagnosis, not the prescription. Over-fetching is real, but GraphQL trades it for under-fetching-N+1 queries, and our squad doesn't have a federation story yet. My recommendation: REST for partner-facing, GraphQL only for the mobile BFF layer behind our gateway. That way the partners see REST — which is the Group Standard — and the mobile app gets a GraphQL endpoint that aggregates the internal services.值得追问。说实话——我同意诊断，不同意药方。Over-fetching 是真问题，但 GraphQL 用 under-fetching 的 N+1 查询来换，我们 squad 还没有 federation。我的建议是：合作方走 REST，手机 BFF 层走 GraphQL，藏在我们的网关后面。这样合作方看到的是 REST（符合集团标准），手机 App 通过 GraphQL 端点聚合内部服务。
Diego (Architect, UK): Quick one — for the internal service-to-service calls between the onboarding service and the KYC service, are we sticking with gRPC?问一下——内部服务到服务的调用，比如开户服务和 KYC 服务之间，还是用 gRPC 吗？
You (Architect): Yes. gRPC for east-west, REST for north-south partner-facing, GraphQL for the mobile BFF. That's the trifecta — and it lines up with the Group Standard on API protocols. Akash, can you confirm the platform team's gRPC service mesh is ready?是的。东西向走 gRPC，南北向合作方走 REST，手机 BFF 走 GraphQL。三件套——这也和集团 API 协议标准对齐。Akash，平台团队的 gRPC 服务网格 ready 了吗？
Akash (Dev, Bangalore): Yes, mesh is up. Quick one on versioning — when the partner team wants to add a new optional field to the onboarding response, do we bump the major version or just add the field?是的，mesh 已经上线。问个版本问题——合作方团队想给开户响应加一个新的可选字段，我们是要跳大版本还是直接加字段？
You (Architect): Good question. URI versioning for partner-facing, header versioning for internal. For partner-facing REST, we use URL path versioning — /v1/onboarding, /v2/onboarding — and the rule is: additive changes don't bump the major version, breaking changes do. A new optional field is additive — it stays on /v1. A removed field is breaking — that's /v2.好问题。合作方走 URI 版本，内部走 header 版本。合作方 REST 我们用 URL 路径版本——/v1/onboarding、/v2/onboarding——规则是：增量变更不跳大版本，破坏性变更才跳。加一个可选字段是增量——留在 /v1。删一个字段是破坏——那才上 /v2。
Rohit (QA Lead): What about idempotency on the partner POST? If the partner retries because of a timeout, we don't want to onboard the same customer twice.合作方 POST 的幂等性怎么办？如果合作方因为超时重试，我们不能给同一个客户开户两次。
You (Architect): Right, so we mandate the Idempotency-Key header on every state-changing endpoint. The partner generates a UUID v7 at their edge, sends it in the header, and we dedup against Redis with a 24-hour window. Same key in, same response out — even if the underlying onboarding is still running. This is the same pattern as ADR-0158 on OneWealth, so we get consistency across the tribe.好，我们对每个有状态变更的接口强制要求 Idempotency-Key 请求头。合作方在他们边缘生成 UUID v7，放在请求头里，我们用 Redis 做 24 小时窗口去重。同一 key 进来，给同一响应——即使底层开户还在跑。这和 OneWealth 的 ADR-0158 一个套路，整个 tribe 一致。
Mei (Tech Lead, Shanghai): Last one — pagination on the partner GET. We have 200K customer records. Offset pagination is easy but slow on deep pages; cursor pagination is faster but harder to debug.最后一个——合作方 GET 的分页。我们有 20 万客户记录。Offset 分页简单但深页慢；cursor 分页快但难调试。
You (Architect): Honest answer — cursor pagination for partner-facing, offset for internal admin UIs. Cursor scales to 200K and beyond, and the partner only ever paginates forward anyway. Internal admin UIs are paginated by humans who want to jump to "page 47" — offset is fine there. Mei, can you write up the cursor contract in the OpenAPI spec — page size capped at 100, opaque cursor, and a hard max of 10,000 records per request?说实话——合作方走 cursor 分页，内部管理 UI 走 offset。Cursor 可以扛 20 万甚至更多，合作方反正也只往前翻。内部管理 UI 是人在翻页，要"跳到第 47 页"——offset 在那种场景没问题。Mei，你能不能在 OpenAPI 规约里把 cursor 契约写清楚——page size 上限 100，cursor 不透明，每次请求最多 1 万条？
Diego (Architect, UK): Just to double-check — the Group Standard on API protocols does permit GraphQL behind the BFF, right? I want to make sure we're not deviating.确认一下——集团 API 协议标准是允许 GraphQL 藏在 BFF 后面吧？我想确认我们没违规。
You (Architect): Good check. I'll confirm with the platform team by EOD tomorrow and add the citation to ADR-0179. If Group Standard says no GraphQL at all, we'll fall back to REST with sparse fieldsets — that's a known pattern and the squad has done it before.好眼力。我明天 EOD 前和平台团队确认，把引用加到 ADR-0179。如果集团标准完全不允许 GraphQL，我们就退回 REST + sparse fieldsets——这是已知模式，squad 之前做过。
You (Architect): Right, so to close — gRPC east-west, REST north-south partner-facing with URL versioning, GraphQL behind the BFF for the mobile app, Idempotency-Key required on all state-changing endpoints with a 24-hour Redis dedup, cursor pagination for partner GET, offset for internal. Mei, can you update ADR-0179 by EOD Friday?好，收尾——东西向 gRPC，南北向合作方走 REST + URL 版本，手机 BFF 走 GraphQL，所有有状态变更的接口强制 Idempotency-Key + 24 小时 Redis 去重，合作方 GET 走 cursor 分页，内部走 offset。Mei，你周五 EOD 前把 ADR-0179 更新好？

3. Key Phrases 关键句型

  • Open We have four things to land today. 今天要敲定 4 件事 State the agenda at minute 0. A 60-minute API discussion without a stated agenda becomes a 3-hour debate on REST vs GraphQL.
  • Push back I agree with the diagnosis, not the prescription. 我同意诊断，不同意药方 The most polite architect pushback. Validate the problem, then challenge the solution. Both sides feel heard.
  • Trifecta gRPC for east-west, REST for north-south partner-facing, GraphQL behind the BFF. 东西向 gRPC，南北向合作方 REST，BFF 后 GraphQL "East-west" = service-to-service. "North-south" = client-to-service. BFF = backend-for-frontend.
  • Versioning URL path versioning — /v1/onboarding, /v2/onboarding. URL 路径版本——/v1/onboarding、/v2/onboarding "Additive changes don't bump the major version, breaking changes do." Memorise this rule.
  • Idempotency We mandate the Idempotency-Key header on every state-changing endpoint. 我们对每个有状态变更的接口强制要求 Idempotency-Key 请求头 "Mandate" is a verb here. The Idempotency-Key is non-optional; reject requests without it at the gateway.
  • Consistency This is the same pattern as ADR-0158, so we get consistency across the tribe. 这和 ADR-0158 一个套路，整个 tribe 一致 Cite the prior ADR. Patterns travel by reference, not by re-derivation. Cite-and-link is faster than re-explain.
  • Trade-off Cursor pagination is faster but harder to debug; offset is easier but slow on deep pages. Cursor 分页快但难调试，offset 简单但深页慢 Always name both sides. Architects don't say "use cursor" — they say "cursor for X because Y; offset for Z because W".
  • Contract Can you write up the cursor contract in the OpenAPI spec? 你能不能在 OpenAPI 规约里把 cursor 契约写清楚 A contract is not a doc — it's the OpenAPI spec, the schema, the protobuf. "Write the contract" means "make it machine-readable".
  • Fallback If Group Standard says no GraphQL, we'll fall back to REST with sparse fieldsets. 如果集团标准不允许 GraphQL，我们就退回 REST + sparse fieldsets Always have a plan B when citing a Group Standard. Group Standards are revisited and sometimes tightened.
  • Cite I'll confirm with the platform team and add the citation to ADR-0179. 我会和平台团队确认并把引用加到 ADR-0179 "Citation" = the link to the Group Standard in the ADR. If the citation is missing, the ADR is a wishlist, not a decision.

4. Vocabulary 词汇表

  • REST (Representational State Transfer) n. REST 架构风格 A resource-oriented API style using HTTP verbs. Group Standard default for partner-facing APIs.
  • GraphQL n. GraphQL 查询语言 A query language that lets clients request exactly the fields they need. Risks include N+1 queries and federation complexity.
  • gRPC n. gRPC 远程过程调用 A high-performance RPC framework using protobuf. Used for service-to-service (east-west) calls.
  • BFF (Backend-For-Frontend) n. 面向前端的后端 A thin API layer tailored to one client (e.g., the mobile app), aggregating internal services.
  • East-west traffic n. 东西向流量 Service-to-service traffic inside the cluster.
  • North-south traffic n. 南北向流量 Traffic between external clients and the cluster (e.g., partner APIs).
  • URL path versioning n. URL 路径版本 Versioning via the URL path, e.g., /v1/onboarding.
  • Additive change n. 增量变更 A change that adds new optional fields but does not remove or change existing ones — does NOT bump the major version.
  • Breaking change n. 破坏性变更 A change that removes or renames a field, or changes a type — bumps the major version.
  • Idempotency-Key n. 幂等键 A client-generated unique key (UUID v7) that lets the server safely dedup retries.
  • Cursor pagination n. Cursor 分页 Pagination using an opaque cursor token — scales to large datasets, no deep-page cost.
  • Offset pagination n. Offset 分页 Pagination using ?offset=100&limit=20 — easy for humans to debug, slow on deep pages.

5. Grammar Notes 语法点

① "I agree with the diagnosis, not the prescription" — the architect's pushback

I agree with the diagnosis, not the prescription is a politeness pattern that has saved many an architect from an argument. The parallel structure (with the diagnosis, not the prescription) signals that you've listened, validated the problem, and are now challenging the solution. The other person doesn't feel dismissed — they feel heard on the part that matters (the problem) and corrected on the part that doesn't (the solution).

I agree with the diagnosis, not the prescription 是架构师最体面的反驳——肯定问题、否定方案。

② "gRPC for east-west, REST for north-south" — the protocol pattern

The parallel for-phrases (gRPC for east-west, REST for north-south, GraphQL for the BFF) are how architects summarise multi-protocol designs in a single sentence. The rule is: name the protocol, then name the direction / use-case. Avoid bare lists (gRPC, REST, GraphQL) — they don't tell the reader which protocol goes where. The prepositional phrase is what carries the design decision.

用 for 引导的介词短语是架构师做多协议选型总结的标准句式：协议 + 适用场景。

③ "Add additive changes don't bump the major version, breaking changes do" — the inversion

Additive changes don't bump the major version, breaking changes do. The trailing "do" is an inverted auxiliary that avoids repeating bump the major version. This pattern is dense — pack two rules into one sentence. It is the architect's voice: maximise information per minute, because the room is busy and the design review is in 3 days. Use inversion when both clauses share a verb; use plain repetition when clarity matters more than density.

句末的 do 用来避免重复动词——架构师讲话追求"高信息密度"。在 NFR 评审这类时间紧的场合特别好用。

6. Cultural Tips 文化 / 职场 Tips

  • Group Standard first, personal preference second. When a Tech Lead says "let's use GraphQL for the partner API", the Group Standard on API protocols says REST. Cite the Group Standard by name, then propose GraphQL behind the BFF. The Tech Lead still gets GraphQL where it helps; the partners stay on REST.
  • "Diagnosis vs prescription" is the architect's politeness device. When a junior architect wants to push back on a senior's design, never say "that's wrong". Say "I agree with the diagnosis, not the prescription". The senior keeps face; the design gets better.
  • Versioning rules must be in the ADR, not in tribal knowledge. "Additive changes don't bump the major version, breaking changes do" — write it down. The next 5 API changes will be debated by 5 different squads; the ADR is the only place the rule is unambiguous.
  • Idempotency-Key is non-negotiable on POSTs. If a partner POST doesn't have the Idempotency-Key header, reject it at the gateway. Don't make it "advisory". The day a partner retries due to a 2-second timeout, "advisory" becomes "we onboarded the same customer twice".
  • Cursor pagination is for machines; offset is for humans. Partner UIs are written by code — they paginate forward forever, cursor is fine. Internal admin UIs are operated by humans — they want "page 47", offset is fine. Match the pagination to the consumer.
  • Always have a fallback when citing a Group Standard. "If Group Standard says no GraphQL, we'll fall back to REST with sparse fieldsets." This sentence turns a potentially blocking question into a known plan B. Group Standards are revisited; your design should survive the revision.
  • Cross-region courtesy on API reviews. When Diego (UK) joins late, recap the protocol decision in 30 seconds before continuing. UK is 8 hours behind — Diego probably hasn't had coffee and definitely hasn't read the ADR. Recap is courtesy.

Exercise 2 — Spot the architect mistake

Which of these API design proposals would pass a strict HSBC design review and which would not? Why?

  1. "Let's use GraphQL for everything — partner, mobile, internal. It's flexible."
  2. "gRPC for east-west, REST for partner-facing, GraphQL behind the BFF, with URL versioning and the Idempotency-Key mandated on state-changing endpoints."
  3. "We removed the `email` field from the /v1/onboarding response. Partners should use /v2 instead. Oh, and we still updated /v1 in place."
  4. "Pagination: offset. Everyone uses offset. Why complicate things?"

② passes — it has the protocol trifecta, the versioning rule, and the idempotency mandate. ① fails (GraphQL for partner-facing violates the Group Standard). ③ is a disaster — removing a field and updating /v1 in place is a breaking change without a version bump; the next partner integration breaks silently. ④ is a PM-style "keep it simple" answer that doesn't scale to 200K records; offset on deep pages will time out the partner's request.

Exercise 3 — Your turn

You're a Solution Architect in the CMB Digital Channels tribe. The squad is designing the CMB onboarding API for a new partner — a corporate KYC vendor. The partner wants gRPC, but the Group Standard says REST for partner-facing. The squad also wants offset pagination, but the partner will have 500K customer records. Write your 60-second response to the squad in English. Use at least 3 patterns from the Key Phrases section.

"OK, two trade-offs to land. First, protocol — the partner wants gRPC, but the Group Standard on API protocols says REST for partner-facing north-south traffic. I agree with the diagnosis — gRPC has a tighter contract and lower latency — but not the prescription. Our recommendation: REST for the partner-facing API, gRPC for the east-west calls between our onboarding service and the KYC service. That way the partner stays REST-compliant, and we get the gRPC benefits internally. We'll cite the Group Standard in the ADR and have a fallback ready if the standard is revised. Second, pagination — the squad wants offset, but 500K records means deep-page latency will kill the partner. Honest answer: cursor pagination for the partner GET, offset for our internal admin UIs. Page size capped at 100, opaque cursor, max 10,000 records per request. Mei, can you update ADR-0190 with the protocol decision and the cursor contract by EOD Wednesday?"
