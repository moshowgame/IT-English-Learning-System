System Design Whiteboarding

Article 07 · Module 4: Live Design Sessions · Architect English

⏱ 9 min read 🟡 Intermediate 🎯 Drive a live system design on the whiteboard, take "what about…?" questions, and steer to a decision

1. Scenario 场景背景

It is 2:00 PM HKT. You are a Solution Architect in the RBWM Digital tribe, running a 90-minute system design session in a Hong Kong meeting room — but with Akash dialing in from Bangalore and Priya from London. The whiteboard is up, the topic is the OneWealth real-time portfolio rebalancing service, and you are designing it live in front of: Margaret (Chief Architect, HK), Diego (peer architect, UK), Mei (Tech Lead, Shanghai), and Rohit (Product Owner, RBWM Wealth).

下午 2 点香港时间，你是 RBWM 数字化 tribe 的 Solution 架构师，在香港会议室主持一场 90 分钟的系统设计会——Akash 从班加罗尔、Priya 从伦敦拨入。白板已摆好，议题是 OneWealth 实时投资组合再平衡服务，你将在 Margaret（首席架构师，香港）、Diego（同侪架构师，英国）、 Mei（Tech Lead，上海）和 Rohit（产品负责人，RBWM 财富）面前现场设计。

The goal of a whiteboarding session is not to draw the perfect diagram — it is to align on the trade-offs and capture a decision everyone can live with. You will take "what about…?" questions, push back on scope creep, and steer toward an ADR. You already have ADR-0158 in flight, scoped to the rebalancing flow only.

白板会的目标不是画出完美的图，而是对齐 trade-off，并落定一个大家都能接受的决策。 你会面对 "what about…" 的追问，需要对范围蔓延说不，并把结论收敛到 ADR 里。 ADR-0158 已经在进行中，范围限定在再平衡流程。

2. Dialogue 英文对话

You (Architect): OK, everyone — thanks for joining. Right, so the goal of the next 90 minutes is to land on a design for the OneWealth rebalancing service, and to close out ADR-0158. I'll draw first, then we go round for "what about…?" questions.好，谢谢大家参加。接下来的 90 分钟，我们要为 OneWealth 再平衡服务敲定一个设计，把 ADR-0158 收尾。我先画，画完大家按顺序提 "what about…" 的问题。
You (Architect): Stepping back — rebalancing is triggered by three things: market-data ticks, customer-driven allocation changes, and an end-of-day scheduled rebalance. Honest answer: for v1, we are only designing the customer-driven and scheduled paths. The market-data tick path is descope for this ADR — that's a separate conversation.退一步看——再平衡由 3 件事触发：行情 ticks、客户主动调仓、日终调度再平衡。说实话，v1 我们只设计客户主动和调度两条路径。行情 tick 路径在本次 ADR 里 descope，那是另一场对话。
Margaret (Chief Architect): Quick one before you draw — is this event-driven or request-response? That choice changes the whole picture.你画之前问一下——这个是事件驱动还是请求-响应？这个选择会改变整张图。
You (Architect): Good question. Hybrid. Customer-driven allocation changes come in via an API call from the mobile app — that's request-response. Once we accept the change, we publish an event to Kafka and downstream services consume it for rebalancing, notification, and audit. The scheduled path is pure event-driven — a daily cron publishes a rebalance-required event.好问题。混合模式。客户主动调仓走手机银行 API 调用——这是请求-响应。一旦接受，我们就发事件到 Kafka，下游服务订阅做再平衡、通知和审计。调度路径是纯事件驱动——日终 cron 发出 rebalance-required 事件。
Diego (Architect, UK): What about the idempotency on the customer-driven path? If the mobile app retries because of a flaky network, we'd double-rebalance.如果客户主动路径上的幂等性怎么处理？如果手机银行因为网络抖动重试，就会重复再平衡。
You (Architect): Good catch. The client must send an Idempotency-Key header — a UUID v7 generated at the edge. We dedup at the API gateway for 24 hours against a Redis store. Same key in, same response out — even if the underlying rebalance is still running.抓得好。客户端必须带 Idempotency-Key 请求头——UUID v7 在 API 边缘生成。我们在 API 网关对 Redis 存储做 24 小时去重。同一个 key 进来，给同一个响应——即使底层再平衡还在跑。
Rohit (PO, RBWM): What about the audit trail? Compliance will ask — who triggered the rebalance, what was the pre/post allocation, and was the customer's risk profile respected?审计怎么办？合规会问——谁触发的再平衡、调仓前后配置是什么、客户风险偏好是否被尊重？
You (Architect): Fair point. Every rebalance event carries a correlation ID and we write an immutable audit record to the GLCM audit store — pre-allocation, post-allocation, risk-profile match, and a hash of the customer's consent. Group Standard on auditability requires 7-year retention, and the audit store is the only system that holds the personal data in cleartext.说得对。每个再平衡事件带一个 correlation ID，我们往 GLCM 审计存储写一条不可变记录——调仓前、调仓后、风险偏好匹配、客户同意书哈希。集团审计标准要求 7 年保留，审计存储是唯一保留个人数据明文的系统。
Mei (Tech Lead, Shanghai): What about the failure mode? If the downstream rebalancing service is down, does the API call hang or do we 5xx?失败模式怎么设计？下游再平衡服务挂了，API 调用是挂起还是 5xx？
You (Architect): Honest answer — we accept the change first and publish the event, then respond 202 Accepted to the client. The actual rebalance happens asynchronously. If downstream is down, the event stays in Kafka with retries. The client doesn't wait — that would tie up a mobile thread for 30 seconds and tank the user experience.说实话——我们先接受这次调仓并发事件，然后给客户端返回 202 Accepted。真正的再平衡是异步发生的。下游挂了，事件留在 Kafka 里重试。客户端不等——让客户端等 30 秒会毁掉用户体验。
Akash (Dev, Bangalore): What about the read-after-write consistency? If I update my allocation on the app, the next screen should show the new allocation — not the old one.读写一致性怎么办？如果我在 App 上改了配置，下一屏应该看到新配置——不是旧的。
You (Architect): Good question. Cache invalidation is the trick. We use a versioned read model — every allocation change bumps a customer-level version, and the mobile app does a read-your-writes query against the latest version. For now, we accept up to a 2-second read lag; we'll re-evaluate in the next quarter once we have real load data.好问题。关键是缓存失效。我们用带版本号的读模型——每次调仓把客户级版本号加一，手机 App 用read-your-writes 查询最新版本。目前，我们允许最多 2 秒的读延迟；下个季度有了真实负载数据再重评。
Margaret (Chief Architect): Right, so to summarise before we close — hybrid API + event-driven, idempotency via UUID v7, async 202 Accepted, GLCM audit, versioned read model with a 2-second read lag. Are we aligned on this for ADR-0158?好，收尾前总结一下——API + 事件驱动混合，UUID v7 幂等，异步 202 Accepted，GLCM 审计，2 秒读延迟的版本化读模型。ADR-0158 我们对齐到这个方向了吗？
You (Architect): Yes. I'll update ADR-0158 by EOD Friday with these decisions and the explicit descope of the market-data tick path. Thanks, everyone — I'll circulate the whiteboard photo on the chapter Teams channel.好。我周五 EOD 前把 ADR-0158 更新好，把这些决策写进去，并显式 descope 行情 tick 路径。谢谢大家——我会把白板照片发到 chapter 的 Teams 频道。

3. Key Phrases 关键句型

  • Open Right, so the goal of the next 90 minutes is… 好，接下来的 90 分钟目标是…… Whiteboard sessions need a stated goal at minute 0. If you don't say it, scope creeps.
  • Scope Honest answer: for v1, we are only designing the customer-driven and scheduled paths. 说实话，v1 我们只设计客户主动和调度两条路径 Use "honest answer" when you're about to say "no" to scope. Polite, firm, and adult.
  • Descope The market-data tick path is descope for this ADR. 行情 tick 路径在本次 ADR 里 descope "Descope" = explicitly remove from current scope. Naming it prevents "but what about…?" later.
  • Hybrid Hybrid. Customer-driven comes in via an API call, then we publish an event. 混合模式。客户主动走 API 调用，然后发事件 "Hybrid" beats "it depends". When asked binary questions in whiteboard sessions, name a default pattern.
  • Idempotency The client must send an Idempotency-Key header — a UUID v7 generated at the edge. 客户端必须带 Idempotency-Key 请求头——UUID v7 在边缘生成 Always cite the dedup window. "24 hours against Redis" is concrete; "we dedup" is hand-wavy.
  • Audit Every event carries a correlation ID and we write an immutable audit record to GLCM. 每个事件带一个 correlation ID，我们写一条不可变审计记录到 GLCM Cite the Group Standard (7-year retention) and the system of record (GLCM). Don't say "we log it".
  • Async We accept the change first and respond 202 Accepted. The client doesn't wait. 我们先接受调仓然后返回 202 Accepted。客户端不等 202 Accepted = "I got it, processing async". Don't ever make a mobile client wait 30 seconds.
  • CAP Cache invalidation is the trick. We use a versioned read model. 关键是缓存失效。我们用带版本号的读模型 "Read-your-writes" is the consistency promise. Pair it with a quantified read lag (e.g., 2 seconds).
  • Re-evaluate We'll re-evaluate next quarter once we have real load data. 下个季度有了真实负载数据再重评 Pairs with "for now". Promises a future review without committing to scope today.
  • Close Are we aligned on this for ADR-0158? ADR-0158 我们对齐到这个方向了吗？ Always close with a decision and a name (the ADR). If you don't, the session was a chat, not a decision.

4. Vocabulary 词汇表

  • Whiteboarding session n. 白板会 A live design meeting where the architect draws the system in real time and takes questions.
  • Request-response n. 请求-响应 A synchronous pattern where the client waits for an answer (REST APIs).
  • Event-driven adj. 事件驱动的 A pattern where services communicate by publishing and consuming events (e.g., Kafka).
  • Hybrid pattern n. 混合模式 Combining request-response (for sync API entry) with event-driven (for async fan-out).
  • Idempotency-Key n. 幂等键 A client-generated unique key (often UUID v7) that lets the server dedup retries.
  • 202 Accepted n. HTTP 202 The HTTP status code meaning "received, processing async".
  • Correlation ID n. 关联 ID A unique ID propagated across all services touched by one user action — for tracing and audit.
  • Read-your-writes n. 读写一致性保证 The consistency guarantee that after a write, the same client can immediately read the new value.
  • Versioned read model n. 版本化读模型 A read store that carries a per-record version to support cache invalidation.
  • Audit trail n. 审计链 An immutable record of who did what, when — required for regulatory compliance.
  • Read lag n. 读延迟 The maximum delay between a write completing and reads reflecting it (e.g., 2 seconds).
  • Descope v. 削减范围 Explicitly remove something from current scope to prevent feature creep.

5. Grammar Notes 语法点

① "What about…?" — the live-design probe

The single most important phrase in a whiteboarding session is What about…?. Architects should invite it: then we go round for "what about…?" questions. It signals psychological safety — no one is afraid to ask the awkward question. Junior devs will probe things no senior will. What about the failure mode? What about idempotency? — these are the questions that catch design holes.

白板会上最重要的句型是 What about…?。架构师要主动邀请这种问题，给团队心理安全感。

② "Honest answer:" — the soft "no"

Honest answer: for v1, we are only designing the customer-driven and scheduled paths. The colon after answer signals "I'm about to tell you something you might not want to hear". It's a politeness device — you're not being evasive, you're being direct but kind. Pair it with the explicit descope so the team can challenge the descope, not the principle.

Honest answer: 后的冒号表示"我接下来说的你不一定爱听"。配合显式 descope 使用，更体面。

③ "Right, so…" and "OK, so…" — the discourse markers

Right, so the goal of the next 90 minutes is… and OK, so to summarise…. These discourse markers signal transitions in a live meeting: opening, mid-meeting pivot, and closing. Use them to give the room a beat — they tell people "we are now moving from X to Y". Without them, whiteboards feel chaotic and decisions get lost.

Right, so… 和 OK, so… 是现场会议里的过渡标记——开篇、转折、收尾都用得上。

6. Cultural Tips 文化 / 职场 Tips

  • Whiteboard ≠ perfect diagram. The goal is alignment on trade-offs, not a beautiful UML sequence diagram. Arrows that are roughly right beat arrows that are precisely wrong. If a dev can't follow your arrows, simplify, don't add more detail.
  • Always invite "what about…?" questions. If nobody asks, the room is either intimidated or not paying attention. Both are your fault. Seed one yourself: "What about the failure mode? Anyone want to push on that?"
  • Cross-region whiteboard etiquette. With Akash in Bangalore and Priya in London, dial-in folks see the board last. Take a photo every 5 minutes, drop it in the Teams channel, and read it out loud. Don't make Bangalore chase the whiteboard.
  • Name a default pattern, not "it depends". "Hybrid" is a better answer than "it depends". Junior architects hide behind "it depends" because it sounds safe. Naming a pattern and its trade-off is the architect's job.
  • Cite the Group Standard by name. When the PO asks about audit, don't say "we'll log it". Say "we'll write an immutable audit record to GLCM, 7-year retention per the Group Standard on auditability". The Group Standard trumps local preference.
  • Close with a decision, not a summary. Bad close: "So we've discussed a lot of things." Good close: "Are we aligned on hybrid, UUID v7 idempotency, 202 Accepted, GLCM audit? — I'll update ADR-0158 by EOD Friday." A whiteboard without a decision was a chat.
  • Descope in public, never in private. If you're going to cut market-data ticks from this ADR, say it out loud in the room. Otherwise, the PO will write a follow-up Jira asking "what about the tick path?" and you'll be back at the whiteboard in 2 weeks.

7. Practice 练习题

Exercise 1 — Translate the whiteboarding summary

Translate the following whiteboarding closing summary into natural Chinese. Pay attention to how the architect names the patterns and closes the session.

"Right, so to summarise before we close. We're recommending a hybrid pattern — REST for the customer-driven allocation change, then publish a rebalance event to Kafka. Idempotency via a UUID v7 key, dedup at the API gateway for 24 hours. We return 202 Accepted — the client doesn't wait for the rebalance. Every event carries a correlation ID and we write an immutable audit record to GLCM per the Group Standard on auditability. Read-your-writes via a versioned read model, with a 2-second read lag for now. Market-data tick path is descope — that's a separate ADR. Are we aligned? I'll update ADR-0162 by EOD Wednesday."

"好，收尾前总结一下。我们推荐混合模式——客户主动调仓走 REST，然后发再平衡事件到 Kafka。幂等用 UUID v7 key，在 API 网关做 24 小时去重。我们返回 202 Accepted——客户端不等再平衡完成。每个事件带 correlation ID，按集团审计标准写一条不可变审计记录到 GLCM。读模型用版本化读模型保证 read-your-writes，目前读延迟 2 秒。行情 tick 路径 descope——那是另一个 ADR。我们对齐了吗？我周三 EOD 前把 ADR-0162 更新好。"

Exercise 2 — Spot the architect mistake

Which of these whiteboard closes is most "architect-like" and which is most "PM-like"? Why?

  1. "OK, so we've discussed a lot of things today. Lots of good ideas. Let's keep talking next week."
  2. "Right, so to summarise: hybrid REST + Kafka, UUID v7 idempotency, 202 Accepted, GLCM audit, 2-second read lag. Market-data tick is descope. Are we aligned? I'll update ADR-0162 by EOD Wednesday."
  3. "Great session everyone. I'll send out the slides tomorrow and we can pick this up in the next chapter."
  4. "I think we're almost there on the design. Let me just polish the diagram a bit and circulate it."

② is architect-like (named patterns, explicit descope, ADR number, named deadline). ①, ③, and ④ are PM-like or facilitator-like — vague, action-less, no ADR. A common mistake new architects make is closing a whiteboard session like a facilitator. The architect closes with a decision and an ADR number.

Exercise 3 — Your turn

You're running a 90-minute whiteboarding session on the OnePay real-time fraud-screening service. The room includes a Chief Architect, a peer architect, a Tech Lead, a PO, and a dev dialling in from Bangalore. A junior dev asks: "What about the cold-start case — when a customer opens the app for the first time and we have no history?" Write your 60-second answer in English. Use at least 3 patterns from the Key Phrases section.

"Good question. Honest answer — for v1, we accept a higher false-positive rate on the cold-start case and lean on a manual review queue. We score on a small set of static signals — device fingerprint, IP geo, KYC risk grade — and any score above 0.6 routes to a human reviewer. We do NOT use a deny-by-default policy; that would block legitimate new customers. The ML-based personalised model kicks in once we have at least 10 transactions for that customer. For now, we accept the higher manual-review load — we'll re-evaluate next quarter once we have cold-start data. I'll capture this as an explicit non-goal in the ADR."
