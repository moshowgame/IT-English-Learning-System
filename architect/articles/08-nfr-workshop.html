NFR Workshop

Article 08 · Module 4: Live Design Sessions · Architect English

⏱ 9 min read 🟡 Intermediate 🎯 Quantify the NFRs — performance, security, scalability, auditability — with numbers, owners, and evidence

1. Scenario 场景背景

It is 3:00 PM HKT. You are a Solution Architect in the CMB Digital Channels tribe, hosting a 60-minute Non-Functional Requirements workshop in a Hong Kong meeting room, with Akash dialling in from Bangalore and Wei in Shanghai. The squad is finalising ADR-0167 for the HSBCnet corporate cash-flow dashboard, and the design review next Tuesday will reject the ADR if the NFRs are not quantified.

下午 3 点香港时间，你是 CMB 数字渠道 tribe 的 Solution 架构师，在香港会议室主持一场 60 分钟的非功能需求（NFR）工作坊，Akash 从班加罗尔拨入，Wei 在上海。这个 squad 正在收尾 ADR-0167，主题是 HSBCnet 公司网银现金流看板，下周二的设计评审如果 NFR 没有量化就会 reject。

In the room: Mei (Tech Lead), Jonathan (BA), Rohit (QA Lead), Priya (peer architect, dialling in from London), and Sara from Group Cyber, invited specifically to challenge the security NFRs. Your job: turn vague NFRs like "fast" and "secure" into quantified, owner-attributed, evidence-backed statements.

会议室里：Mei（Tech Lead）、Jonathan（BA）、Rohit（QA 负责人）、 从伦敦拨入的同侪架构师 Priya，以及专程被请来挑战安全 NFR 的 集团网络的 Sara。你的工作：把"快"、"安全"这种模糊的 NFR 变成有数字、有 owner、有证据的陈述。

2. Dialogue 英文对话

You (Architect): OK, everyone — right, so the goal of the next 60 minutes is to quantify every NFR in the design. If it doesn't have a number, an owner, and evidence, it doesn't ship to the design review. Let's go round by category — performance first, then security, scalability, and auditability.好——接下来 60 分钟的目标是把每个 NFR 都量化。没有数字、没有 owner、没有证据的就进不了设计评审。我们按类别走——先性能，再安全，再扩展性，再审计性。
Mei (Tech Lead, Shanghai): For performance — the dashboard has to load fast. We've said "sub-second" in the story, but we haven't pinned down the percentile yet.性能这块——看板要快。Story 里写了"亚秒级"，但还没定百分位。
You (Architect): Good catch. Let's be specific: dashboard initial render at p95 ≤ 1.5 seconds and p99 ≤ 3 seconds, measured from the API gateway to the last byte of HTML. The owner is Akash, evidence is the JMeter scripts in the performance Git repo, run on the pre-prod environment with a 5,000-concurrent-user load profile. Quantified?抓得好。明确一下：看板首屏渲染 p95 ≤ 1.5 秒，p99 ≤ 3 秒，从 API 网关到 HTML 末字节计。Owner 是 Akash，证据是性能 Git 仓库里的 JMeter 脚本，在 pre-prod 环境跑 5000 并发用户负载。这样量化可以吗？
Rohit (QA Lead): Hold on — 5,000 concurrent users? Last quarter the peak was 1,800. Are we sizing for 3x growth or are we over-engineering?等一下——5000 并发用户？上个季度峰值才 1800。我们是按 3 倍增长规划还是过度设计？
You (Architect): Honest answer — we're sizing for 2.5x. The CMB growth target for FY26 is 40% YoY, and we want headroom. If Rohit, you'd rather cap at 1.5x and re-evaluate mid-year, we can do that — but we'd be buying back performance in 6 months. I'd recommend we hold the 2.5x.说实话——我们按 2.5 倍规划。CMB FY26 的增长目标是同比 40%，我们要留余量。如果 Rohit 你倾向于按 1.5 倍然后年中再重评，也可以——但6 个月后我们还得再花钱买性能。我建议先按 2.5 倍。
Sara (Group Cyber): Quick one on security — what encryption standard are we using for the data-in-transit between the browser and the API gateway?问一下安全——浏览器到 API 网关之间的传输加密标准用哪个？
You (Architect): Good question. TLS 1.3 only, with the Group Standard cipher suite — that's the SDL baseline. We also have HSTS with a 1-year max-age and certificate pinning for the mobile client. Owner: Mei for the API side, mobile TL for the app. Evidence: the Group Cyber penetration test report from Q4, plus a quarterly Nessus scan in CI/CD.好问题。只允许 TLS 1.3，用集团标准密码套件——这是 SDL 基线。同时启用 1 年 max-age 的 HSTS，移动端证书锁定。Owner：API 侧是 Mei，移动端是移动 TL。证据：Q4 集团网络渗透测试报告，加上 CI/CD 里每季度的 Nessus 扫描。
Priya (Architect, London): Just to flag — the data-at-rest encryption for the cash-flow snapshots. Is that AES-256 with HSBC HSM-managed keys, or are we using AWS KMS?提一下——现金流快照的数据静态加密。是用 AES-256 + 汇丰 HSM 托管密钥，还是用 AWS KMS？
You (Architect): Right, so we're using HSBC HSM-managed keys with envelope encryption. AES-256 for the data, the data encryption key is wrapped by an HSM-held key. Group Cyber mandates HSM, not KMS, for any data classified as Customer Confidential — and cash-flow snapshots are CC by default. Owner: Sara's team for the key rotation policy, our squad for the integration.好，我们用汇丰 HSM 托管密钥 + envelope encryption。数据本身 AES-256，数据加密密钥由 HSM 持有的 key 包裹。集团网络要求 HSM 而不是 KMS，只要数据被归类为客户机密（CC）就用 HSM——现金流快照默认是 CC。Owner：密钥轮换策略归 Sara 团队，集成归我们 squad。
Jonathan (BA): For scalability — when the squad asked the PO about the projected user growth, he just said "a lot". Can we make the scalability NFR evidence-based?扩展性这块——squad 问 PO 预测的用户增长，他就说"很多"。扩展性 NFR 能不能做成有证据的？
You (Architect): Good push. Let's lock the scalability NFR on the dashboard API at 2,500 RPS sustained, 5,000 RPS peak, with horizontal auto-scaling on the EKS cluster. The evidence is the load test report; the trigger to scale up is CPU > 65% over 5 minutes. Action: Jonathan, can you get the PO to commit a number — even a 12-month projection — by end of this week?好追问。看板 API 的扩展性 NFR 锁定在 2500 RPS 稳态、5000 RPS 峰值，EKS 集群横向自动伸缩。证据是负载测试报告；触发扩容的条件是 5 分钟内 CPU > 65%。Action：Jonathan，你能不能让 PO 给个数字——哪怕是 12 个月的预测——本周 EOD 前定下来？
Rohit (QA Lead): For auditability — the Group Standard on auditability says 7-year retention, but where does the audit log actually live?审计性这块——集团审计标准说保留 7 年，但审计日志到底放在哪？
You (Architect): Every API call writes a structured audit record to GLCM — that includes user ID, action, timestamp, IP, and a hash of the request body. GLCM is the system of record for audit, with WORM (write-once-read-many) storage and 7-year retention. Owner: Rohit for the test coverage, InfoSec for the retention policy sign-off.每次 API 调用都写一条结构化审计记录到 GLCM——包括用户 ID、操作、时间戳、IP、请求体的哈希。GLCM 是审计的唯一数据源，用 WORM（一次写多次读）存储，保留 7 年。Owner：测试覆盖归 Rohit，保留策略签批归 InfoSec。
You (Architect): Right, so to wrap up — performance is p95 ≤ 1.5s at 5K concurrent, security is TLS 1.3 + HSM-managed keys, scalability is 2.5K RPS sustained with auto-scale, auditability is GLCM WORM 7-year retention. Mei, can you update ADR-0167 with these by EOD Friday?好，收尾——性能是 5K 并发下 p95 ≤ 1.5 秒，安全是 TLS 1.3 + HSM 托管密钥，扩展性是 2500 RPS 稳态 + 自动伸缩，审计性是 GLCM WORM 7 年保留。Mei，你能不能在周五 EOD 前把这些写进 ADR-0167？

3. Key Phrases 关键句型

  • Open Right, so the goal of the next 60 minutes is to quantify every NFR. 好，接下来的 60 分钟目标是把每个 NFR 都量化 NFR workshops need a stated rule: if it doesn't have a number, an owner, and evidence, it doesn't ship.
  • Quantify Let's be specific: p95 ≤ 1.5 seconds and p99 ≤ 3 seconds. 明确一下：p95 ≤ 1.5 秒，p99 ≤ 3 秒 Always cite both p95 and p99. p95 alone hides tail latency — and tail latency is what kills mobile UX.
  • Owner The owner is Akash, evidence is the JMeter scripts. Owner 是 Akash，证据是 JMeter 脚本 "Owner" is non-negotiable. An NFR without an owner is an NFR that never gets measured.
  • Evidence Evidence: the JMeter scripts in the performance Git repo. 证据：性能 Git 仓库里的 JMeter 脚本 Evidence is the artefact, not the assertion. "It's fast" is not evidence. The JMeter report is evidence.
  • Baseline TLS 1.3 only, with the Group Standard cipher suite — that's the SDL baseline. 只允许 TLS 1.3，用集团标准密码套件——这是 SDL 基线 Cite SDL by name. "TLS 1.3" alone is not enough — you need the cipher suite, the HSTS, the cert pinning.
  • Mandate Group Cyber mandates HSM, not KMS, for any CC-classified data. 集团网络要求 HSM 而非 KMS，只要数据被分类为 CC "Mandates" = a hard rule. Distinguish it from "prefers" or "recommends". Mandates cannot be negotiated.
  • Capacity 2,500 RPS sustained, 5,000 RPS peak. 2500 RPS 稳态，5000 RPS 峰值 "Sustained" + "peak" is the standard capacity pair. Sustained = what you hold all day; peak = what you survive.
  • Auto-scale Horizontal auto-scaling on the EKS cluster, trigger at CPU > 65% over 5 minutes. EKS 集群横向自动伸缩，触发条件是 5 分钟内 CPU > 65% Always cite the trigger condition. "Auto-scaling" without a trigger is just a Terraform module.
  • WORM WORM storage and 7-year retention. WORM 存储，保留 7 年 "WORM" (write-once-read-many) is the audit standard. Plain S3 won't do for an audit trail.
  • Action Action: Jonathan, can you get the PO to commit a number by end of this week? Action：Jonathan，你能不能让 PO 本周 EOD 前给个数字？ End every NFR workshop with named actions. NFRs that don't get owned in the room don't get owned at all.

4. Vocabulary 词汇表

  • NFR (Non-Functional Requirement) n. 非功能需求 Performance, security, scalability, auditability — quantified with numbers.
  • p95 / p99 latency n. P95 / P99 延迟 The 95th / 99th percentile response time — what the slowest 5% / 1% of users experience.
  • RPS (Requests Per Second) n. 每秒请求数 The number of API calls a system can serve per second.
  • Sustained vs peak load n. 稳态 / 峰值负载 Sustained = what you handle all day; peak = what you survive for a short burst.
  • TLS 1.3 n. TLS 1.3 协议 The current transport-layer encryption standard. The Group SDL baseline.
  • HSTS (HTTP Strict Transport Security) n. 严格传输安全 A header that forces browsers to use HTTPS only.
  • Certificate pinning n. 证书锁定 A mobile-app technique that only trusts a specific server certificate.
  • HSM (Hardware Security Module) n. 硬件安全模块 A physical device that holds cryptographic keys — mandated by Group Cyber for CC data.
  • Envelope encryption n. 信封加密 A pattern where a data key is encrypted by a master key, allowing the data key itself to be stored alongside the data.
  • EKS (Elastic Kubernetes Service) n. 弹性 K8s 服务 AWS managed Kubernetes — used by most HSBC squads for container workloads.
  • WORM (Write-Once-Read-Many) n. 一次写多次读存储 The storage class for audit logs that cannot be modified after write.
  • SDL (Secure Development Lifecycle) n. 安全开发生命周期 HSBC's mandatory security process baked into delivery.

5. Grammar Notes 语法点

① "Let's be specific:" — the quantifier opener

Let's be specific: p95 ≤ 1.5 seconds and p99 ≤ 3 seconds. The imperative "let's" + be specific is the architect's way of saying "we're done with hand-waving". It's a soft power move — you're not telling the room they're being vague, you're inviting them to be precise. Pair it with numbers immediately, so the room sees the shape of a quantified NFR.

Let's be specific: 是架构师打断模糊讨论的软开场——接着立刻给数字。

② "The owner is X, the evidence is Y" — the NFR triplet

Every quantified NFR has three parts: a number, an owner, and evidence. In the workshop, repeat this triplet out loud for every NFR. The repetition sounds mechanical — that's the point. It trains the room to think in triplets, and the design review will check that every NFR has all three. The owner is Akash, the evidence is the JMeter scripts in the performance Git repo.

每个量化 NFR 有三件套：数字、Owner、证据。在工作坊里反复说三件套——这是 NFR Workshop 的节奏。

③ "Group X mandates Y" — citing the policy by name

Group Cyber mandates HSM, not KMS, for any CC-classified data. The active-voice mandate (Group Cyber mandates…) is more authoritative than we should use HSM. It removes the room's ability to negotiate — you cannot argue with a Group Standard, only with a colleague's opinion. Architects who cite the policy by name are architects who don't get pushed back on.

Group Cyber mandates… 比 we should use… 有权威得多——因为集团标准不能谈判。

6. Cultural Tips 文化 / 职场 Tips

  • "Fast" is not an NFR. If a Tech Lead says "the dashboard has to be fast", ask: fast for whom, at what percentile, under what load, measured how? An NFR without a number is a wish. Architects turn wishes into commitments.
  • Always cite p95 and p99. p50 (median) hides tail latency. p95 is what most users feel; p99 is what your worst user feels. Tail latency is what causes mobile-app crashes, not medians. If a dev quotes only p50, push back.
  • Group Cyber is a stakeholder, not a gate. Invite Sara into the room, not just into the email thread. When she challenges a security NFR live, the rest of the room hears the rationale. Don't fight her in a DM later.
  • Distinguish "mandates" from "prefers". "Group Cyber mandates HSM" is non-negotiable. "InfoSec prefers mTLS" is a soft preference. Mixing them up is a common mistake — and it leads to architectures that look Group-Standard-compliant but aren't.
  • Size for 2.5x, not 5x. 5x sizing is over-engineering; 1x sizing is fragile. 2.5x is the HSBC sweet spot — it gives you a sprint to react when the PO overshoots projection. Rohit will push back on cost; 2.5x is your defensible middle.
  • HSM, not KMS, for CC data. This is the single most common Group-Standard trap. AWS KMS is easier to use, but Group Cyber requires HSM for any data classified as Customer Confidential. Default to HSM, not KMS, on HSBC work.
  • End with named actions, not a recap. Bad close: "So we covered a lot." Good close: "Jonathan, get the PO a number by Friday. Mei, update ADR-0167 by Friday EOD. Rohit, drop the test plan in Confluence." NFRs without owners don't get measured.

7. Practice 练习题

Exercise 1 — Translate the NFR workshop summary

Translate the following NFR workshop closing summary into natural Chinese. Pay attention to the NFR triplet (number, owner, evidence) and how the architect cites Group Standards.

"Right, so to wrap up. Performance: API p95 ≤ 800ms, p99 ≤ 1.5 seconds, under 3,000 concurrent users, measured by the Gatling scripts in the perf repo — owner Akash. Security: TLS 1.3 only, HSM-managed AES-256 envelope encryption per the Group Cyber mandate for CC data, pen test from Q1 — owner Mei, evidence the pen test report. Scalability: 1,800 RPS sustained, 3,500 RPS peak, auto-scale trigger at CPU > 70% over 5 minutes — owner Akash. Auditability: GLCM WORM, 7-year retention per the Group Standard on auditability — owner Rohit for tests, InfoSec for sign-off. Action: Mei, can you update ADR-0171 with all four by EOD Wednesday? Jonathan, please get the PO to confirm the 12-month user projection by Friday."

"好，收尾。性能：API p95 ≤ 800 毫秒、p99 ≤ 1.5 秒，3000 并发用户，证据是 perf 仓库里的 Gatling 脚本——Owner Akash。安全：只允许 TLS 1.3，HSM 托管 AES-256 信封加密（按集团网络对 CC 数据的强制要求），Q1 渗透测试——Owner Mei，证据是渗透测试报告。扩展性：1800 RPS 稳态、3500 RPS 峰值，5 分钟内 CPU > 70% 触发自动扩容——Owner Akash。审计性：GLCM WORM，按集团审计标准保留 7 年——Owner Rohit 负责测试，InfoSec 负责签批。Action：Mei，你能不能在周三 EOD 前把 ADR-0171 这 4 项都更新好？Jonathan，麻烦你周五前让 PO 把 12 个月的用户预测确认一下。"

Exercise 2 — Spot the architect mistake

Which of these NFR statements would pass a strict design review at HSBC and which would not? Why?

  1. "Performance: the API should be fast."
  2. "Performance: p95 ≤ 1.5s at 5K concurrent users, owner Akash, evidence the JMeter report."
  3. "Security: we use strong encryption."
  4. "Security: TLS 1.3 with the Group Standard cipher suite, AES-256 HSM-managed envelope encryption per Group Cyber mandate for CC data, owner Mei, evidence the Q1 pen test report."

② and ④ would pass — they have a number, an owner, and evidence, and ④ explicitly cites the Group Standard. ① and ③ would be rejected: "fast" and "strong encryption" are not NFRs, they are wishes. A common mistake new architects make is shipping ①/③-style NFRs into the design review and getting the ADR bounced.

Exercise 3 — Your turn

You're running an NFR workshop for the OnePay real-time fraud-screening service. The room includes a Tech Lead, a BA, a QA Lead, a peer architect dialling in from London, and a Group Cyber rep. The squad has written four vague NFRs: "fast", "secure", "scales", and "auditable". Write your 90-second workshop opening in English. Use at least 3 patterns from the Key Phrases section.

"OK everyone, right, so the goal of the next 90 minutes is to quantify every NFR. If it doesn't have a number, an owner, and evidence, it doesn't ship to the design review. Let's go round by category — performance first, then security, then scalability, then auditability. For performance — let's be specific: fraud-screening API p95 ≤ 300ms, p99 ≤ 800ms, under 2,000 RPS sustained, measured by the Gatling scripts in the perf repo. Owner: the Tech Lead. So we're using AES-256 with HSM envelope encryption, not KMS. Mei, can you confirm the Q1 pen test is current? For scalability — 2,000 RPS sustained, 4,000 RPS peak, auto-scale trigger at CPU > 65% over 5 minutes. For auditability — every scoring call writes a structured audit record to GLCM, WORM storage, 7-year retention per the Group Standard on auditability. Action at the end: BA, please get the fraud ops team to commit the 12-month RPS projection by Friday. Tech Lead, you own the ADR-0173 update by EOD Wednesday."
