Event-Driven Architecture Conversation

Article 10 · Module 5: Cross-Cutting Architecture · Architect English

⏱ 10 min read 🟠 Upper-intermediate 🎯 Design Kafka topics, event schemas, ordering, exactly-once delivery, and the saga pattern

1. Scenario 场景背景

It is 4:00 PM HKT. You are a Solution Architect in the RBWM Digital tribe, running a 75-minute design conversation in a Hong Kong meeting room on the OneWealth event-driven rebalancing flow. The squad is implementing ADR-0184, which has already been signed off at the chapter level but is now in detailed design. Akash is dialling in from Bangalore, Priya from London, and Mei is in Shanghai.

下午 4 点香港时间，你是 RBWM 数字化 tribe 的 Solution 架构师，在香港会议室主持一场 75 分钟的设计对话，主题是 OneWealth 事件驱动的再平衡流程。Squad 正在实施 ADR-0184，ADR 已经在 chapter 层面签过，现在进入详细设计阶段。Akash 从班加罗尔拨入， Priya 从伦敦，Mei 在上海。

The decisions still open: the Kafka topic naming, the event schema (Avro vs Protobuf), the ordering guarantee per customer, the delivery semantics (at-least-once vs exactly-once), and the saga pattern for the cross-service rebalance flow. The Chief Architect Margaret is in the room, and she will sign off on the schema evolution policy before the squad starts coding.

还没敲定的决策：Kafka topic 命名、事件 schema（Avro 还是 Protobuf）、 按客户的顺序保证、投递语义（at-least-once 还是 exactly-once），以及跨服务再平衡流程的 saga 模式。首席架构师 Margaret 在场，squad 开始写代码前她要先签 schema 演进策略。

2. Dialogue 英文对话

You (Architect): OK everyone — right, so the goal of the next 75 minutes is to lock down the event-driven design for OneWealth rebalancing. We have five open items: topic naming, event schema, ordering, delivery semantics, and saga. Let's start with topic naming — that's the easiest win and the platform team needs it for the topic-provisioning script.好——接下来 75 分钟目标是把 OneWealth 再平衡的事件驱动设计敲死。5 个开放项：topic 命名、事件 schema、顺序、投递语义、saga。先从 topic 命名开始——这是最简单的一个，平台团队要拿去配 topic 脚本。
You (Architect): My recommendation: reverse-DNS style, kebab-case — hsbc.rbwm.onewealth.rebalance.requested, hsbc.rbwm.onewealth.rebalance.completed. The pattern is hsbc.{tribe}.{domain}.{entity}.{event-type}. The platform team's Kafka topic standard already mandates this — and it gives us a clean prefix for ACLs.我的建议：反向 DNS 风格、kebab-case——hsbc.rbwm.onewealth.rebalance.requested、hsbc.rbwm.onewealth.rebalance.completed。模式是 hsbc.{tribe}.{domain}.{entity}.{event-type}。平台团队的 Kafka topic 标准已经强制这个——而且给我们一个干净的 ACL 前缀。
Mei (Tech Lead, Shanghai): Quick one — for the event schema, the squad wants Protobuf because that's what the OnePay service uses. But the Group Standard on data interchange says Avro for events going into the central data lake.问一下——事件 schema，squad 想用 Protobuf，因为 OnePay 服务用的就是。但集团数据交换标准要求进中央数据湖的事件走 Avro。
You (Architect): Good catch. Honest answer — we use Avro on the wire, with the Protobuf-generated Java classes as the in-process types. The wire format is Avro, the Confluent Schema Registry enforces it, and the data lake reads it natively. The Protobuf types are still useful for compile-time type-safety in the consumer — we just convert at the boundary. This satisfies both the squad's ergonomics and the Group Standard.抓得好。说实话——线协议用 Avro，进程内类型用 Protobuf 生成的 Java 类。线协议是 Avro，Confluent Schema Registry 强制执行，数据湖原生可读。Protobuf 类型在消费者编译期类型安全上仍然有用——我们只在边界做转换。这样既满足 squad 的工程体验，也满足集团标准。
Akash (Dev, Bangalore): Just to flag — the rebalance events for the same customer need to be ordered, right? If the customer changes allocation at 10:00:01 and the scheduled rebalance fires at 10:00:02, we can't process them out of order.提一下——同一客户的再平衡事件需要有序，对吧？如果客户 10:00:01 改配置、10:00:02 调度再平衡触发，我们不能乱序处理。
You (Architect): Right, so we partition by customer ID — that's the ordering key. Kafka guarantees ordering within a partition, and as long as the same customer ID always hashes to the same partition, the order is preserved. We also pin the partition count at 12 for now — that gives us headroom for parallelism across customers without re-partitioning mid-year.好，我们按客户 ID 分区——这就是排序 key。Kafka 在分区内保证有序，只要同一客户 ID 总是哈希到同一分区，顺序就保留。分区数先锁在 12——给客户维度的并行留余量，年内不用重新分区。
Priya (Architect, London): What about delivery semantics? The squad said they want exactly-once, but I've seen a lot of squads burn time chasing exactly-once when at-least-once with idempotent consumers would have done the job.投递语义呢？Squad 说他们想要 exactly-once，但我见过很多 squad 追 exactly-once 烧了大量时间，其实 at-least-once + 幂等消费者就够用。
You (Architect): Honest answer — at-least-once with idempotent consumers is the default, exactly-once is the exception. Kafka's exactly-once only works inside Kafka — the moment the consumer calls an external service (the rebalancing engine, the audit store), the guarantee evaporates. So the pattern is: at-least-once from Kafka, idempotency key per event, and the downstream consumer dedups against a Redis store. We get the same end-state as exactly-once, with much less operational pain.说实话——默认是 at-least-once + 幂等消费者，exactly-once 是例外。Kafka 的 exactly-once 只在 Kafka 内部有效——一旦消费者调用外部服务（再平衡引擎、审计存储），保证就消失了。所以套路是：Kafka 这边 at-least-once，每个事件带幂等 key，下游消费者对 Redis 存储去重。我们用更少的运维代价拿到和 exactly-once 一样的最终结果。
Mei (Tech Lead, Shanghai): Last big one — the saga pattern. The rebalance flow has four steps: validate allocation, check risk profile, execute the trade, write the audit record. If step 3 fails, we need to compensate step 2 and step 1.最后大件——saga 模式。再平衡流程有 4 步：校验配置、检查风险偏好、执行交易、写审计记录。如果第 3 步失败，我们需要补偿第 2 步和第 1 步。
You (Architect): Right, so orchestrated saga over choreographed, for v1. An orchestrator service holds the state machine — it sends the "validate allocation" command, waits for the result, then sends "check risk profile", and so on. On failure, the orchestrator issues compensating transactions in reverse order. We tried choreographed sagas on the OnePay flow and they were a debugging nightmare — every service owned part of the state and nobody could answer "where are we in the flow?" Honest answer: orchestrated is more code, but it's the only one a junior dev can debug at 2 AM.好，v1 用编排式 saga 而非编舞式 saga。一个 orchestrator 服务持有状态机——发"校验配置"命令、等结果、再发"检查风险偏好"，依次类推。失败时，orchestrator 按反序发起补偿事务。OnePay 流程我们试过编舞式 saga，调试是噩梦——每个服务都持有一部分状态，没人能回答"流程走到哪一步了"。说实话：编排式代码更多，但这是唯一一个初级开发者凌晨 2 点能 debug 的方案。
Margaret (Chief Architect): Quick one on schema evolution — before the squad starts coding, I need to know: if we add a new field to the rebalance event in 6 months, do old consumers break?问一下schema 演进——squad 开始写代码前，我需要知道：6 个月后给再平衡事件加一个新字段，老消费者会不会挂？
You (Architect): Good question. No, old consumers don't break, because Avro's default-value semantics let the consumer ignore unknown fields and fill missing fields with defaults. The rule is: additive changes are backward-compatible, breaking changes require a new topic version. Removing a field or changing a type is breaking — that's a new topic, not a new version of the existing one. I'll write this rule into ADR-0184 by EOD Friday as the schema evolution policy.好问题。老消费者不会挂，因为 Avro 的 default-value 语义允许消费者忽略未知字段、对缺失字段填默认值。规则是：增量变更向后兼容，破坏性变更需要新版本 topic。删字段或改类型是破坏——那是新 topic，不是现有 topic 的新版本。我周五 EOD 前把这条规则写进 ADR-0184，作为 schema 演进策略。
You (Architect): Right, so to close — topics named hsbc.rbwm.onewealth.rebalance.*, Avro on the wire, partition by customer ID, at-least-once with idempotent consumers, orchestrated saga with the orchestrator service. Schema evolution rule: additive is backward-compatible, breaking needs a new topic. Mei, ADR-0184 update by EOD Friday. Margaret, can you sign off on the schema evolution policy this week?好，收尾——topic 命名 hsbc.rbwm.onewealth.rebalance.*，线协议 Avro，按客户 ID 分区，at-least-once + 幂等消费者，编排式 saga + orchestrator 服务。Schema 演进规则：增量向后兼容，破坏性变更需要新 topic。Mei，ADR-0184 周五 EOD 前更新好。Margaret，schema 演进策略本周能签批吗？

3. Key Phrases 关键句型

  • Open The goal is to lock down the event-driven design. 目标是把事件驱动设计敲死 "Lock down" = commit to specific decisions, not explore options. Use once the chapter ADR is approved.
  • Naming Reverse-DNS style, kebab-case — hsbc.{tribe}.{domain}.{entity}.{event-type}. 反向 DNS 风格、kebab-case——hsbc.{tribe}.{domain}.{entity}.{event-type} Cite the platform team's Kafka topic standard. Don't invent your own naming — it breaks the ACL scripts.
  • Boundary Avro on the wire, Protobuf-generated classes as the in-process types. 线协议 Avro，进程内类型用 Protobuf 生成的类 "On the wire" = the bytes that travel. "In-process" = the objects in memory. Always separate the two.
  • Ordering We partition by customer ID — that's the ordering key. 我们按客户 ID 分区——这就是排序 key "Ordering key" = the field you partition by. If the field is wrong, ordering is wrong; no other knob can fix it.
  • Capacity We pin the partition count at 12 for now. 分区数先锁在 12 Partition count is a one-way door — re-partitioning mid-flight is a 6-week project. Pick a number that survives the year.
  • Semantics At-least-once with idempotent consumers is the default; exactly-once is the exception. 默认 at-least-once + 幂等消费者；exactly-once 是例外 Always name the default first, then the exception. "Default + exception" is the architect's pattern.
  • Saga Orchestrated saga over choreographed, for v1. v1 用编排式 saga 而非编舞式 "Orchestrated" = a central service owns the state machine. "Choreographed" = each service owns its own. Junior devs can debug orchestrated; only senior devs can debug choreographed at 2 AM.
  • Compensate On failure, the orchestrator issues compensating transactions in reverse order. 失败时，orchestrator 按反序发起补偿事务 "Compensating transaction" = the undo for a previous step. Always cited in reverse order — the last step is undone first.
  • Evolution Additive changes are backward-compatible, breaking changes require a new topic. 增量变更向后兼容，破坏性变更需要新 topic Write this rule into the ADR. Without it, the squad will rename a field in 6 months and 3 consumers will break silently.
  • Sign-off Margaret, can you sign off on the schema evolution policy this week? Margaret，schema 演进策略本周能签批吗？ Always name who signs what and by when. Unnamed sign-offs don't ship.

4. Vocabulary 词汇表

  • Kafka topic n. Kafka 主题 A named, durable stream of events. The unit of subscription and ACL.
  • Reverse-DNS naming n. 反向 DNS 命名 A naming style that starts with the broadest scope (e.g., hsbc) and narrows (e.g., hsbc.rbwm.onewealth).
  • Event schema n. 事件 schema The contract describing the fields and types of an event (Avro .avsc, Protobuf .proto).
  • Avro n. Avro Schema A schema format that supports default values — enabling backward-compatible evolution.
  • Protobuf n. Protobuf 协议 Google binary serialisation format. Strong in-process type-safety, weaker cross-team schema governance.
  • Partition key n. 分区 key The field used to hash a record to a Kafka partition. Determines ordering.
  • At-least-once delivery n. 至少一次投递 The delivery guarantee that a record will not be lost, but may be redelivered.
  • Exactly-once delivery n. 精确一次投递 The delivery guarantee that a record is delivered exactly once. Only works inside Kafka; evaporates at the network boundary.
  • Idempotent consumer n. 幂等消费者 A consumer that can safely process the same event multiple times because it dedups on a key.
  • Saga pattern n. Saga 模式 A pattern for managing distributed transactions across multiple services via a sequence of local transactions + compensating actions.
  • Orchestrated saga n. 编排式 saga A saga where a central orchestrator service holds the state machine.
  • Choreographed saga n. 编舞式 saga A saga where each service reacts to events from the others — no central state.
  • Compensating transaction n. 补偿事务 The undo operation for a previously-completed saga step.

5. Grammar Notes 语法点

① "Orchestrated saga over choreographed, for v1" — the comparative for choice

Orchestrated saga over choreographed, for v1. The "X over Y" pattern is the architect's way of stating a comparative decision in 4 words. Over signals "I considered both, I chose X, here's the temporal scope". Pair it with for v1 to signal that the decision is re-evaluable. This pattern is denser than we recommend orchestrated, not choreographed, for v1 — and in a 75-minute design conversation, density wins.

X over Y 是架构师做对比决策的紧凑句式，4 个词说完选择。配合 for v1 表明会复评。

② "On the wire" vs "in-process" — the two-boundary distinction

Avro on the wire, Protobuf-generated classes as the in-process types. This two-boundary distinction is a hallmark of senior architect talk. On the wire = the bytes that cross a network. In-process = the objects inside a JVM. Confusing the two is a common junior mistake ("we use Protobuf end-to-end") that breaks the data lake. Naming both boundaries explicitly is what makes the design defensible to Group Cyber and the data team.

明确区分线协议和进程内类型是高级架构师的标志——这两种很容易被初级工程师混为一谈。

③ "We tried X, it was a Y" — the past-tense war story

We tried choreographed sagas on the OnePay flow and they were a debugging nightmare. The past-tense war story is the architect's most convincing tool. You're not saying "I think choreographed is bad"; you're saying "we tried it, here's what happened". A past-tense war story carries more weight than an opinion because it has evidence behind it. Use war stories from your own tribe — OnePay for RBWM, GLCM for CMB — and always name the project, the pattern, and the failure mode.

过去时的"踩坑故事"比单纯说"我不喜欢 X"有说服力得多。要点出项目、模式和失败模式。

6. Cultural Tips 文化 / 职场 Tips

  • Cite the platform standard by name. When you name a Kafka topic, cite the platform team's Kafka topic standard. Don't invent your own naming — it breaks the ACL scripts and the platform team will reject the topic. hsbc.{tribe}.{domain}.{entity}.{event-type} is the only acceptable pattern.
  • Exactly-once is a trap on the network boundary. A junior dev reads "exactly-once" in the Kafka docs and assumes it works across the whole system. It doesn't. The moment the consumer calls the rebalancing engine or the audit store, the guarantee is gone. Architects who let juniors chase exactly-once are architects who will be debugging it at 2 AM.
  • Partition count is a one-way door. Picking 6 partitions vs 12 might seem harmless today, but re-partitioning mid-flight is a 6-week project and the consumer offsets need a re-mapping plan. Always size partitions for the year, not the sprint.
  • Orchestrated saga for junior-heavy squads. Choreographed saga looks elegant on a slide but is undebuggable by anyone except the senior who designed it. If your squad is 60% junior or mid-level, orchestrated saga is the only safe choice. You can always move to choreographed in v2 once the team has the muscle memory.
  • Schema evolution policy is the ADR's most-quoted section. "Additive is backward-compatible, breaking needs a new topic" — write it down, cite the Avro default-value semantics, and link to the Group Standard. 6 months from now, the squad will rename a field and 3 consumers will silently break. The policy is the only thing that prevents the regression.
  • Cross-region courtesy on Kafka designs. When Priya (UK) joins late, recap the schema decision in 30 seconds. UK is 8 hours behind — Priya has had no time to read the ADR. Recap is courtesy, and it's also risk-reduction: a UK consumer team that didn't hear the policy will deploy the wrong schema.
  • End with a named sign-off, not a recap. "Margaret, can you sign off on the schema evolution policy this week?" is the close. A close without a sign-off name is a chat. Margaret's sign-off is the only thing that lets the squad start coding.

Exercise 2 — Spot the architect mistake

Which of these event-driven design statements would pass a strict design review at HSBC and which would not? Why?

  1. "We use Protobuf end-to-end — on the wire and in-process."
  2. "Avro on the wire, Protobuf in-process, partitioned by customer ID, at-least-once with idempotent consumers, orchestrated saga. Schema evolution: additive is backward-compatible, breaking needs a new topic."
  3. "We'll figure out the topic naming when we provision the topics."
  4. "Exactly-once end-to-end. The rebalancing engine, the audit store, the notification service — they all get exactly-once."

② passes — it has the two-boundary distinction, the partition key, the delivery semantics, the saga choice, and the schema evolution policy. ① fails (Protobuf end-to-end violates the Group Standard for events going to the data lake). ③ fails (figuring out naming during provisioning breaks the platform team's ACL scripts). ④ fails (exactly-once across the network boundary is impossible — the guarantee evaporates the moment the consumer calls an external service).

Exercise 3 — Your turn

You're a Solution Architect in the CMB Digital Channels tribe. The squad is designing the CMB trade-settlement event flow. The Tech Lead wants 32 partitions; the platform team recommends 12. The Tech Lead also wants exactly-once; you know it doesn't survive the network boundary. Write your 60-second response in English. Use at least 3 patterns from the Key Phrases section.

"OK, two trade-offs. First, partitions. Honest answer — 12 is enough for now. We're settling roughly 200 trades per second at peak; 12 partitions gives us 17 trades per partition per second, which is well within Kafka's headroom. 32 partitions is a one-way door — re-partitioning mid-flight is a 6-week project and the consumer offsets need a re-mapping plan. I'd recommend 12 for v1, and we re-evaluate at the year mark. Second, exactly-once — that one's off the table. Exactly-once only works inside Kafka; the moment the consumer calls the settlement engine or the audit store, the guarantee evaporates. So the pattern is: at-least-once from Kafka, idempotency key per event, and the downstream consumer dedups against Redis. We get the same end-state with much less operational pain. We tried the exactly-once path on the OnePay trade flow in 2024 — it was a debugging nightmare. Mei, can you update ADR-0195 with the partition count and the at-least-once decision by EOD Wednesday?"
